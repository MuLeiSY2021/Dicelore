# 点子归位 + 一路下沉（idea-to-roadmap）

**点子有海拔。** 用户抛来的一句话可能是一个宏大愿景，也可能是一个具体 bug。本 skill 的活：先把每条点子放到它**真正的海拔**（最高层是里程碑里的宏大目标/愿景），**再顺着海拔一路下沉**到可执行的 backlog 票 + 路线图批次；**若某想法已被设计到「零不确定」**，一路沉到底还能落一份**裁决文档**。先归里程碑、再下沉——别一上来就当成普通 ticket 塞进 backlog，那样会丢掉「这想法到底服务哪个大目标」的脉络。

轻量、不发 subagent、不动代码。

## 输入
用户抛来的一堆散点子/想法，海拔不一：可能是「我想让框架将来能 X」这种愿景，也可能是「这里有个 bug」这种零碎 fix，混杂前后端/core。有的可能刚在对话里被 brainstorm 到设计定型，有的还只是一句粗想法。

## 流程

### 1. 先归里程碑海拔（入口，别跳过）
对**每条**点子先问一句：它服务 / 扩张哪个里程碑（宏大目标·愿景）？据答案分流：

- **全新的宏大目标/愿景** → 在 [`里程碑.md`](../../../docs/wiki/设计/05-现状与计划/里程碑.md) 追加一个新里程碑块，按页内既定格式「宏大目标 → 拆成块 → 怎么实现」写，块内项标 ⬜。
- **扩张/细化已有里程碑** → 在对应里程碑块下补一条 ⬜ 子目标/块。
- **纯属某里程碑内的零碎 fix** → 不必动里程碑正文，但**记清它归属哪个里程碑**（下游 backlog 条目要链接回去，留 traceability）。

**写权边界（要紧）**：里程碑.md 的铁律是「AI 不得在**无人干预**下自行改动」。用户调本 skill 抛点子**本身就是人工干预**，所以 skill 内**可以**写里程碑——但只追加/细化**未来目标块（⬜）与「怎么实现」**；**不擅自标 ✅ 完成、不改写已达成的历史**（完成态由人在真达成那一刻定）。

### 2. 下沉到 backlog 三池
落 [`backlog-<层>.md`](../../../docs/wiki/设计/05-现状与计划/)。把上一步的里程碑块拆成具体 fix/feat，逐条判定 `{层, 类型}` 落对应池（层按**关注点**分，不只看目录——core 与后端都有落点在 `backend/`）：

- **前端** = `frontend/`：组件 / 渲染 / 路由 / i18n / 视觉。
- **后端** = `backend/`：HTTP·WS 接口 / 会话生命周期 / 进程编排。
- **core** = 引擎 / 数据层 / MCP 工具面 / gm-core / 团本构建引擎 / eval harness；落点散在 `backend/`（store/resolve/present/catalog/build/toolgen/expr/eval）、`harness/`（mcp 工具面/dicegm/loregm/eval-*）、`packages/*`（纯库）。

补全字段：`类型(fix|feat)·来源·是否随规模恶化·所属主题·所属里程碑·下一步/依赖`。
**去重聚类**：同一架构病的多个症状归到一个**主题**下（N 个 ticket 常是 1 个决策）；跟既有条目比对，别重开。

### 3. 下沉到路线图
落 [`路线图.md`](../../../docs/wiki/设计/05-现状与计划/路线图.md)。把池里的活按 [里程碑](../../../docs/wiki/设计/05-现状与计划/里程碑.md) 分节、编进/重排四态清单：**反复出现 + 随规模恶化 = 最高优先级**，进靠前批次。每项链接回三池条目 + 里程碑，并按四态归位：
- 无裁决文件 = **未裁决**（默认；哪怕设计已在脑子里，只要没落裁决文件+没被批准，就是未裁决）。
- 有**已勾批准**的裁决文件 = **未完成（已裁决）**。

### 4. （条件）下沉到裁决文档 —— 仅当该想法的设计已「零不确定」
**判据**：该想法是否已在对话里被设计到「没有任何不确定项、仅剩代码实现」？（例如刚用 `superpowers:brainstorming` 走完设计、或与用户逐条敲定过所有 产品 / 承重 / 外部可见行为 / 边界 / 安全策略 决策。）

- **是** → 在 [`裁决记录/`](../../../docs/wiki/设计/05-现状与计划/裁决记录/) 落一份 `<需求名>.md`（格式对齐 [裁决记录 README](../../../docs/wiki/设计/05-现状与计划/裁决记录/README.md) 与同目录既有裁决文件）：
  - 顶部留一个**未勾**的批准勾：`- [ ] 用户已批准本裁决（勾上前视为未裁决，不可进交付波）`。**绝不替用户勾**——这是有意保留的人类设计评审闸。
  - 正文写零不确定设计：`来源/依赖` → `设计（零不确定，仅剩代码实现）` → `验收` → `owns（预期触及）` → `完成后（沉哪个 wiki 设计页 + 关哪条 backlog + 勾路线图 + 删本裁决文件）`。
  - 路线图对应项**挂该裁决文件链接**。⚠️ **挂了链接但没勾 = 铁律仍算未裁决**：此刻路线图该项仍在「未裁决」态、只是已备好待批准的裁决文件；**用户勾上才转「未完成（已裁决）」**、才能进 [`roadmap-delivery-workflow`](../../../.claude/skills/roadmap-delivery-workflow/SKILL.md) 的交付波。写完裁决文档要**停下、明确请用户去审并勾**。
- **否**（还是粗愿景 / 没设计定型） → **不写裁决文档**，停在路线图「未裁决」。裁决留到它真被设计时再补（可日后再进本 skill 一轮，或由 `roadmap-delivery-workflow` 阶段1 在交付前 just-in-time 补）。

> **别全量前置**：只给**已设计到零不确定**的想法写裁决文档，不给一堆粗想法硬凑裁决（Fowler 返工坑——与 `roadmap-delivery-workflow`「just-in-time、不给 47 个路线图项全写」是同一条纪律）。两个 skill 共用 `裁决记录/` 目录 + 同一个用户批准闸——**谁把设计钉死到零不确定，谁就落裁决文档**；本 skill 落的是「随想法一起、上游就已定型」的那些，delivery 阶段1 补的是「交付前才定型」的那些。

> **一路沉到底**：每条点子都要走完里程碑 → backlog → 路线图（→ 已定型则再落裁决），别停在里程碑层。万一某愿景太大、现在无法负责任地拆细，也至少在 backlog 留一个粗主题、在路线图留一个「未来批次」stub，让它有去处、不在里程碑层断头。

### 5. 收尾：提交 + 合并 main（不 push）
落盘即闭环——开分支提交（`git checkout -b docs/idea-<主题>`，message 用 scoped 前缀：写了裁决文档用 `docs(裁决): ...`，否则 `docs(idea): ...` / `docs(backlog): ...`），切回 main `git merge --ff-only <分支>`、删分支。**不向用户问"要提交吗"**，也**不 push**（push 由用户单独指令）。git 命令一律 `--no-pager`。
（注：写了裁决文档时，提交只是把「待批准的裁决文件」落盘；批准勾仍由用户在审阅后自己勾——提交动作不代表已裁决。）

## 硬约束
- **海拔优先**：先归里程碑、再一路下沉到 backlog + 路线图，一次到底；别把宏大愿景直接当普通 ticket 塞 backlog。
- **里程碑.md**：AI 不得在**无人干预**下自行改动；本 skill 由用户抛点子触发即干预，故可在其内追加/细化未来目标块（⬜）与「怎么实现」，但**不标 ✅、不改写已达成历史**。
- **路线图.md**：AI 维护、可重排。
- **裁决文档是条件产物**：只在设计已「零不确定」时写，**不替用户勾批准**（勾前一律未裁决）；不给粗想法全量前置裁决。
- 只做归位/拆解/聚类/排序/编批（+ 设计已定型时落裁决文档），**不直接动手实现**——实现走 `roadmap-delivery-workflow`。
- **落盘即提交合并 main**：不停下来问用户；不 push。
- git 命令一律 `--no-pager`（否则 less 卡死 Bash 会话）。
