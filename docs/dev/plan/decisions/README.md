# 裁决记录 —— 路线图需求的「已裁决」详细设计

> 路线图里每个标「已裁决」的需求，对应这里一份**裁决文件**。
> **铁律：没链接裁决文件、或裁决文件未经用户批准 → 一律视为「未裁决」**（还只是想法，不可进交付波）。

## 裁决文件是什么

某个路线图需求的**详细设计方案**，详尽到**没有任何不确定项、仅剩代码实现**的程度——所有产品 / 承重 / 外部可见行为 / 边界 / 安全策略决策都已拍定写死，交付 agent 照着写代码即可、不再需要做任何设计判断。

## 用户批准（硬闸）

每份裁决文件**顶部留一个批准勾**：

```markdown
- [ ] 用户已批准本裁决（勾上前视为未裁决，不可进交付波）
```

设计阶段写好裁决文件后**停下等用户勾**；**用户勾上才算「已裁决」**、才能进 [`roadmap-delivery-workflow`](../../../../.claude/skills/roadmap-delivery-workflow/SKILL.md) 的交付波。这是有意保留的**人类设计评审闸**——不可逆决策不再靠零散 AskUserQuestion，而是攒进完整裁决文件让用户一次审清、打勾。

## 命名 & 链接

- 一需求一文件：`<需求名或编号>.md`（与路线图条目对应）。
- 路线图标「已裁决」的项**必须链到对应裁决文件**；链接缺失 = 该项实为未裁决。

## 生命周期

```
未裁决想法
  → 写裁决文件（消尽一切不确定项，仅剩代码实现）
  → 用户勾批准（[x]）
  → 路线图挂链接、该项转「已裁决 / 未完成」
  → 交付（roadmap-delivery 波）
  → 完成 / 归档
  → 把最终设计结论沉淀进 wiki（04 子系统设计页 / 02 概念 / 设计页「决策与权衡」节）
  → 删除本裁决文件
```

裁决文件是**过渡设计稿、非长存权威**——交付完它的内容已落进 wiki，留着就是双写漂移源，故删。

## 与别处的分工（别混）

| 处 | 是什么 | 寿命 |
|----|--------|------|
| **裁决记录/（本目录）** | 需求级「做什么·怎么设计」的零不确定设计 + 用户批准闸 | **临时**（交付后删） |
| [`docs/delivery/`](../../../delivery/) | 「怎么并发交付这一轮」的编排运行记录（DAG / 波 roster / 复盘） | 长存 |
| wiki 设计页（04 / 02 / 决策节） | 项目**最终权威**设计结论（去过程化） | 长存 |
