> **本页职责**:把万域界修仙手札角色卡的九组机制,逐项映射到 dicelore 的 canonical 概念(域/skill/工具),给出迁移路径、优先级、风险张力与不可丢约束。
> **依据**:[`01-真实人物卡分析.md`](01-真实人物卡分析.md)(人物卡机制全貌)、[`04-差距分析.md`](04-差距分析.md)([§一 能力盘点](04-差距分析.md) 为底座 + 三组差距),佐证 [`02-论文结论.md`](02-论文结论.md) / [`03-现状分析.md`](03-现状分析.md)。
> **约束**:落到 dicelore canonical 概念;引用 01/04 带节号;不改代码;只写本页。术语遵 [`docs/wiki/术语表.md`](../../../wiki/术语表.md)。

## 〇 概览

### 0.1 迁移总判

万域界角色卡是**方向对、接口错、层级错**的工程实例([04 §A.4](04-差距分析.md)):

- **方向对**:结构化状态 schema、难度裁决约束、动作裁决出口概念,与论文 `FG-all` / PAYADOR / AI-Native Games Survey 同向([04 §A.1](04-差距分析.md))。
- **接口错**:状态更新走"AI 输出文本块 + regex 提取 + 外部 MVU 引擎应用",是 `FG-gen` 路线变体,被 Labyrinth 6 组对照直接否定(0.267 vs 0.422,p=0.0025,[02 §一.2](02-论文结论.md));门禁走 L2 prompt 教条而非 L1 引擎强制,被 PTCG legal-masking −118 否定([02 §二.5](02-论文结论.md));纯叙事裁决无骰子函数,被 `FG-states` 失衡结论否定([02 §一.3](02-论文结论.md))。
- **论文未覆盖的工程化创新**:COT 7 Step / 10 档难度 / 关键词触发 / 字面原则 / 八卦铁律 / insertion_order 工程——需 dicelore 自行判别([04 §A.3](04-差距分析.md))。

故迁移**不是照搬**,而是**拆解后按 dicelore 既有杠杆(L1 工具强制 / visible 横切 / 版本化 / watcher / 明暗骰)重新落位**:可机械判定的落 L1 工具,需语义推理的留 L2 skill,品类特定的进团本 references,框架通用的不覆盖 gm-core。

### 0.2 迁移三原则

1. **接口升级**:凡万域靠"prompt 教条 + regex 提取 + 外部引擎"实现的,迁移到 dicelore 的"工具调用 + 引擎内应用 + L1 强制"。状态更新走 `sheet_update`(FG-all 路线),不走 `<UpdateVariable>` 文本块(FG-gen 路线,被否定)。
2. **层级升级**:凡万域靠 L2 prompt 教条(要求 LLM 自觉过门禁/自觉更新八卦/自觉遵守难度)的,迁移到 L1 引擎强制(watcher 自驱 / legality mask / 引擎参数化)。门禁不照搬 prompt。
3. **载体下沉**:凡万域把机制拍平进世界书散文/规则散文的,迁移到结构化可机械消费载体(pool / params / entity schema / front watcher)。呼应 [04 §D.1 共指#3](04-差距分析.md)。

## 一 机制逐项迁移映射表

### 1.1 设定层 YAML 结构化(`description` 5412 字,9 顶层键)

> 源:[01 §一](01-真实人物卡分析.md)。9 顶层键:`game_info`/`world_setting`/`resources`/`pill_system`/`stable_resources`/`social_system`/`gameplay_modules`/`game_features`/`roleplay_settings`。

| 万域顶层键 | 性质 | dicelore 落位 | 判据 |
|------------|------|---------------|------|
| `game_info` / `world_setting` / `resources` / `pill_system` / `stable_resources` / `social_system` | 世界观散文(势力/资源/丹品/法则) | **lore 域**(FTS5 散文设定,deny-by-default) | 叙事氛围、可被任意改写不破坏规则 → 自由文本([02 §五问题a 实操判据](02-论文结论.md)) |
| `gameplay_modules` / `game_features` | 机制定义(玩法模块/游戏特性) | **rule + pool + params**(机制下沉结构化载体) | 影响合法性判定/被引擎引用 → 结构化([02 §五问题a 实操判据](02-论文结论.md));呼应 [04 §D.1 共指#3](04-差距分析.md) 机制不该拍平进 rule 散文 |
| `roleplay_settings`(输出语言/人称/字数/文风) | 协议耦合(COT Phase 0.2 从上下文提取,[01 §一.3](01-真实人物卡分析.md)) | **manifest 富字段 + skill references**,不进 lore | 输出规范是裁决/呈现约束非世界观散文;且 lore 是 deny-by-default 被动召回,放输出规范会召回不稳 |

**是否新增结构化设定域**:**不新增**。dicelore 已有 lore(散文) + pool(结构行) + state(机械态)三分,覆盖万域设定层的全部语义。万域的 YAML 骨架之所以"看着像结构化设定域",是因为 silly_tavern 无 lore/pool 二分,只能塞进 `description`;dicelore 已有更细的域划分,无需新增。关键是不把机制定义误塞 lore(那是 gacha-summon 把品质表塞 rule 散文的同类病灶,[03 §五.1](03-现状分析.md))。

| Step 2.3 三层门禁(物理可行/难度合规/代价相称) | 合法性裁决 | **L1 门禁工具**(见 1.4),不进 skill 散文 | L1 |

**协议优先级工程**(万域 io=9002 最高确保协议最贴近用户输入,[01 §五.3](01-真实人物卡分析.md)):dicelore 不靠 prompt 注入优先级,而靠 skill 加载顺序 + L1 工具强制替代——工具结构上绕不过,比 prompt 优先级更可靠([04 §B.2](04-差距分析.md) L1 更强)。

### 1.3 难度系统(10 档 × 4 模块 = 40 段)

> 源:[01 §三](01-真实人物卡分析.md)。10 档(一星~十星)线性强度轴,4 模块(天道衍算/因果契约/众生心智/越阶天堑),默认五星。全 `constant=true` 注入。dicelore 现状:无难度档位([04 §B.1 第一项](04-差距分析.md) 缺失)。

**迁移路径:manifest 富字段 + 引擎参数化**,顺修 flows 断链。

| 万域 4 模块 | 万域实现(prompt 禁令) | dicelore 落位(引擎可消费参数) | 层级 |
|-------------|----------------------|-------------------------------|------|
| 天道衍算(合理性审查) | L2 禁令散文 | **legality mask 约束**(物理可行性边界,境界天堑阈值) → L1 门禁工具查表 | L1 |
| 因果契约(代价换算) | L2 禁令散文 | **代价相称性阈值**(收益/付出比) → `resolve_outcome` 后置校验参数 | L1 |
| 众生心智(NPC 图谱) | L2 禁令散文 | **NPC 行为约束 schema**(npc entity 的 behavior 字段) → 类型化工具承载 | L1+L2 |
| 越阶天堑(破境难度) | L2 禁令散文 | **DC/bands 偏移表** → `resolve_outcome.bands` / `rollDice` DC 随难度查表 | L1 |

**关键设计:参数化压缩,不照搬 10 档全量条目**。万域 10 档各 ~650 字独立全量条目、无参数化压缩、作者改一处需改 10 处([01 §八.4 #4](01-真实人物卡分析.md))是欠账。dicelore 应:基线五星(物理守恒铁律,等价万域默认档)+ 偏移向量(豁免方向 / 苛审方向两个轴的强度参数),引擎按偏移查表调 DC/bands/代价阈值。档位是偏移向量的预设值,非独立全量条目。

**"禁止难度软化"(L2 ⑨)落 L1**:万域靠 prompt 教"不得淡化威胁/延缓惩罚/无故施恩",dicelore 把"威胁淡化检测"做成 L3 auditTurn 硬指标 + 代价相称性 L1 后置校验——骰完后引擎校验收益是否超付出比,超则强制补代价 mutation(消耗品清零/伤势加重等由 watcher 落地)。

### 1.4 三层难度门禁 → 引擎 L1 门禁工具(闸 B 前加一道)

> 源:[01 §二.3](01-真实人物卡分析.md)。三层(物理可行性→难度合规→代价相称),任一层否→【不可执行】禁入创作,不可跳过/合并/追认。万域是 L2 prompt 教条,被 [04 §A.2](04-差距分析.md) 否定层级(PTCG legal-masking −118 指向 L1 引擎强制)。

**迁移判据:不照搬 prompt,落成 L1 门禁工具**。

dicelore 现状:闸 A(能动性:谁拥有决定)+ 闸 B(该不该骰:不确定 ∧ 失败有意义)两闸,L2 教条为主([04 §B.1 第二项](04-差距分析.md) 更粗)。

迁移在**闸 B 前加一道 legality check**(L1 工具):

```
闸 A(能动性) → [新]legality check(物理可行 + 难度合规) → 闸 B(该不该骰) → [新]代价相称后置校验
```

- **物理可行性 + 难度合规**(前置 mask):借鉴 PTCG `available_actions` authoritative 思路([02 §二.5](02-论文结论.md)),引擎据 entity schema(境界/能力边界)+ difficulty 参数(越阶天堑阈值)计算合法行动集,集外行动引擎层即拒,不进入 LLM 上下文污染后续推理。这是 [04 §D.1 共指#1](04-差距分析.md) 裁决规则需显式成体系的落点。
- **代价相称性**(后置校验):人物卡独创维度([04 §D.2](04-差距分析.md) 单源)。骰判后、mutation 前,引擎校验"收益是否超本难度投入产出比",超则强制补代价 mutation(从 difficulty 因果契约阈值查表)。这是 L1 强制,非 L2 教条。

**【不可执行】叙事处理**(万域:禁 OOC 拒绝,叙事呈现失败/反噬/天道干预,且失败须产生后果,[01 §二.3](01-真实人物卡分析.md)):dicelore 已有机制承接——失败由 `narrate` 呈现,后果由 `sheet_update` + watcher 落地(灵力白耗/暴露/伤势加重等 mutation)。门禁产出"不可执行"标记进 event log,L3 可审计。

### 1.5 七/九模块状态 schema → sheet/state(已对齐,补 entity schema)

> 源:[01 §四.1](01-真实人物卡分析.md)。9 模块,模块 1-7 有完整 `check`(何时更新/数值边界/单次变化幅度),模块 8-9 仅靠输出格式示例定义(双源漂移)。字段带数值边界(好感度 -100~100、资质 -100~1000、互动记录保留最新 5 条)。

**已对齐**:结构化状态 + 工具改状态 = `FG-all` 最佳配置([04 §C.2](04-差距分析.md) 符合)。dicelore `sheet_update` 工具调用直接写结构化 `state` 表,不走 regex 提取文本块,优于万域 `FG-gen` 变体([04 §B.2 第一项](04-差距分析.md) L1 更强)。

**补 entity schema**:dicelore 现状 state 表六概念(player/npc/world/relation/flag/clock)塌一张表,`kind`+行形态分化,无逐字段 check 规则、无数值边界 schema([04 §B.1 第六项](04-差距分析.md) 更粗)。借鉴万域"每字段附 check"([01 §八.3 #4](01-真实人物卡分析.md)):

- 补 entity schema:每实体类型(player/npc/world/relation)带 attr 边界(min/max/单次变化幅度)+ check 谓词(何时更新)。
- 承载方式:走 [主题 A′](03-现状分析.md) 类型化工具(`player_card`/`npc_list`/`world_state` + update)替代裸 `sheet_*`,而非新增域。entity schema 是类型化工具的守卫。
- **不照搬 9 模块**(那是修仙品类特定)。dicelore 的实体类型按 canonical 六概念 + 团本自定义 entity(经 toolgen 声明)。

**双源漂移教训**:万域模块 8-9(修仙万象/其他)仅靠输出格式 `// Example` 定义 schema,与规则单源冲突([01 §四.1 架构观察](01-真实人物卡分析.md))。dicelore 迁移时**单源**:entity schema 只在 entity 定义一处,不在示例/prompt 重复。

**八卦九卷铁律**(每轮强制更新≥1 卷第三方传闻,≥80 字,[01 §四.1](01-真实人物卡分析.md) 模块 7):dicelore 已有更强机制——watcher 表 edge-triggered([04 §B.2 第四项](04-差距分析.md))。设"周期强制更新"watcher(condition: 回合钟推进 → 触发 world_register 第三方传闻),L1 强制自驱,优于万域 L2 教条。这把"世界鲜活感"从"教 AI 自觉"升级为"引擎自驱"。

### 1.6 JSON Patch(replace/delta/insert/remove)→ sheet_update 操作语义

> 源:[01 §四.2](01-真实分析.md)。RFC6902 变体(replace/delta/insert/remove),非标准 `delta`,靠 regex 提取 + 外部 MVU JS 引擎应用。

**已对齐且更强**:`sheet_update` 工具调用 + `applyMutations` 整批事务 + 账本标 `rolled`/`set`([04 §C.2](04-差距分析.md) 符合 FG-all)。不走 regex 提取(被 [04 §A.2](04-差距分析.md) 否定),不依赖外部 MVU 引擎(强外部依赖是万域最大障碍,[01 §八.4 #1](01-真实人物卡分析.md))。

**操作语义借鉴**:

| 万域 op | 语义 | dicelore 落位 |
|---------|------|---------------|
| `replace` | 替换值 | `sheet_update` 已有(set) |
| `delta` | 数值增减(非标准) | `sheet_update` mutation 补 delta op(账本标 delta,区别于 set/rolled) |
| `insert` | 插入完整对象(array 用 `-` 追加) | `sheet_update` 已有(整批事务);insert 完整性由 entity schema 校验 |
| `remove` | 删除路径 | `sheet_update` 已有 |

**三大铁律落 L1 校验**(万域靠 prompt 教,[01 §四.4](01-真实人物卡分析.md)):

| 万域铁律 | dicelore 落位 | 层级 |
|----------|---------------|------|
| 完整性(insert value 须完整对象) | `sheet_update` 时 entity schema 校验(缺字段拒绝) | L1 |
| 同步性(位置↔坐标↔区域名同步) | `sheet_update` 时引擎校验关联字段同步(声明 sync group) | L1 |
| 时效性(消耗品清零即 remove/任务完成即 remove) | watcher edge-triggered(归 0 触发 remove mutation) | L1(已有 watcher) |

### 1.7 关键词触发场景(164 条触发条目,401 个 keys)

> 源:[01 §五.4](01-真实人物卡分析.md)。171 条 `selective=true` + 401 个 keys,关键词命中即注入对应世界书条目。dicelore 现状:lore 表 deny-by-default,靠 `world_search` FTS5 主动召回,无"关键词命中自动注入"([04 §B.1 第三项](04-差距分析.md) 缺失)。这是单源(仅人物卡指向,论文未覆盖,[04 §D.2](04-差距分析.md))。

**迁移判据:不给 lore 加 keys 字段做静默注入,改走 watcher + world_show 通路**。

理由:
- lore 的 deny-by-default + 主动召回是更受控的设计(避免上下文膨胀,呼应 [02 §六启示6](02-论文结论.md) 上下文管理)。给 lore 加 keys 静默注入会破坏 deny-by-default 单源。
- 万域的关键词触发是 silly_tavern 世界书机制产物,`use_regex` 全 true 但 keys 多为普通字符串、正则能力未被利用([01 §八.4 #5](01-真实人物卡分析.md))——dicelore 的 FTS5+jieba 召回已够。
- 但"进入区域即注入区域设定""提及 NPC 即注入 NPC 卡"有场景价值。

**落法**:关键词触发场景作为**第四类内容(场景触发)走 watcher 通路**——"进入区域 X"/"提及 NPC Y"作为 watcher condition,触发时 `world_register` + 前端 `reveal_once`/`world_show`。即把万域的"prompt 静默注入"升级为"L1 可审计事件":触发进 event log,可见性走 visible 横切,玩家可见可控。这比万域的静默注入更工程化,且不破坏 deny-by-default。

### 1.8 regex 前端解析(9 regex_scripts + 211KB HTML 三栏 UI)

### 1.9 depth_prompt(depth=4 注入)

> 源:[01 §七.1](01-真实人物卡分析.md)。`depth=4` 在聊天历史倒数第 4 条注入(接近底部,强影响力),`prompt=""` 当前为空占位。COT 靠世界书 io=9002 注入已足够接近底部,depth_prompt 沦为空占位。

## 三 迁移风险与张力

### 3.1 prompt 堆叠 vs 工具强制的张力

**张力**:万域 COT 6987 字 prompt 堆叠,靠 io=9002 注入优先级确保"协议不可跳过"。论文否定 L2 教条(PTCG −118 指向 L1 引擎强制,[02 §二.5](02-论文结论.md))。但 COT 的推理编排(Step 3 四维评估/Step 4 硬约束/Step 5 叙事构建)难以完全机械化,部分必须留 L2 教条。

**张力点**:若把推理也塞进 L1 工具,会变成 Labyrinth `FG-states`(仅状态函数过度调用、破坏流程,[02 §一.3](02-论文结论.md));若全留 L2 prompt,又回到被否定的教条路线。

**判据(可机械判定 vs 需语义推理)**:
- **凡可机械判定的**(合法性边界 / 代价数值 / 字段边界 / 同步性)→ L1 工具(legality mask / 代价校验 / entity schema check)。
- **凡需语义推理的**(形势紧迫度评估 / 叙事走向 / 事件选择 / 反面检查)→ L2 flow skill references。
- **不可把语义推理伪装成机械判定塞进工具**(如"形势判断工具"会变成 FG-states 过度调用)。

**借鉴门禁不能照搬 prompt**:万域三层门禁的"常见违规识别清单"(无故瞬移/实力暴增/越阶碾压等,[01 §二.3](01-真实人物卡分析.md))是 L2 教条;迁移时把"瞬移距离>境界能力上限""收益>付出比阈值"这些**可量化部分**落 L1 mask,把"NPC 无故放水"这类**语义部分**留 L2 + L3 审计抽查。

### 3.2 掷骰密度矫正(corpus-distill 裁决)vs 叙事裁决派的张力

**张力**:万域是叙事裁决派(无骰子函数),被论文否定(`FG-states` 失衡,[04 §A.2](04-差距分析.md))。但 dicelore 有 corpus-distill 裁决要矫正掷骰密度(不能骰太密)。门禁是骰判前置预处理(非替代)——门禁拦掉"物理不可行/代价失衡"的行动,但不该把所有骰都拦掉(否则又变叙事裁决派)。

**张力点**:三层门禁的"代价相称性"若阈值过严,会退化成"门禁裁决一切"=无骰叙事派,与 dicelore anti-F1(必掷骰)冲突。

**判据(门禁 vs 骰判分工)**:
- **门禁只做两件事**:① 合法性 mask(该不该做——物理可行+难度合规);② 代价校验(做了该付多少——后置 mutation 校验)。
- **不确定性(成不成功)仍交闸 B 骰判**:门禁产出"不可执行"→叙事呈现失败(非骰判);"可执行但不确定"→仍走闸 B 骰判。
- **门禁不裁决成功概率**:成功概率是骰子+DC 的事,不是门禁的事。门禁拦的是"不该做",骰判裁的是"做不做得成"。
- **代价相称性是后置,非前置**:骰判出结果后,引擎校验"这个结果对应的收益是否超付出比",超则补代价 mutation。门禁不在骰前用代价相称性拦行动(那会变叙事裁决),只在骰后用代价相称性保证"失败有后果/成功有代价"。

这样门禁是骰判的前置预处理(拦非法)+ 后置校验(保代价),非替代骰判,与 corpus-distill 掷骰密度矫正不冲突——密度矫正管"该骰的频率",门禁管"骰之外的合法性/代价"。

### 3.3 gm-core 框架统一 vs 团本级定制 skill 的张力

**张力**:gm-core 是框架级单源(Agenda/Moves/Principles),团本级 flow skill 只编排工具序不新增工具([03 §二.1](03-现状分析.md))。万域 COT 是团本级完整协议(覆盖框架,12 全局禁令 + 7 Step)。团本级 skill 若太强会覆盖框架教条,破坏单源一致性。

**张力点**:COT 的 12 全局禁令中,既有与 gm-core Principles 重合的(⑤ 禁止主角中心化 ≈ anti-讨好 F2),也有品类特定的(④ 禁止违和=符合修仙世界观)。若全塞团本 skill,会与 gm-core 重复或冲突。

**判据(框架通用 vs 品类特定)**:
- **框架管机制通用教条**:gm-core Agenda(诚实仲裁者)/Moves(闸 A/B/形状表/谁掷)/Principles(F1 必掷骰/F2 双边护栏/F3 选对方式/明骰默认/可见性)是跨品类通用的,团本 skill **不覆盖**。
- **团本管品类特定编排**:COT 的修仙世界观禁令(④ 禁止违和)、境界体系优先级(铁律① 世界法则>境界体系>难度规则)、四维评估(事态紧迫度/主角影响力等修仙语境)、叙事走向判定(大势裹挟/因名而动等)——这些进团本 flow skill **references**,作 L2 教条。
- **重合部分不重复**:12 禁令中与 gm-core Principles 重合的(⑤禁主角中心化≈F2 anti-讨好、⑥禁过度反应≈烈度匹配)不重复写,引用 gm-core;品类特定的(⑦禁文学收束/⑧禁意象堆砌)进 references。
- **团本 skill 不新增工具**:flow skill 只编排已有工具调用序([03 §二.1](03-现状分析.md)),COT 的 Step 3/4/5 推理编排落 references 教 GM "怎么想",不新增"形势判断工具"。

这样 gm-core 保持框架单源一致性,团本 flow skill 承载品类特定编排,二者不冲突——框架是骨架,团本是血肉。

## 四 不破坏 dicelore 既有优势的约束

迁移中以下六处 dicelore 既有优势([04 §B.2](04-差距分析.md))不可丢。每条给出迁移时的守护约束。

### 4.1 工具强制 L1(随机真值引擎内)

**既有优势**:随机真值在引擎(`rollDice`/`rangeMap`/`commitPendingRoll`),AI 给不出真值;掷骰/选择/状态写必经工具,结构上绕不过([04 §B.2 第一项](04-差距分析.md))。

**迁移守护**:
- 所有"门禁/校验"落 L1 工具,不退化为 L2 prompt。万域的门禁是反例(L2 教条被否定,[04 §A.2](04-差距分析.md)),dicelore 必须保持 L1。
- 难度系统的 4 模块(天道衍算/因果契约/众生心智/越阶天堑)落引擎参数(legality mask / 代价阈值 / NPC schema / DC 偏移),不落 prompt 禁令散文。
- "骰值→档位→后果"映射必须机械化(`resolve_outcome.bands` / `rangeMap`),不在 GM 脑里做(修 [03 §五.2 病灶5](03-现状分析.md))。

### 4.2 visible 横切三态

**既有优势**:`visible` 列三态(0 默认隐/1 已 show/2 强制隐暗值)横切 sheet/world/event + 6 命名视图投影([04 §B.2 第二项](04-差距分析.md))。万域可见性只在 HTML UI 渲染层。

**迁移守护**:
- 迁移难度预设系统时,难度档位的"暗骰/明骰"仍走 visible 三态,不另建可见性机制。
- 关键词触发场景走 watcher + `world_show`/`reveal_once`,可见性走 visible 横切,玩家可见可控——不照搬万域的 prompt 静默注入(静默注入无可见性)。
- 八卦九卷铁律落 watcher 时,第三方传闻的可见性(visible=1 已 show / 2 强制隐暗值)由 visible 横切管,不另建。

### 4.3 版本化 Catalog

**既有优势**:Catalog DB + commit/tag/checkout;import 闸门 `checkout` → `validatePack` 重验 → 物化;rule 表带 `version` 列人类热更新([04 §B.2 第三项](04-差距分析.md))。万域单卡 JSON 无版本概念。

**迁移守护**:
- 难度预设 / entity schema / 团本 flow skill 变更要版本化:manifest difficulty 富字段随 Pack commit 进 Catalog,rule 表已有 `version` 列承载热更新。
- 不照搬万域"单卡 JSON 文件"形态——团本是版本化 Catalog 实体,非单卡。
- flows 断链修复后,manifest.flows 声明进版本化 Pack,import 闸门消费。

### 4.4 watcher 世界自驱

**既有优势**:watcher 表 edge-triggered,`sheet_update` 写后 `recomputeWatchers` 自动触发;front 凶兆 import 预声明 watcher([04 §B.2 第四项](04-差距分析.md))。万域八卦九卷靠 COT 教 AI "每轮至少更新 1 卷"(L2 教条)。

**迁移守护**:
- 八卦九卷铁律(每轮强制更新≥1 卷)落 watcher(condition: 回合钟推进 → 触发 world_register),L1 强制自驱,不退化为 L2 教条。
- 状态时效性铁律(消耗品清零即 remove/任务完成即 remove)落 watcher edge-triggered,不靠 prompt 教 AI 自觉删除。
- 关键词触发场景(第四类内容)走 watcher 通路,不引入静默注入——watcher 触发进 event log 可审计,静默注入不可审计。

### 4.5 声明式 toolgen

**既有优势**:`tools/*.json` → `compileTool` 守 DT-9 → `extraTools` 注入 DiceSession;团本可声明自定义 GM 工具([04 §B.2 第五项](04-差距分析.md))。万域无团本自定义工具,9 regex 是前端解析非 GM 工具。

**迁移守护**:
- 团本品类特定工具(如修仙的"突破检定""炼丹结算")走 toolgen 声明,不硬编进框架。
- entity schema 的品类特定字段(如修仙的境界/灵根/资质)经 toolgen 类型化工具承载,不污染框架 stdlib。
- 不照搬万域 regex_scripts——regex 是前端解析,toolgen 是 GM 工具,二者不同层。

### 4.6 明暗骰玩家亲手掷

**既有优势**:明骰经 `pending_roll` 槽 + `commitPendingRoll` 玩家点击时掷;暗骰 `_hidden` 引擎内掷([04 §B.2 第六项](04-差距分析.md))。万域无骰子,玩家不掷骰。

**迁移守护**:
- 门禁不能替代骰判(见 [§3.2](#32-掷骰密度矫正corpus-distill-裁决-vs-叙事裁决派的张力)):门禁产出"不可执行"→叙事呈现失败(非骰判);"可执行但不确定"→仍走明暗骰 + 玩家亲手掷。
- 难度系统的"越阶天堑"调 DC/bands,不取消骰子——难度调的是骰判难度,不是用门禁代替骰判。
