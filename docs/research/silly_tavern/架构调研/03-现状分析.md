# 03 · Dicelore 架构现状分析

> **本页职责**：摸清 dicelore 当前架构现状——四域物理 schema、play/build 双侧 skill 体系、工具链与 L1/L2/L3 强度杠杆、build→pack→import 全链路、真实团本实证。给文件路径行号佐证，不改代码。
> **术语**：遵 [`docs/wiki/术语表.md`](../../../wiki/术语表.md) canonical——逻辑层 `sheet`/`event`/`world`/`rule`，物理表 `state`/`log`/`lore`/`pool`/`rule`；`Adventure`=团本实体、`Pack`=文件打包层、`Catalog`=库。Front/Clock 大写=概念、小写=实现标识符。
> **调研时点**：2026-07-13，main 分支。

## 〇 概览

Dicelore 是「AI 当 GM 的文字冒险游戏框架」，两侧对称：

- **play 侧（dice）**：被嫁接的 AI 扮演 GM，经一组 `dicelore_*` MCP 工具读写 per-session SQLite 运行库，驱动掷骰/选择/状态/可见性/叙事。核心约束是 anti-F1（不可跳骰）/ anti-F2（不软着陆）/ anti-F3（选对机制），靠 L1 工具强制 + L2 skill 教条 + L3 hook 审计三杠杆。
- **build 侧（lore）**：构建 GM 把作者素材（小说/设定集）提炼成团本包（Pack = MD+CSV 文件），commit 进 Catalog DB，play 开局时 `importPack` 物化进运行库。

两侧物理隔离：build 会话只见 `dicelore_build_*` 工具，play 会话只见 `dicelore_*` 运行时工具，互不在场（[`harness/src/loregm/skills/dicelore-build-core/SKILL.md:22`](../../../harness/src/loregm/skills/dicelore-build-core/SKILL.md)）。

**一句话现状**：数据层（四域 + 叙事三表 + watcher + 快照）物理 schema 已成型且改名收口；L1 工具面 19 个内置工具可用但**框架 stdlib（叙事八工具/npc/player/world/memory 类型化工具）声明已写却未接进生产 MCP**——`narrationStdlibTools()`/`npcStdlibTools()` 等只定义导出、零生产调用（[`backend/src/index.ts:151-155`](../../../backend/src/index.ts)）；build 侧富字段（flows/clock/entry/front 凶兆）在 manifest.yaml 设计层定稿但 build 工具面产不出、import 闸门不消费；真实团本实证里 gacha-summon 把 d100 品质表/融合结算/周期事件全塞进单条 `write_rule` 散文，完全没用 `add_pool`/`set_state`/`add_front`。

### 1.3 实际用法观察（dice session.db 实证）

扫 `.dicelore-eval/sessions/dice/*/session.db`，多数是 eval fixture（`adventure_name=eval-fixture`，state 1 行、log 0-2 行、front/plotline/foreshadow/pool/lore 多为 0）。真实多轮 play 数据稀薄——`demo` 库全空，最「富」的库仅 2 条 log（`choice`×1 + `note`×1）。说明：**跑团侧真 LLM 多轮 play 仍未规模化跑起来**（呼应 [backlog-core 主题F](../../../wiki/设计/05-现状与计划/backlog-core.md) F2「终局观测待多轮跑」、F4「真 GM 不遵守开局教条」）。

## 二 skill 体系

### 2.1 play 侧（`harness/src/dicegm/skills/`）

**gm-core**（[`dicelore-gm-core/SKILL.md`](../../../harness/src/dicegm/skills/dicelore-gm-core/SKILL.md)）：三段式 PbtA 借词。
- **Agenda**（北极星）：第 0 条「你是诚实仲裁者、不是取悦者」凌驾一切；第 1-3 条描绘呼吸的世界/选择有重量/play to find out（`SKILL.md:10-17`）。
- **Moves**（动作 + 判定时机）：闸 A（能动性：谁拥有决定）→ 闸 B（该不该骰：不确定 ∧ 失败有意义）→ 形状表（label/verdict/number/content 四家族）→ 谁掷（明骰 `_open` / 暗骰 `_hidden`）（`SKILL.md:19-58`）。关键反保守默认：「推进叙事先问该不该骰、再问要不要给选项」，闸 B 该骰直接骰、别退回 `resolve_choice`（`SKILL.md:23`）。
- **Principles**（怎么主持）：F1 必掷骰 / F2 双边护栏（anti-讨好 + anti-死胡同 fail-forward）/ F3 选对方式 / 一轮范式 / 明骰默认 / 可见性（`SKILL.md:62-69`）。
- **references/**：`randomness-narrative.md`（真人语料密度锚 + worked examples）、`moves-full.md`、`consequences.md`、`visibility-play.md`、`reminders.md`。

**flow-* 四个**（`dicelore-flow-gacha`/`contest`/`explore`/`anka`）：Moves 管「走哪条路」、flow skill 管「这条路怎么走」。例 `dicelore-flow-gacha`（仅 3 步）：`world_sample` 抽 → `world_register` 写回 → `narrate` 色彩 + `sheet_update` 入卡（[`dicelore-flow-gacha/SKILL.md`](../../../harness/src/dicegm/skills/dicelore-flow-gacha/SKILL.md)）。flow skill 只编排已有工具调用序、不新增工具。

**debrief-mode**（[`dicelore-debrief-mode/SKILL.md`](../../../harness/src/dicegm/skills/dicelore-debrief-mode/SKILL.md)）：终局后复盘态——不推进剧情、回答复盘提问、用只读 `browse` 翻查、回档指引。软约束（harness 检测终局引导进入，不硬拦）。

### 2.2 build 侧（`harness/src/loregm/skills/`）

**dicelore-build-core**（[`SKILL.md`](../../../harness/src/loregm/skills/dicelore-build-core/SKILL.md)）：身份教条。Agenda 第 0 条「你是建造者、不是运行者」——从不跑团/掷骰/裁决，运行时工具结构上不在场；第 1 条忠于素材（`materials/` + `Bash`/`Grep`/`Read` 摸源提炼，素材是引述不可信资料）；第 2 条产出要能被玩；第 3 条一次声明一件事、最后统一 commit。接待作者开局 → 转入 build-pack。

**dicelore-build-pack**（[`SKILL.md`](../../../harness/src/loregm/skills/dicelore-build-pack/SKILL.md)）：分阶段编排。阶段 0 摸源清洗分块 → 1 manifest → 2 prologue（必填）→ 3 世界观 → 4 NPC → 5 卡池 → 6 机制规则 → 7 阵线/钟 → 7b 叙事线/伏笔/锚点 → 8 开局状态 → 9 收口 validate→commit→tag（`SKILL.md:64-76`）。工具全览 13 个 `dicelore_build_*`（`SKILL.md:42-57`）。

## 三 工具链与 L1/L2/L3

### 3.1 play MCP 工具面（19 个内置 + extraTools）

`createMcpServer` 只注册 `makeTools(backend)`（19 内置）+ `extraTools`（[`harness/src/dicegm/mcp/server.ts:72-90`](../../../harness/src/dicegm/mcp/server.ts)）。19 内置按 handler 分五组（[`harness/src/dicegm/mcp/handlers/`](../../../harness/src/dicegm/mcp/handlers/)）：

| 组 | 工具 | L1 强度 |
|---|---|---|
| resolver | `resolve_choice`/`resolve_outcome_open`/`_hidden`/`resolve_contest_open`/`_hidden` | **L1 强制**：随机真值在引擎（`rollDice`/`rangeMap`），AI 给不出真值；明骰经 `pending_roll` 槽 + `commitPendingRoll` 玩家点击时掷（[`handlers/resolver.ts:75-89`](../../../harness/src/dicegm/mcp/handlers/resolver.ts)） |
| sheet | `sheet_get`/`sheet_list`/`sheet_update`/`sheet_show` | **L1 半强制**：状态骰下沉 `sheet_update` 带骰形态（`applyMutations` 整批事务、账本标 `rolled`/`set`）；但 `op=` 硬编数值与带骰的区分降级为 L2 教 + L3 抓 |
| event | `event_append`/`event_recall` | L1：append-only log，`seq` 单调 |
| world | `world_register`/`world_sample`/`world_search`/`world_show` | L1：`world_sample` 抽样结果空间来自 store（不编造） |
| io | `narrate`/`reveal_once`/`watcher_set`/`watcher_list` | L1：`narrate` 落 `kind=narrate` event 才能审计/召回；`watcher_set` 创建 edge-triggered 触发器 |

### 3.2 L1/L2/L3 落点

| 杠杆 | 机制 | 现状 |
|---|---|---|
| **L1 工具强制** | 随机真值引擎内（`rollDice`/`rangeMap`/`commitPendingRoll`）、`sheet_update` 带骰事务、`world_sample` 抽样、`narrate` 落 event、watcher 就地重算 | ✅ 落地。掷骰/选择/状态写必经工具，AI 结构上绕不过 |
| **L2 skill 教条** | gm-core Agenda/Principles/Moves + flow-* + build-core/build-pack | ✅ 落地。但教条措辞 eval-pending（`SKILL.md:6`），F4 实证「真 GM(glm-5.2)读了教条仍 0 次明骰、建世界空转 180s 超时」（[backlog-core F4](../../../wiki/设计/05-现状与计划/backlog-core.md)） |
| **L3 hook 审计** | turn-start hook 被动 rule 召回（`recallRules`）+ turn-end hook `auditTurn`（[`adapter/l3.ts`](../../../harness/src/dicegm/adapter/l3.ts)） | 🟡 薄。`auditTurn` 仅两条档 A block：① 非终局轮 `pendingChoiceEmpty` → block；② 有 transcript 文本却无 narrate → block；档 B 只写统计 note（机械事件数 vs narrate 数）。**不检测 F1「该骰却没骰」**（`verdictCount=0` 纯 narrate 编结果不会 fail，[backlog-core F1-floor](../../../wiki/设计/05-现状与计划/backlog-core.md)）、**不检测 F2 软着陆** |

### 3.3 框架 stdlib 工具：声明已写、生产未接（重大断节）

`backend/src/stdlib/` 下五个 stdlib 模块声明了类型化/叙事工具：
- `narration.ts`：`front_open`/`plotline_open`·`advance`·`close`/`foreshadow_plant`·`recall`·`abandon`/`tension_board` 八工具（[`backend/src/stdlib/narration.ts:17`](../../../backend/src/stdlib/narration.ts)）
- `npc.ts`：`npc_register`/`npc_update_affinity`/`npc_update_hp`/`npc_set_identity`（[`backend/src/stdlib/npc.ts:75`](../../../backend/src/stdlib/npc.ts)）

每个模块导出 `*StdlibTools()` 编译函数（[`backend/src/index.ts:151-155`](../../../backend/src/index.ts)），但**全仓非测试代码零调用**——`createMcpServer` 不注入、`DiceSession` 构造 `extraTools` 仅来自团本 `tools/*.json`（[`DiceSession.ts:79-103`](../../../harness/src/dicegm/DiceSession.ts)）。后果：**生产跑团 GM 调不到 `front_open`/`plotline_*`/`foreshadow_*`/`tension_board`/`npc_*`**——叙事三表有存储、有 status 生命周期、有聚合视图，但 GM 无工具开/推进/收口它们。这正是 [backlog-core 主题A](../../../wiki/设计/05-现状与计划/backlog-core.md)「CROSS-TOOLGEN P0」的活体断节：引擎建好、dogfooding 测试跑通，但生产接线未做。`present/tensionBoard`、`present/playerView` 同为孤儿模块（无运行时消费者）。

## 四 build→pack→import 全链路（含 flows 断链）

### 4.1 build 侧：Draft 累积 → toPackFiles → commitDraft

`Draft` 类按域累积（lore/rule/pool/state/front/plotline/foreshadow/anchor/manifest/prologue），`toPackFiles()` 序列化成包文件（[`backend/src/build/draft.ts:142-171`](../../../backend/src/build/draft.ts)）。`commitDraft` 经 `commit()` 落 Catalog（[`draft.ts:175-180`](../../../backend/src/build/draft.ts)）。

**build 工具面产不出 manifest 富字段**：`set_manifest({name,id})` 仅写 name/id（[`buildMcp.ts:66`](../../../backend/src/build/buildMcp.ts)），`Draft.setManifest` 只存这两个字段（[`draft.ts:86-90`](../../../backend/src/build/draft.ts)）。`toPackFiles` 产出的是 `manifest.md`（H1 标题 + id 列表，[`draft.ts:145`](../../../backend/src/build/draft.ts)），**不是 manifest.yaml**。故 `manifest.yaml` 设计层声明的 `flows`/`clock`/`entry`/`version`/`description` 字段（[`团本与manifest.md §2`](../../../wiki/设计/04-子系统设计/团本与manifest.md)）——build 工具面根本产不出。

### 4.2 pack 格式校验：validatePack（`backend/src/build/pack/validate.ts`）

7+ 规则：顶层段白名单（`KNOWN_TOP`，含 `manifest.yaml`/`manifest.md`/`prologue.md`/`tools` 等，[`validate.ts:27-35`](../../../backend/src/build/pack/validate.ts)）、`prologue.md` 必填（Rule 0c，[`validate.ts:258-261`](../../../backend/src/build/pack/validate.ts)）、state/sheets CSV 列、叙事域 CSV 必需列、`tools/*.json` 经 `compileTool` 守 DT-9（Rule 8，[`validate.ts:300-329`](../../../backend/src/build/pack/validate.ts)）、`manifest.yaml` 字段校验（Rules 1-4，[`validate.ts:331-392`](../../../backend/src/build/pack/validate.ts)）、`fronts/*.md` frontmatter + 凶兆阶梯（Rules 6-7，[`validate.ts:394-454`](../../../backend/src/build/pack/validate.ts)）。

**注意**：`manifest.yaml` 在此仅做 DX 校验，**import 闸门不消费**（[`validate.ts:33-34` 注](../../../backend/src/build/pack/validate.ts)：「manifest.yaml 仅 DX 校验，import 闸门暂未消费；终态待 ADR」）。

### 4.3 import 闸门：importPack（`backend/src/catalog/import.ts`）

`checkout` → `validatePack` 重验（信任闸门，[`import.ts:111-116`](../../../backend/src/catalog/import.ts)）→ 按顶层段物化：

| `rules/*.md` | `ruleUpsert` | ✅ |

| `fronts/*.md` | `frontUpsert` + `stateSetClock` + **凶兆→`watcherSet` 预声明** | ✅ **凶兆物化已落地**（[`import.ts:148-177`](../../../backend/src/catalog/import.ts)）——`source='front:<id>'`、`condition={clock} >= threshold` |

### 4.4 flows/clock/entry 富字段断链（设计有、build 产不出、import 不消费）

| 字段 | 设计层（manifest.yaml） | build 工具面 | import 闸门 | 运行时消费 |
|---|---|---|---|---|
| `flows`（选 flow skill） | [`团本与manifest.md §2`](../../../wiki/设计/04-子系统设计/团本与manifest.md) | ❌ `set_manifest` 无此参 | ❌ 不消费 | ❌ 无机制据 flows 加载/限制 flow skill |
| `clock`（钟 attr 声明） | 同上 | ❌ | ❌ | ❌ watcher 比对靠 front `clock_attr` 兜底，manifest.clock 无运行时作用 |
| `entry`（开局引子锚点） | 同上 | ❌ | ❌ | ❌ prologue 已接手开场，entry 无消费方 |
| `version`（semver） | 同上 | ❌ | ❌ | ❌ 热更新迁移未实现 |

**断链根因**：build 侧用 `manifest.md`（H1 + id）而非 `manifest.yaml`，设计层的 YAML 富字段无生产路径产出；import 侧 `KNOWN_TOP` 虽含 `manifest.yaml` 做 DX 校验，但物化循环无 `manifest.yaml` 分支。整条 flows/clock/entry 链路是「设计页定稿、代码未跟进」的活体断节。

### 4.5 front 凶兆→watcher 物化：已落地（与 backlog 标注差异）

[backlog-core 主题A A2](../../../wiki/设计/05-现状与计划/backlog-core.md) 与 [团本构建工具链 §6](../../../wiki/设计/04-子系统设计/团本构建工具链.md) 均标注「凶兆阶梯 → watcher 物化仍欠（留主题 A2）」，但实测 `import.ts:169-177` **已物化**：每条 omen → `watcherSet({condition:'{clock} >= threshold', payload, mode, source:'front:<id>'})`。这是 wiki 与实现已对齐但 backlog 文字未回改的漂移（属 [M1 设计-实现漂移](../../../wiki/设计/05-现状与计划/backlog-core.md) 类）。`frontier-saga` eval 场景 `expects.watchers=3` 也印证已落地（[`backend/eval/loreScenarios/frontier-saga.json:120`](../../../backend/eval/loreScenarios/frontier-saga.json)）。

## 五 真实团本实证：rule 病灶五条

### 5.1 gacha-summon 实证

`.dicelore-eval/sessions/lore/gacha-v2/gacha-v2_session.jsonl` 是构建 GM（glm-5.2）把真实安价《总之，来抽卡吧》实跑串（`materials/gacha-source.md`，10717 行）提炼成团本的真实记录。该会话仅跑 1 turn（`gacha-v2-l1`），调用的 build 工具：`set_manifest`/`set_prologue`/`write_lore`/`write_rule`/`read` 各 1 次——**未调 `add_pool`/`set_state`/`add_front`/`commit`/`tag`**（会话未收口提交）。

**关键实证**：构建 GM 把全部机制塞进单条 `write_rule({name:"抽卡机制"})`，content 1711 字（提取自 jsonl）。该 rule 散文内含：
- **d100 品质表**（1-15 平凡 / 16-30 稀有 / … / 91-100 无上，七档 markdown 表）——应落 `add_pool` 或 `params/*.csv` 分档表，供 `resolve_outcome` 的 `bands` 或 `world_sample` 消费
- **融合结算**（r1d100：<10 大成功 / <50 成功 / 50-90 失败毁一素材 / >90 大失败全毁）——应落 `params/*.csv` 分档表
- **周期事件**（每 10 回合世界巨变 r1d100 查表 / 每 15 回合世界意志卡 / 系列>15 文明检定）——应落 `add_front`（钟 + 凶兆阶梯 watcher）+ `set_state`（回合钟 attr）
- **成功层级**（普通/困难/极限/大成功）+ **卡牌特质跨阶生效**检定——应落 `params/*.csv`

完全没用 `add_pool`（卡池/随机表）、`set_state`（世界回合钟）、`add_front`（周期威胁 watcher）。整本机制变成一条 rule 散文，运行时 dice GM 须「整段读、自己套用」。

### 5.2 rule 运行时兑现通路：五条病灶

rule 域设计为「被动、整段读的约束，AI 只读，不抽样不过滤」（[`内层能力库.md §4.4`](../../../wiki/设计/04-子系统设计/内层能力库.md)）。兑现通路现状：

1. **不常驻**：rule 不进 system prompt、不进 openingPrompt（[`术语表.md openingPrompt 条`](../../../wiki/术语表.md)：openingPrompt = signpost + prologue，教条经 skill 单独投递）。rule 只在 turn-start hook 被动召回时进上下文。
2. **词面召回**：`recallRules`（[`adapter/ruleRecall.ts:13-18`](../../../harness/src/dicegm/adapter/ruleRecall.ts)）走 `backend.ruleSearch(prompt, 5)`——FTS5 关键词召回，top 5 条整段 content 拼进「相关团本规则(被动召回,务必遵守)」前缀。召回质量靠 jieba 分词 + bm25，对「融合检定」「品质表」这类语义召回无保证。
3. **gm-core 不教查 rule**：gm-core SKILL.md 全文无「调 `event_recall`/读 rule」的动作教条；Moves 形状表只教「骰什么→哪个工具」，不教「机制在哪→先查 rule」。flow-gacha 仅 3 步、不提 rule。故 GM 不会主动查 rule，全靠 turn-start hook 被动注入碰运气。
4. **软约束**：rule 召回进上下文后，遵守与否全靠 L2 教条（「务必遵守」前缀），无 L1 强制——GM 完全可以无视召回的 rule 散文、自行编造结算。F4 实证「GM 读了教条仍不照做」印证软约束不可靠。
5. **结构化数据拍平进散文**：品质表/分档表本应是 `pool`/`params`（可抽样/可被 `resolve_outcome.bands` 消费的结构），拍平进 rule markdown 表后，GM 须肉眼读表+心算档位，无法经 `rangeMap` 机械落档——anti-F1 的「随机真值在引擎」对 rule 内表格失效（骰子虽在引擎掷，但「骰值→档位→后果」的映射在 GM 脑里做，可被讨好本能污染）。

**五条病灶合起来**：当机制被塞进 rule 散文（如 gacha-summon），运行时兑现完全依赖「被动召回命中 + GM 自觉遵守 + GM 正确心算映射」三重软约束，L1 强制退化为 L2 教条，anti-F1/F2 防线被绕过。这是 build 侧富字段断链（build GM 默认 `write_rule` 一把梭）与 play 侧 rule 兑现通路薄弱的合流病灶。
