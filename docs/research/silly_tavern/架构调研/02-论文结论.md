# 02 论文结论:结构化状态 vs 万物皆文本

> 调研对象:三篇主论文 + 四篇补充论文,聚焦"LLM 游戏状态架构"。
> 三篇主论文:Labyrinth(2409.06949,function calling 增强 AI GM)/ PTCG-Bench(2605.29653,状态格式消融 + legal-action masking)/ Game Multiverse(2605.09965,三表示 + Harness Paradox)。
> 四篇补充:PAYADOR 2504.07304 / Static vs Agentic GM 2502.19519 / DnD Reasoning 2510.18112 / AI-Native Games Survey 2607.00527。

## 〇 概览

三篇主论文从三个互补角度回答同一组问题:**游戏状态该用什么表示、谁来更新、用什么接口校验**。

| 论文 | 视角 | 核心实验 | 一句话结论 |
|------|------|----------|------------|
| Labyrinth 2409.06949 | AI GM 的 function calling | 6 组对照(FG-all/dice/states/default/gen/DG),7 评委 Likert + 30 单元测试 + p 值 | function calling 更新结构化状态,一致性/正确率显著优于 GPT-4 自行生成状态(单测 0.422 vs 0.267,p=0.0025) |
| PTCG-Bench 2605.29653 | harness 模块消融 | Glicko-2 评分,逐个拆 structured observation / legal-action masking / history | legal-action masking 杠杆最大(−118),是结构化状态格式(−33)的 3.6×;引擎强制校验 >> 状态格式 |
| Game Multiverse 2605.09965 | 综述 + 五权衡 + 五级路线图 | 文献综合 | Harness Paradox:结构化+工具是当前必要脚手架,但引入脆弱性;Breadth vs Depth 不可兼得 |

三篇合起来给出的总图:**状态不是"结构化 vs 文本"二选一,而是分层混合;一致性主要由"谁校验、用什么接口"决定,而非"用什么格式存";当前阶段结构化 harness 是承重墙,不可省。**

## 一 Labyrinth:function calling 增强 AI Game Master

> arXiv:2409.06949v1,2024-09-11。场景:桌游《魔幻迷宫》TTRPG 的 AI GM 模拟。模型:GPT-4。

### 1.1 方法:6 组 GM 设置

论文把"状态怎么更新 + 用不用函数"拆成 6 组对照,这是本研究最有价值的实验设计:

| 设置 | 状态 | 函数 | 含义 |
|------|------|------|------|
| **FG-all** | 全部状态 | 骰子+状态函数 | 完整 function calling |
| FG-dice | 全部状态 | 仅骰子函数 | 只用骰子函数 |
| FG-states | 全部状态 | 仅状态函数 | 只用状态函数 |
| FG-default | 全部状态(初始化后冻结) | 无函数 | 状态不变 |
| **FG-gen** | 全部状态,由 GPT-4 自行生成更新 | 无函数 | "万物皆文本"基线——状态以文本生成 |
| DG | 无状态 | 无函数 | 纯叙事,只看规则 |

`FG-gen` 是关键的对照锚点:状态全部以自然语言生成、不走函数接口,等价于"纯 prompt 协议 + 万物皆文本"。`FG-all` 等价于"结构化状态 + function calling 接口"。两者的差就是本任务问题(a)(b)的核心答案。

数据规模:144 个脚本、24 场景、4937 条话语、620 次函数调用(表 1)。

### 1.2 实验数据

**(1) 人工评测**(7 评委,Likert 1–5,3 次采样取总均分,表 2):

| 维度 | FG-all | FG-dice | FG-states | FG-default | FG-gen | DG |
|------|--------|---------|-----------|------------|--------|----|
| Consistency(一致性) | **4.388** | 3.983 | 3.358 | 3.420 | 3.698 | 3.691 |
| Reliability(规则遵循) | **3.949** | 3.650 | 3.272 | 3.364 | 3.628 | 3.855 |
| Interest(趣味) | **3.765** | 3.400 | 3.444 | 2.795 | 2.942 | 3.364 |

- `FG-all` 在三个维度全部第一。
- 注意 `DG`(无状态纯叙事)在 Reliability 排第二(3.855)——因为它没有状态干扰,反而更频繁地显式陈述规则、纠正玩家,显得"严格";但这是幻觉式严格(评委也承认其正确性存疑)。这提示:**状态多未必更可靠,关键在状态是否被正确维护**。

**(2) 单元测试**(30 例,状态更新正确率,表 3,仅测 FG-all / FG-states / FG-gen):

| 采样 | FG-all | FG-states | FG-gen |
|------|--------|-----------|--------|
| 1 | 0.400 | 0.433 | 0.233 |
| 2 | 0.417 | 0.483 | 0.300 |
| 3 | 0.450 | 0.467 | 0.267 |
| **均值** | **0.422** | **0.461** | **0.267** |

- `FG-all`(function calling)0.422 vs `FG-gen`(GPT-4 文本生成状态)0.267:**function calling 的状态更新正确率比纯文本生成高 58%**。
- `FG-states`(仅状态函数)0.461 短期单测最高,但论文明确指出这不转化到实际对局——缺骰子函数会导致过度调用状态函数、引入新状态未解决旧挑战,破坏对局流程。`FG-all` 的骰子函数与状态函数互补才整体最佳。

**(3) 统计显著性**(表 7,FG-all 对其余设置的 t-test p 值):

| 维度 | vs FG-dice | vs FG-states | vs FG-default | vs FG-gen | vs DG |
|------|-----------|--------------|---------------|-----------|-------|
| Consistency | 0.0835 | **0.0000** | **0.0001** | **0.0025** | **0.0064** |
| Reliability | 0.2722 | **0.0094** | **0.0338** | 0.1806 | 0.7400 |
| Interest | 0.0965 | 0.1248 | **0.0000** | **0.0005** | **0.0427** |

- 一致性上 `FG-all` 对 `FG-gen` 的优势**统计显著(p=0.0025 < 0.05)**,对 `FG-states`/`FG-default`/`DG` 也显著。
- Reliability 上 `FG-all` 对 `FG-gen` **不显著(p=0.1806)**——论文承认 reliability 指标定义不清、有评委偏置,是未来要重设计的指标。这是一个诚实的负面结果,提示"规则遵循"维度不能只靠人工 Likert。

### 1.3 结论

- **function calling 提供细粒度的游戏特定控制,显著提升叙事一致性与状态更新正确率**(摘要 + §6 + §7)。
- 但函数类型要平衡:仅状态函数会过度调用、破坏流程;仅骰子函数会过于宽松地放行玩家不合理动作;**骰子(动作裁决)函数与状态函数相互制约**才能设出合适节奏(§6.1.2)。
- "骰子死锁"(dice roll deadlock)是 LLM GM 的典型失败模式:无骰子函数时双方无限等骰子结果。需要一个权威的"动作裁决"出口。
- reliability 指标设计是开放问题——Likert 人工评分不足以判定规则遵循。

## 二 PTCG-Bench:状态格式消融 + legal-action masking

> arXiv:2605.29653。场景:宝可梦卡牌(PTCG),2 人零和、不完全信息、长程决策。10 个 LLM backbone(含 GPT-5.4 / Claude Sonnet 4.6 / Gemini 3.1 Pro 等)。评估:Glicko-2 评分 + 锦标赛 + 8 轮自演化。

### 2.1 方法:模块化 harness 消融(RQ3)

固定 LLM backbone、牌组池、锦标赛设置,逐个拆 harness 三模块:

- **H1 Observation structure(观测结构)**:把结构化状态描述换成"最小格式化的原始状态或 game-log 表示"。
- **H2 Legal-action masking(合法动作掩码)**:移除引擎计算的合法动作集,要求 agent 自行推断合法动作与参数。
- **H3 Context management(上下文管理)**:移除近期轨迹历史,只保留当前状态与合法动作。

评估用 Glicko-2 评分均值 μ,Δμ 相对 full harness;并用"相邻 LLM backbone 评分中位差(55 分)"做归一化倍率。

### 2.2 实验数据(表 2,核心结果)

| 配置 | Struct | Mask | Hist | μ | Δμ(归一化) | Inv. Rate(非法/不可解析动作率) | Tool Calls |
|------|--------|------|------|----|------------|--------------------------------|-----------|
| **Full Harness** | ✓ | ✓ | ✓ | **1726** | — | 3.3% | 78.8 |
| w/o Structured Obs(H1) | ✗ | ✓ | ✓ | 1693 | −33(0.6×) | 5.2% | 68.4 |
| **w/o Legal Masking(H2)** | ✓ | ✗ | ✓ | 1608 | **−118(2.1×)** | **15.9%** | 86.7 |
| w/o History(H3) | ✓ | ✓ | ✗ | 1611 | −115(2.1×) | 19.0% | 385.7 |
| Minimal(三者全拆) | ✗ | ✗ | ✗ | 1575 | −151(2.7×) | 27.3% | 357.6 |
| 相邻 LLM 中位差(参考) | — | — | — | 55 | 1.0× | — | — |

### 2.3 结论:三个杠杆的量级排序

1. **legal-action masking 是最大杠杆(−118,2.1× 相邻模型差)**:移除后非法动作率从 3.3% 飙到 15.9%。引擎强制校验合法性的作用,远超状态表示格式本身。
2. **history/context 管理次之(−115,2.1×)**:去掉历史后工具调用数从 78.8 暴涨到 385.7(约 5×)——agent 陷入重试/循环,说明上下文管理几乎与合法校验同等重要。
3. **structured observation 第三(−33,0.6×)**:状态从结构化换成 raw game-log,掉分 33,影响"中等",非法率 3.3%→5.2%。状态格式有效但不是主杠杆。
4. **三者全拆(minimal)−151(2.7×)**:超过多数相邻模型代差,接近同家族模型档位差。**harness 设计对测得能力的影响,是 central factor**(§4.4)。

### 2.4 其他发现(RQ1/RQ2)

- **RQ1**:10 个 backbone 的 Glicko-2 从最低到 Gemini 3.1 Pro(1854)差 617 分,显著超过多数 agent 的 rating deviation,说明 PTCG-Bench 能按实力拉开差距,而非把模型挤进窄带。
- **RQ2 自演化**(Reflexion / ExpeL / Long-Term Memory / Prompt Optimization / Skill Library,8 轮):**五种机制无一表现出单调提升,无一可靠超过非演化的同 backbone ReAct 基线**(§4.3)。即"积累经验→稳定变强"在当前 LLM agent 上还不成立。
- PTCG-Bench 排名与 LiveBench / SWE-Bench Pro / GPQA 仅部分对齐(§4.2,附录 E),说明它测的是静态语言理解之外的能力(不完全信息推理、长程规划)。

### 2.5 引擎强制校验的启示(legal-mask −118 vs 状态格式 −33)

这是本文对"引擎强制校验"最强的数据支撑:

- **不要靠 prompt 让 LLM 自律合法性**——把合法动作集作为 authoritative 字段(`available_actions`)直接暴露,引擎拒绝一切集外动作。PTCG 的动作规范甚至要求"复制 observation 里的精确卡名/攻击名/索引,不得发明"(§A 动作纪律)。
- 状态表示格式(结构化 vs raw)有用但只是 0.6× 杠杆;**把合法性校验从"模型推理"移到"引擎强制"才是 2.1× 杠杆**。
- 对应到通用架构:规则引擎应作为"合法动作闸门"前置,GM 叙事/状态更新在闸门之后;非法动作在引擎层即被拒,不进入 LLM 上下文污染后续推理。

## 三 Game Multiverse:三表示 + Harness Paradox

> arXiv:2605.09965。综述性论文,四支柱(Dataset / Model / Harness / Benchmark)+ 五权衡 + 五级路线图。本文取其 Harness 支柱与权衡分析。

### 3.1 状态/动作表示的三种形态

Harness 支柱(§5)把观测与动作表示分成多种范式,可归并为三形态:

**形态 A:Structured NL / Text-Symbolic(结构化自然语言)**
- 文本化接口把游戏状态转为自然语言或结构化数据,**显式暴露规则、状态变量、历史、动作约束**,支持高度可控的诊断式探测(§6,benchmarks Modality 轴)。
- LLM 原生对齐,诊断清晰。代价:扩展到复杂视觉/商业游戏需要定制 API/wrapper/状态抽取,绕过了"从游戏表面推断状态"的挑战(§6 观测通道)。
- 代表:TextWorld、PTCG-Bench 结构化观测、Labyrinth 渲染状态、clembench、GTBench。

**形态 B:Tabular / Raw game-log(原始/最小格式化表示)**
- 最小格式化的原始状态或 game-log;视觉侧等价于 raw screenshot/pixel stream。保留原生感知负担,更接近人类玩家,但**重新引入状态噪声、grounding 歧义、时间不稳定性**(§6)。
- 生态保真度高、无需定制 API,适合测端到端能力或商业游戏。代价:测得能力下降(PTCG −33)、低分诊断性差(同分可能源于误识别/grounding/延迟/规划弱等任一)。
- 代表:PTCG-Bench raw game-log 消融、VideoGameBench raw GUI、GameVerse。

**形态 C:Symbolic Code(符号代码)**
- Programmatic Control / Code World Models:LLM 合成可执行代码作为动作空间或世界模型。代码含状态转移、合法动作枚举、终止判定函数,可做 MCTS/规划(§5.2)。
- Voyager(Mineflayer JS 技能库)、GameSense(Python "Game Sense Modules")、Code World Models(把规则翻译成 Python 世界模型供 MCTS)。PTCG 的规则引擎本身即等价于预先编译的符号代码表示。
- 优势:可验证奖励信号、可复用技能、可规划;代价:需要游戏有足够"可形式化"的规则,工程量大。
- 适用:规则清晰、可形式化的博弈;需要规划/搜索/可验证奖励时。

**适用场景小结**:
- 结构化 NL → 诊断/评测,可控 probing,LLM 对齐;
- Tabular/raw → 生态保真、商业游戏、端到端测量;
- 符号代码 → 可形式化规则 + 规划/搜索 + 可验证奖励。

### 3.2 Harness Paradox(模块化工作流 vs 模型即整体)

> §7.1.4,五权衡之四。

**悖论:当前模型需要外部脚手架才能运作,但脚手架本身引入脆弱性并限制通用性。**

- **端到端模型无 harness 支持会失败**:在 VideoGameBench / GameVerse 上,前沿 VLM 直接面对游戏截图,多数游戏撑不过开场几分钟(§7.1.4)。
- **模块化 harness 定性强**:Cradle 在《荒野大镖客 2》完成 40 分钟任务;Voyager 在 Minecraft 构建开放式技能库。差距说明:持久记忆、自我纠错、低延迟反射等能力仍超出当前基础模型原生容量。
- **但模块有代价**:每个模块增加工程量;接口制造信息瓶颈;§5.2 记录的"knowing-doing gap"——即便 agent 正确识别最优策略,也常无法转成精确动作参数(坐标/时序)。
- **理想解**:模型原生具备记忆/反思/精细控制,使外部 harness 不再必要。当前 VLA 是朝此方向的一步,但仍缺持久记忆与自我纠错。关闭此 gap 是从"游戏专用 agent 管线"走向"通用游戏玩家"的前提。

**对本任务的含义**:结构化状态 + 工具接口不是"可选项",而是当前阶段让 LLM GM 能跑起来的**承重墙**。但同时要清醒:每多一个模块就多一个信息瓶颈与失效点,接口 schema 要精确(对应 PTCG "复制精确名称/索引"的动作纪律)。

### 3.3 Breadth vs Depth(广度 vs 深度)

> §7.1.2,五权衡之二。

- **异质性之墙**:策略游戏(外交/狼人杀,需深度多步推理,消耗数百 token)与动作游戏(FPS/MOBA,需 100ms 内响应)对同一模型提出相反要求,当前系统无法同时满足(§7.1.2)。
- **VideoGameBench**:最强的前沿 VLM Gemini 2.5 Pro 实时模式只完成 0.48% 任务,不得不引入暂停模式才到 1.6%。
- 现有方案都在两端交易:Combat-VLA 截断 Action-of-Thought 链换 50× 加速但牺牲可解释性与战略深度;Lumine 5Hz 推理/30Hz 动作;NitroGen 并行动作块消除自回归瓶颈但也消除推理。
- **根本问题是架构性的**:自回归 LM 把推理深度耦合到推理延迟——想得越多=token 越多=越慢。当前没有架构能在单次前向里既深度沉思又反射式反应。
- **路线图层面**:Level 3(跨品类)的规律是"游戏覆盖越广,每个游戏的精通度越低",无一系统同时实现广覆盖 + 深精通(§7.2.3)。

**对本任务的含义**:dicelore 是**单品类深度场景**(文字冒险/TTRPG),不是 breadth。应把预算押在深度(状态精度、合法校验、引擎、snapshot、长程一致性)而非跨游戏泛化。

### 3.4 其余三权衡(简述)

- **Trade-off 1 Scale vs Fidelity vs Diversity**:数据三难,无单一语料同时在规模/标注质量/游戏多样性领先(§3.3)。
- **Trade-off 3 Reasoning vs Reactivity**:见上,深度推理与快速反应的架构性冲突。
- **Trade-off 5 Code Engine vs World Model**:代码引擎给确定性奖励与任意长 rollout,但动作空间/多样性受限于人造接口;世界模型(Genie 3/GameGen-X)突破上限但当前只能维持几秒到 60s 一致性,且缺确定性奖励信号(§7.1.5)。**当前阶段规则引擎仍是可靠评估的唯一地基**。

## 四 补充论文

### 4.1 PAYADOR(2504.07304):极简 grounding + 结构化世界表示

> Góngora et al.,ICCC'24。LLM = Google Gemini。

**核心方法**:针对"world-update problem",提出**平衡表示(balanced approach)**——在"通篇流水账文本"与"全结构化数据"之间取中间:

- 只把**对世界一致性至关重要的属性**结构化(Item: Name / Descriptions / Gettable:Boolean;Location: Name / Descriptions / Items / Connecting / Blocked locations;Character: Name / Descriptions / Location / Inventory);其余丰富细节用 `Descriptions: List[String]` 自由文本承载(表 1、表 2)。
- **焦点转换**:不预测玩家"能做什么动作",而预测"动作之后世界应发生什么变化"——predict outcomes, not actions。LLM 输出格式化的变化清单(`- Moved object: <X> now is in <Y>`),再解析回结构化状态更新(图 3)。
- **一致性检查**:因状态是结构化的,系统可对 LLM 输出做校验。经典反例:复现 AIDungeon 的"bazooka"bug——Gemini 幻觉出"bazooka 掉在地上",但 PAYADOR 查库存无 bazooka,**拒绝写入状态**,下一轮从结构化状态重渲染,bazooka 消失(图 5)。
- **选择性渲染**:只渲染玩家当前位置可见/可访问的组件,prompt 长度不随世界规模膨胀,缓解上下文窗口限制。

**对本任务的含义**:PAYADOR 是"结构化 vs 文本"二选一的**第三条路**——结构化只覆盖一致性不变量,叙事细节留给 NL。这与 Labyrinth 的 FG-gen(全文本)最差、PTCG 结构化观测有用但非主杠杆,共同指向"分层混合"结论。

### 4.2 Static vs Agentic GM(2502.19519):solo RPG GM

> Jørgensen et al.,2025-02。单人 TRPG 的 GM AI。

**核心**:v1 = 简化 prompt engineering(静态);v2 = 多 agent 架构 + ReAct 框架(推理+动作,agentic)。对比评测显示 v2 在保持可玩性的同时,**显著提升模块化与游戏体验(沉浸感、好奇心)**。

**对本任务的含义**:agentic(显式 reasoning+action 循环、模块化)优于纯静态 prompt,与 Labyrinth 的 function calling 结论同向——结构化的动作/推理接口比纯 prompt 叙事更可玩、更可维护。但本文是 HCI 体验评估,无硬正确率指标,权重低于 Labyrinth。

### 4.3 Does Reasoning Help LLM Agents Play D&D(2510.18112)

> Delafuente et al.,EMNLP 2025 Wordplay Workshop。FIREBALL 数据集,任务=预测 DnD 玩家动作并格式化为 Avrae Discord bot 命令。模型:DeepSeek-R1-Distill-LLaMA-8B(推理)vs LLaMA-3.1-8B-Instruct(指令)。

**核心结论**:
- **instruct 模型对此任务已足够,推理模型并不更优**;
- 提示的**单句改动就会大幅影响输出**——specific instructions 比 reasoning 更重要。

**对本任务的含义**:对"状态更新/动作格式化"这类**结构化输出任务,精确的 prompt/schema 约束 > 推理模型深度思考**。这与 PTCG(legal-mask 强于一切)、Labyrinth(function calling 强于纯生成)同向:把约束做进接口/schema,而非寄望于模型自己想清楚。

### 4.4 AI-Native Games Survey(2607.00527)

> Xu et al.,2026-07。分析 53 个公开 AI-native 游戏/原型。

**核心**:
- **AI-native 的判据是反事实的**:runtime 生成式 AI 是核心循环的构成性成分——若移除或平凡替换 AI,玩法会崩塌或根本不同。以此区分 AI-native 与 AI-augmented。
- **中心设计问题是:把语义开放性组织成稳定玩法**。AI-native 设计依赖**机械不变量(mechanical invariants):目标、规则、状态、反馈、节奏、玩家能动性**——这些不变量让开放的 AI 输出变得可解释、有后果。
- G/N 双轴分类(G 轴=玩家面向的游戏类型;N 轴=使 GenAI 不可或缺的主导 AI 机制)。语料集中在语言前置设计(叙事冒险/认知互动/生成叙事),而语义裁决、多 agent 模拟、生成式建造、关系/陪伴玩法较少。
- 路线图:可控生成、AI-as-mechanic 设计、多模态/多 agent、推理经济学、评测、安全、监管。

**对本任务的含义**:dicelore 正是"语义裁决 + 叙事冒险"型 AI-native 游戏。survey 的"mechanical invariants"清单(目标/规则/状态/反馈/节奏/能动性)就是引擎层必须固化的东西;生成式叙事(语义开放)必须锚定在这些不变量上才 playable——否则就是 AIDungeon 式幻觉漂移。这给"结构化状态层"提供了独立的、来自游戏设计学的正当性。

## 五 对三个问题的综合结论

### 问题(a):状态该用结构化(DB)还是万物皆文本?

**结论: neither——分层混合。结构化只覆盖一致性不变量,叙事细节留给自由文本。**

论文佐证:

1. **万物皆文本最差**:Labyrinth `FG-gen`(GPT-4 自行生成状态文本)单测正确率 0.267,是三组里最低;人工一致性 3.698 < `FG-all` 4.388,差值统计显著(p=0.0025)。纯文本生成状态会幻觉、漏更新、不可校验(arXiv:2409.06949 表 3、表 7)。
2. **全结构化过度约束玩家能动性**:PAYADOR 指出经典全结构化把每个属性都建表,严重限制玩家自由意志(玩家想做什么都被映射到预编动作);其"balanced approach"只结构化关键一致性字段(位置/库存/blockage/gettable),其余用 descriptions(2504.07304 表 1、表 2)。
3. **结构化观测有用但非主杠杆**:PTCG 结构化 vs raw game-log 仅 −33(0.6× 相邻模型差),远低于 legal-masking 的 −118(2605.29653 表 2)。即"用什么格式存"的影响 < "谁来校验合法性"。
4. **结构化 enabling 校验**:PAYADOR 因状态结构化才能拒绝 bazooka 幻觉(2504.07304 图 5);PTCG 因 `available_actions` authoritative 才能 mask 非法动作。结构化的真正价值不是"格式好看",而是"可被引擎校验"。

**实操判据**:某字段若影响合法性判定、连续性、或会被引擎/规则引用 → 结构化(位置、库存、状态标记、骰子结果、HP、回合阶段);若只服务叙事氛围、可被任意改写而不破坏规则 → 自由文本描述。

### 问题(b):function calling vs 纯 prompt 协议,哪个状态更新更一致?

**结论:function calling(或更广义的"引擎校验的工具接口")显著更一致;但函数类型要平衡。纯 prompt 协议只适合不涉及状态正确性的格式化任务。**

论文佐证:

1. **Labyrinth 6 组对照是直接证据**:`FG-all`(function calling)单测 0.422 vs `FG-gen`(纯文本生成)0.267,一致性人工分 4.388 vs 3.698(p=0.0025)。function calling 把状态更新从"模型自由生成"变成"调用受 schema 约束的函数",正确率提升 58%(2409.06949 表 3、表 7)。
2. **但单一函数类型会失衡**:`FG-states`(仅状态函数)短期单测 0.461 最高,但实际对局因过度调用状态函数、未解决旧挑战就引入新状态而破坏流程;`FG-dice`(仅骰子)又过于宽松放行不合理动作。**骰子(动作裁决)函数与状态函数相互制约**才整体最佳(2409.06949 §6.1.1–6.1.2)。
3. **引擎强制校验是 function calling 的极端形态且杠杆最大**:PTCG legal-action masking −118,移除后非法率 3.3%→15.9%(2605.29653 表 2)。把合法性从"模型推理"移到"引擎强制"是 2.1× 相邻模型差的杠杆。
4. **纯 prompt 协议的适用边界**:DnD Reasoning 发现,对"把玩家动作格式化成 Avrae 命令"这类纯结构化输出任务,instruct 模型 + 精确 prompt 已足够,推理模型不更优,单句 prompt 改动影响巨大(2510.18112)。即:若任务只是"格式化/翻译"而不涉及"状态正确性维护",纯 prompt 可行;一旦要维护跨轮一致性,纯 prompt(FG-gen)即败给 function calling。
5. **Static vs Agentic GM 同向**:agentic(ReAct,显式 reasoning+action)在沉浸感/好奇心/模块化上优于静态 prompt(2502.19519),定性支持"结构化动作接口 > 纯叙事 prompt"。

**实操判据**:涉及跨轮状态维护/合法性裁决 → function calling + 引擎校验;仅涉及本轮输出格式化 → 精确 prompt + instruct 模型即可,不必上推理模型。

### 问题(c):状态表示三形态(structured NL / tabular / symbolic code)适用场景

**结论:三形态各有适用域,按"是否要诊断清晰 / 是否要生态保真 / 是否要可验证规划"选择。**

| 形态 | 代表 | 适用场景 | 代价 |
|------|------|----------|------|
| **Structured NL** | PTCG 结构化观测、Labyrinth 渲染状态、TextWorld | 诊断/评测、可控 probing、LLM 原生对齐、需显式暴露规则与约束 | 扩展到复杂游戏需定制 API,绕过感知挑战(2605.09965 §6;2605.29653 §4.4) |
| **Tabular / Raw game-log** | PTCG raw 消融、VideoGameBench raw GUI、GameVerse | 生态保真、商业游戏无 API、端到端测量 | 重新引入状态噪声/grounding 歧义;测得能力下降(PTCG −33);低分诊断性差(2605.09965 §6;2605.29653 表 2) |
| **Symbolic Code** | Voyager JS 技能库、GameSense GSM、Code World Models Python、PTCG 规则引擎 | 规则可形式化的博弈、需规划/搜索(MCTS)、需可验证奖励、需复用技能 | 工程量大;需游戏规则足够"可编译"(2605.09965 §5.2、§7.1.5) |

**交叉判据**:
- 要做 eval/controlled probing → Structured NL(诊断清晰)。
- 要测真实端到端能力或无 API 的商业游戏 → Tabular/raw(生态保真)。
- 规则清晰且要做规划/搜索/可验证奖励 → Symbolic Code(可编译可验证)。
- 三者非互斥:PTCG 实际上同时用了 Symbolic Code(规则引擎)+ Structured NL(暴露给 LLM 的结构化观测);Labyrinth 用 Structured NL + function calling(轻量 Symbolic 接口)。

## 六 对 dicelore 的启示

1. **状态层走"分层混合",不走万物皆文本,也不走全结构化**。结构化只覆盖一致性不变量(位置/库存/状态标记/骰子/HP/回合阶段/事件契约字段),叙事细节留给 NL descriptions。这是 PAYADOR balanced approach + Labyrinth FG-gen 最差 + PTCG 结构化非主杠杆 三者汇出的结论。对应 dicelore 已有的 sheet/state 二分:state(机械态)要结构化且可被引擎引用,sheet 的叙事字段可保持自由文本。

2. **引擎强制校验是最大杠杆,把合法性从"模型推理"移到"引擎强制"**。PTCG legal-masking −118(2.1× 模型代差)是本文最强数据点。dicelore 的规则引擎应作为 authoritative `available_actions` 闸门前置:非法动作/非法状态转移在引擎层即被拒,不进入 LLM 上下文污染后续推理。GM 叙事与状态更新都在闸门之后。这与 AI-Native Games Survey 的"mechanical invariants"正当性吻合——目标/规则/状态/反馈/节奏/能动性必须由引擎固化,生成式叙事锚定其上。

3. **function calling 给状态更新一致性,但函数类型要平衡**。Labyrinth FG-all 最佳源于骰子(动作裁决)函数与状态函数互补;单一状态函数过度调用、破坏流程。dicelore 的 MCP/工具接口应同时提供"动作裁决"(骰子/判定)与"状态读写"两类函数,并让前者制约后者的调用节奏——避免 GM 不断引入新状态而不解决当前挑战。

4. **Harness Paradox:接受结构化+工具是当前承重墙,别幻想端到端 GM**。VideoGameBench 上前沿 VLM 实时只完成 0.48%;模块化 harness 才能跑出 40 分钟任务。dicelore 当前阶段的引擎+snapshot+工具接口不是技术债,而是必要脚手架。同时警惕 knowing-doing gap:动作参数要精确 schema(PTCG 要求"复制精确名称/索引,不得发明"),接口 schema 是一等公民。

5. **dicelore 是 depth 场景,把预算押深度不押广度**。Multiverse Breadth vs Depth:覆盖越广精通越低,无一系统兼得;VideoGameBench 实时 0.48%。dicelore 是文字冒险单品类深度玩法,应投入状态精度/合法校验/snapshot/长程一致性,而非跨游戏泛化。

6. **上下文管理几乎与合法校验同等重要**。PTCG 去 history → 工具调用 78.8→385.7(5×)、掉分 −115。dicelore 的 snapshot 机制 + 选择性渲染(PAYADOR 只渲染玩家可见组件)是控制 prompt 长度、避免 agent 陷入重试循环的关键手段——snapshot 不只是回滚用,更是上下文治理工具。

7. **状态更新/格式化任务,instruct + 精确 schema > 推理模型**。DnD Reasoning 表明推理模型对此类任务不更优,prompt 单句改动影响巨大。dicelore 选模型时不必盲目上 reasoning 模型;把约束做进 schema/接口(问题 b 结论)比堆推理深度更有效。

8. **reliability/规则遵循指标要硬指标,不能只靠 Likert**。Labyrinth 的 reliability 维度对 FG-gen 不显著(p=0.1806),论文自承指标不清、有评委偏置。dicelore 评测应优先用单元测试式硬正确率(状态更新正确率、非法动作率)而非人工评分;人工评分只作补充。

9. **自演化在当前 LLM agent 上不可靠**。PTCG 五种自演化机制 8 轮无一单调提升、无一超 ReAct 基线。dicelore 不应把"agent 越玩越强"作为短期依赖;稳定 gameplay 优先于经验积累。
