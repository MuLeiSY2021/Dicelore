## 一 设定层(`description`, 5412 字 YAML)

### 1.1 全结构(9 顶层键)

`description` 是结构化 YAML,非 prompt 散文。9 顶层键及段落字符量:

| 顶层键 | 行号 | 约字符 | 内容 |
|--------|------|--------|------|
| `game_info` | 4 | 205 | name/name_en/genre[修仙,玄幻,RPG]/description |
| `world_setting` | 13 | 485 | cultivation_systems(灵修/魔修)/core_laws/天地法则 |
| `resources` | 36 | 167 | 灵气/灵石/天材地宝资源体系 |
| `pill_system` | 44 | 913 | 丹药品阶/效果/炼制(本层最重之一) |
| `stable_resources` | 84 | 140 | 稳定产出资源 |
| `social_system` | 91 | 934 | 势力/宗门/世家/凡人(本层最重) |
| `gameplay_modules` | 133 | 765 | 玩法模块定义 |
| `game_features` | 180 | 1097 | 游戏特性(全卡最大段) |
| `roleplay_settings` | 240 | 507 | 扮演设定(输出语言/人称/字数/文风等) |

提取命令:`jq -r '.description' 万域界修仙手札.json | grep -nE '^[a-zA-Z_]+:'`

### 1.2 嵌套层级

二级字段数(每顶层键下的直接子键):

```
social_system:5  gameplay_modules:5  game_features:5  game_info:4
roleplay_settings:3  world_setting:2  pill_system:2  stable_resources:1  resources:1
```
总字段行(含嵌套)约 172:`grep -cE '^[ #]? *[a-zA-Z_]+:' /tmp/desc.yaml`

### 1.3 关键发现

- `roleplay_settings` 段把**输出规范(语言/人称/字数/对话占比/是否扮演 user)写成数据**,COT Phase 0.2 显式从上下文"提取"这些项——设定层与协议层通过 YAML 键耦合。
- `description` 只占全卡 5412 字,而世界书 275 630 字是它的 **51 倍**。设定层是"骨架索引",血肉全在世界书。
- `personality`(0)/`scenario`(0)/`mes_example`(0) 全空——传统角色卡的字段在此被废弃,改由世界书条目承担。

### 2.3 Step 2.3 三层难度门禁(核心裁决机制)

对每条待执行新行动依次过三道门,**任一层否→标记【不可执行】,禁止进入创作。不可跳过/合并/事后追认**:

- **第一层·物理可行性**:境界能力(移动方式/灵力消耗/前置条件/感知操控范围/伤病影响)→【是】继续│【否】→不可执行
- **第二层·难度合规性**:逐条比对四模块禁令(天道衍算/因果契约/众生心智/越阶天堑),附"常见违规识别"清单(无故瞬移/实力暴增/越阶碾压/必死局脱身/NPC无故放水等)→【全部未违反】继续│【任一违反】→不可执行
- **第三层·代价相称性**:收益是否超本难度投入产出比(低难度容许大收益差,高难度要求付出≥收益)→【合理】可执行│【超出可补】须补代价│【严重超标】不可执行

裁决输出固定格式(2.4):
```
行动：{意图}
├ 物理可行：【是/否】{依据}
├ 难度合规：【是/否】{若否：违反{模块}·{禁令}}
├ 代价相称：【相称/须补代价/失衡】{依据}
└ 最终裁决：【可执行 / 可执行·代价X / 不可执行：原因】
```
**【不可执行】叙事处理**:禁 OOC 拒绝,叙事中呈现为失败/反噬/天道干预/环境阻断,且失败须产生后果(灵力白耗/暴露/伤势加重),不可"什么都没发生"。

### 4.2 JSON Patch(RFC6902 变体)操作语义

变量输出格式明确:"works like the **JSON Patch (RFC 6902)** standard, must be a valid JSON array containing operation objects, but supports the following operations **instead**":

| op | 语义 | 与标准 RFC6902 关系 | 示例 |
|----|------|---------------------|------|
| `replace` | 替换已存在路径的值 | 同标准 `replace` | `{"op":"replace","path":"/交际网络/人物/林婉儿/关系类型","value":"朋友"}` |
| `delta` | 对已存在数字路径增减一个 delta | **非标准**(标准无此 op) | `{"op":"delta","path":"/交际网络/人物/林婉儿/好感度","value":5}` |
| `insert` | 向 object 或 array 插入新项;array 用 `-` 作索引表追加末尾 | 非标准(标准用 `add`;`-` 末尾语法同标准) | `{"op":"insert","path":"/主角信息/持有物品/功法/烈阳真诀","value":{…完整对象}}` / `{"op":"insert","path":"/修仙万象/个人纪事/-","value":{…}}` |
| `remove` | 删除路径 | 同标准 `remove` | `{"op":"remove","path":"/主角信息/持有物品/储物囊/疗伤丹"}` |

- **只读约束**:字段名以 `_` 开头的(如 `_变量`)readonly,禁更新。
- **insert 完整性**:`value` 必须是符合该模块 schema 的**完整对象**,不得遗漏字段(完整性原则)。
- **path 路径**用中文键名 + `/` 分隔,如 `/主角信息/持有物品/法宝/玄龟盾/状态`。

### 4.3 变量输出格式条目全文结构

每回合正文末尾必须输出(格式硬约束):

```
<content>${叙事正文(玄妙磅礴,禁写 user 心理,禁【】禁旁白)}</content>
<UpdateVariable>
<Analysis>$(简体中文,≤80 字)
- ${计算时间流逝}
- ${是否允许戏剧性更新}
- ${逐变量按 check 分析}
</Analysis>
<JSONPatch>
[
  { "op": "replace", "path": "/...", "value": "..." },
  { "op": "delta", "path": "/...", "value": -10 },
  { "op": "insert", "path": "/.../-", "value": {...} },
  { "op": "remove", "path": "/..." }
]
</JSONPatch>
</UpdateVariable>
```

### 4.4 三大更新铁律(变量更新规则末尾)

1. **完整性原则**:严禁留空/未知/暂无,新建变量字段必须填具体内容;`insert` 的 `value` 必须是符合 schema 的完整对象。
2. **同步性原则**:位置与坐标必须同步(`/地图/所处区域名称` ↔ `/地图/当前坐标X/Y` ↔ `/万域/位置`),禁只更其一;位置必须实时反映 user 当前位置。
3. **时效性原则(主动删除)**:消耗品清零即 `remove`;任务完成即 `remove`;状态效果到期即 `remove`;`命脉信息.状态` 有 `剩余时辰` 的每轮递减,归 0 立即删。**遗忘和清除是世界运转的一部分**。

> **注入原理**:AI 输出 `<content>叙事</content>`,前端 regex #1 把整个 `<content>` 块替换为这套 HTML,HTML 内 JS 再从 MVU 引擎取 `stat_data` 渲染三栏。叙事正文进中栏,状态数据进左右栏。**用户看到的是一整个游戏化界面,而非聊天文本**。

## 八 总结

### 8.1 工程化特征

1. **机制即内容**:传统角色卡字段(`personality`/`scenario`/`mes_example`/`first_mes`)几乎全空,团本逻辑全压在 213 条世界书(275 K 字)+ 9 条 regex + 外部 MVU 引擎里。`description` 仅 5412 字 YAML 骨架。
2. **协议优先级工程保证**:`万域COT` 以 `insertion_order=9002`(全卡最高)注入,工程上确保协议层最贴近用户输入、压过所有设定/机制/难度,与"协议不可跳过"铁律一致。
3. **三层门禁 + 字面原则**:COT Step 2.3 的三层裁决(物理可行性→难度合规→代价相称)+ 字面原则(用户没写=没发生)+ 12 全局禁令,构成比 dicelore 当前闸 A/闸 B 更细的 GM 决策链。
4. **难度系统是线性强度轴**:10 档(非 5 档)从绝对豁免到绝对抹杀,四模块同步滑变,默认五星(中正大道=物理守恒铁律)。档位间是 quantitative 强度差,作者按需 `enabled` 切换。
5. **状态层单源不彻底**:9 模块中 7 模块在"变量更新规则"有完整 `check`,但"修仙万象""其他"两模块仅靠"变量输出格式"的 `// Example` 定义 schema——规则与示例双源,易漂移。
6. **强外部依赖**:状态层执行完全依赖外部 MVU JS 引擎(jsdelivr CDN),JSON Patch 是非标准 RFC6902 变体(自创 `delta`/`insert`,标准无 `delta`)。无 MVU 则变量系统不工作。

### 8.2 与 dicelore 的同构/差异(要点,详见 [`../与dicelore架构对比.md`](../与dicelore架构对比.md))

| 万域界 | dicelore | 关系 |
|--------|----------|------|
| COT 三层门禁 + 字面原则 | dicegm 闸 A(该不该骰)/闸 B(谁掷) | 万域更细(三层 vs 两闸),dicelore 可补"字面原则"显式表述 |
| `万域COT` io=9002 注入 | dicegm skill prompt | 同构(协议优先级) |
| 10 档难度世界书条目 | 团本难度预设 | 同构,dicelore 可借鉴"四模块禁令逐条摘录"结构 |
| 9 模块 schema + JSON Patch | sheet/state + `sheet_update` 工具调用 | 同构不同实现:万域靠 regex 提取文本+外部引擎应用,dicelore 靠工具调用直接写 |
| `<UpdateVariable>` regex 提取 | 工具调用 `sheet_update` | dicelore 走工具调用更可靠;万域 regex 方式可作 fallback 参考 |
| 213 世界书(常驻+触发) | build 端机制文档+场景库 | 同构(常驻=机制层,触发=场景层) |

| 叙事裁决派(无显式骰) | 随机驱动派(dicegm 强调掷骰密度) | 路线之争:万域证明纯叙事裁决+门禁可撑长篇 RPG,但缺随机张力 |

### 8.3 对 dicelore 的可借鉴点

1. **三层门禁细化**:dicegm 闸 A/闸 B 之间可插入"代价相称性"层(收益 vs 付出比),尤其 genre-aware 时。
2. **字面原则显式化**:在 dicegm Principles 里显式写"用户没写=没发生;行为即字面意思;仅新行动过门禁",防 AI 脑补。
3. **难度四模块结构**:天道衍算/因果契约/众生心智/越阶天堑四切面,可作 dicelore 团本难度预设的 schema 参考。
4. **状态模块 check 规则**:每字段附 `check`(何时更新/数值边界/单次变化幅度),比纯 schema 更约束 AI 行为,dicelore sheet_update 可借鉴。
5. **八卦九卷铁律**:每轮强制更新≥1 卷第三方传闻,渲染世界鲜活感——dicelore 可借鉴"世界动态"强制更新机制。
6. **`insertion_order` 优先级工程**:用数值精确控制注入顺序,协议层最高——dicelore 若用 prompt 组装可借鉴此量化优先级。

### 8.4 风险/欠账

1. **强外部依赖 MVU**:无引擎则状态层全瘫,跨平台/长期维护风险高。dicelore 走工具调用路线更稳。
2. **状态层双源漂移**:修仙万象/其他两模块靠示例定义,与规则单源原则冲突。
3. **regex 提取脆弱**:`<UpdateVariable>` 靠 AI 自觉输出 + regex 解析,AI 漏输出/格式错则状态丢失(工具调用更可靠)。
4. **难度档位线性堆砌**:10 档全是独立全量条目(各 ~650 字),无参数化压缩,作者改一处需改 10 处。
5. **`use_regex` 全 true 但 keys 多为普通字符串**:正则能力未被利用,徒增解析复杂度。
