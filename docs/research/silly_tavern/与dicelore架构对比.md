# 万域界（silly_tavern）与 dicelore 架构对比 —— 差距与兼容性

> **性质**：研究产物(长存)。把一个工程级 silly_tavern 预制团本与 dicelore 当前架构对齐,找差距与可吸收方向。
> **上游**：[`万域界结构分析.md`](万域界结构分析.md)（万域界五层拆解）/ dicelore wiki [`术语表.md`](../../../wiki/术语表.md) + [`设计/04-子系统设计/`](../../../wiki/设计/04-子系统设计/)
> **本文只研究、不改代码**。术语用 dicelore canonical（Adventure/sheet↔state/event↔log/Moves/Agenda/Principles/闸A/闸B/watcher/Front/Clock）。

---

## 〇 一句话画像


|                     | 万域界（silly_tavern）                                        | dicelore                                                |
| ------------------- | ------------------------------------------------------------- | ------------------------------------------------------- |
| **GM 决策怎么落地** | 超长 COT 协议 prompt 驱动 LLM 自律(7 Step+12 禁令)            | 封闭 MCP 工具集机械兑现 Moves(L1 强制)                  |
| **状态怎么更新**    | LLM 正文末尾输出`<UpdateVariable><JSONPatch>`,前端 regex 提取 | GM 调`sheet_update` 工具直接写 sheet/state              |
| **随机真值在哪**    | LLM 自己产出裁决(可编造)                                      | 引擎内,GM 给不出(抗编造结构性)                          |
| **难度/门禁**       | 团本声明 5 星×4 模块预设 + COT 三层门禁逐条比对              | 团本不声明;运行时 DC 二元过线 + bands 分档,无独立门禁层 |
| **派别**            | **叙事裁决派**(门禁判可否,无显式骰)                           | **掷骰驱动派**(闸A→闸B→形状→谁掷)                    |

**根本差异是架构哲学**:万域界重 prompt、轻基础设施(纯前端+LLM 即可跑);dicelore 重基础设施、轻 prompt(工具强制,抗编造靠结构不靠自律)。两者优缺互补——见下。

---

## 一 架构哲学:提示词工程派 vs 工具强制派


| 维度           | 万域界(prompt 派)                | dicelore(工具派)                                 |
| -------------- | -------------------------------- | ------------------------------------------------ |
| 决策严谨性来源 | COT 协议文本(LLM 须逐步走完)     | L1 封闭工具集( Moves 是 GM 唯一合法动作)         |
| 抗编造         | 靠 prompt 自律(可被绕过)         | 结构性(随机真值引擎内,GM 给不出)                 |
| 状态可靠性     | regex 提取(AI 漏输出/格式错则丢) | 工具调用直接写(可靠,但依赖模型工具能力)          |
| 可审计性       | 弱(全在 LLM 黑箱)                | 强(每步工具调用+event/log 留痕)                  |
| 灵活性         | 高(改 prompt 即改行为)           | 受工具面约束(加行为=加工具)                      |
| 基础设施依赖   | 零(一个角色卡 JSON)              | 重(MCP 引擎+sheet/world/event 四域+presentation) |
| 玩家亲手掷骰   | 无(叙事裁决,无"谁掷"概念)        | 有(明骰玩家客户端点击,亮 DC)                     |

→ **这不是谁对谁错,是两个 trade-off 点**。dicelore 选工具强制是对的(抗编造、可审计、状态可靠是 TRPG 引擎的硬要求);但万域界用 prompt 堆叠换来的**决策链系统化自检**(门禁/形势/校验/终检)是 dicelore 当前 Moves 链没有的。借鉴方向见 §五。

## 三 GM 决策链逐项对照(COT 七步 vs Moves 闸链)

### 3.1 决策链全景

```
万域COT:  Phase0加载 → Init设定/模块 → Step1状态快照 → Step2意图+门禁 → Step3形势判断 → Step4硬约束校验 → Step5叙事构建 → Step6数据结算 → Step7终检
dicegm :                              [闸A能动性]→[闸B该不该骰]→[形状表]→[谁掷明暗] ──────────────→ narrate/resolve_* ─→ 回合末暂存
```

dicelore 的 Moves 链重心在**"这一拍该不该骰/谁掷/骰什么"**(骰判),万域COT 重心在**"这一拍行动可否执行/代价多少/如何系统化自检"**(门禁+自检)。两者覆盖的决策阶段**部分重叠、部分互补**。

### 3.2 逐项对照


| 万域COT 环节                                                                        | dicelore 对应                                                                                                      | 判断                                                                                                                                                         |
| ----------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Phase0 难度加载**(摘录四模块禁令缓存【难度基线】)                                 | 无。难度只以 DC 单常数 + bands 表达                                                                                | **缺失**。dicelore 团本不声明难度预设                                                                                                                        |
| **Init 设定/模块初始化**                                                            | 开局轮建卡(r 六维明骰)+`set_state` 开局.csv                                                                        | ✅ 对齐(实现不同)                                                                                                                                            |
| **Step1 七模块状态快照**(标注临界项,自然语言摘要)                                   | `PresentationSnapshot` 九字段(seq/sheets/mechanics/choices/pendingRoll/narrativeCursor/plotlines/foreshadows/lore) | ✅ dicelore 更结构化。**但**:万域界快照是**喂回 GM 的 COT 提示模块**(每回合强制盘点);dicelore 快照是**数据契约(供前端渲染)**——是否每回合喂回 GM 提示需确认 |
| **Step2.2 字面原则**(玩家没写=没发生/行为即字面/不替玩家决定)                       | 闸A「玩家已说死则不造假分叉」+「别替玩家骰」+ 明骰默认                                                             | △**隐含不显式**。dicelore 有"不造假分叉/不替玩家骰",但**没上升为独立命名字面原则,不在 Agenda 层**;"玩家没写的就没发生"**缺失**                              |
| **Step2.3 三层难度门禁**(物理可行→难度合规→代价相称,创作前强制,不可跳过)          | 闸B(该不该骰:不确定∧失败有意义→骰,否则 narrate)                                                                  | **更粗/缺失**。dicelore 是"骰 or 不骰"二分,**无独立行动可行性门禁**。万域界门禁是"创作前判可否执行",dicelore 无此层                                          |
| **Step2.4 裁决输出格式**(每条行动强制输出裁决清单)                                  | 工具调用即裁决(resolve_* 返回 verdict)                                                                             | ✅ 等价(实现不同:万域界文本格式,dicelore 工具返回)                                                                                                           |
| **Step3 形势判断**(四维评估1-10 + 时空推演 + 走向判定5种)                           | 无                                                                                                                 | **缺失**。dicelore 无"形势/时空系统化推演"环节                                                                                                               |
| **Step4 硬约束校验**(世界法则/角色合理性/主角定位/交叉验证——防门禁裁决被隐性复活) | 无显式环节。靠 L1 工具抗编造 + anti-F2 双边护栏                                                                    | △**机制不同**。dicelore 靠工具结构性防编造;万域界靠 Step4 自检防 LLM"事后追认"。dicelore 的 anti-F2(fail-forward)≈ 万域界"不可执行须有后果"                |

### 3.3 状态更新机制对照(关键)


|          | 万域界                                                        | dicelore                                                         |
| -------- | ------------------------------------------------------------- | ---------------------------------------------------------------- |
| 传输通道 | LLM 输出`<UpdateVariable><JSONPatch>`,前端 regex 提取         | GM 调`sheet_update` 工具                                         |
| 操作语义 | JSON Patch RFC6902(replace/delta/insert/remove)               | 工具参数(带骰算数值 / op= 赋文本)                                |
| schema   | 七模块(万域/主角/交际/修仙万象/宗门/地图/其他),每模块详细字段 | sheet 三元格`{attr,value,visible}` 按 entity 分组 + 横切 visible |
| 可见性   | 靠 prompt 红线"别吐隐藏值"                                    | **结构性**(visible 横切三域 deny-by-default + 暗值强制隐藏)      |
| 失败模式 | AI 漏输出/格式错→状态丢                                      | 模型不调工具→状态不更新(可被 hook 审计)                         |

→ dicelore 的 `sheet_update` 与万域界的 JSON Patch 是**同一思路两种实现**。dicelore 走工具调用是对的(可靠+可见性结构性保障),但万域界 JSON Patch 的**操作语义(replace/delta/insert/remove)比 dicelore 当前 sheet_update 更显式**——这点可借鉴。

## 四 差距:万域界有、dicelore 缺

1. **三层难度门禁**(物理可行→难度合规→代价相称)——dicelore 无独立"行动可行性门禁",只有 DC 二元过线+bands。万域界门禁是"创作前强制裁决可否执行,不可跳过/合并/事后追认"。
2. **难度预设系统**(5 星级×4 模块=20 段,开局缓存【难度基线】)——dicelore 团本不能声明难度,只能隐式写进 rules 散文/params 分档。`flows/clock/entry` 字段设计有、validate 也校验,但 build MCP 产不出、import 不消费,manifest 退化为 name+id。
3. **字面原则**(独立命名:"玩家没写=没发生""行为即字面""不替玩家决定")——dicelore 只有"不造假分叉/不替玩家骰"隐含版,不在 Agenda 层,"玩家没写的就没发生"缺失。
4. **关键词触发的场景条目**——dicelore fronts 是数值/时钟门槛触发,无关键词触发机制。万域界 164 条目按 keys 激活,适合"到了某地/提到某物才展开设定"。
5. **GM 决策协议的系统化自检环节**(Step1 快照→Step3 形势判断/时空推演→Step4 硬约束校验→Step7 终检 checklist)——dicelore Moves 链偏骰判,缺"创作前后系统化自检"。
6. **叙事裁决驱动模式**——万域界证明纯叙事裁决+门禁能撑长篇 RPG;dicelore 当前偏掷骰驱动(且正往更高骰密度矫正),无"本团走门禁裁决非掷骰"的团本声明。

## 五 dicelore 有、万域界缺(dicelore 优势)

1. **L1 封闭工具集机械兑现 Moves**——抗编造结构性,万域界全靠 prompt 自律可被绕过。
2. **明暗骰工具名分流 + 玩家客户端亲手掷**——万域界叙事裁决无"谁掷"概念。
3. **随机真值引擎内 GM 给不出**——万域界 LLM 自产裁决可编造。
4. **visible 横切三域 deny-by-default + 暗值强制隐藏**——万域界靠 prompt 红线,无结构保障。
5. **版本化 Catalog**(git-like commit+tag)——万域界单 JSON 文件无版本化。
6. **watcher/Clock/Front 世界自驱**(edge-triggered,引擎重算回灌)——万域界"动态面板/宗门状态"靠 LLM 每回合自检,无机械 watcher。
7. **声明式 toolgen**(SQL 声明视图/工具→编译)——万域界靠 regex 硬编码。
8. **fail-forward 双边护栏 + 后果手法菜单**(`consequences.md`)——万域界有"不可执行须有后果"但无系统手法库。
9. **构建期/运行时物理隔离**(换装,两套 MCP 互不出现)——万域界单一角色卡全包。

## 六 兼容性:可借鉴方向 + 落地取舍

### 6.1 核心洞察:门禁 ≠ 骰判的对立面,是决策链不同阶段

万域界"门禁"(判可否/代价)与 dicelore"骰判"(判成败/分档)**不是非此即彼,是同一决策链的先后两步**:

```
玩家行动 → [门禁:可否执行?代价多少?] → 通过 → [骰判:成败?分档?] → 叙事
                                  ↘ 不通过 → narrate 失败/反噬(有后果)
```

- 万域界停在门禁(叙事裁决派,门禁即终判)。
- dicelore 当前跳过门禁直接骰判(掷骰驱动派)。
- **统一形态**:dicelore 可在闸B**前**加一道"可行性门禁"(吸收万域界第一层物理可行+第二层难度合规),不通过的 narrate 失败/反噬(呼应 anti-F2 fail-forward),通过的才进闸B骰成败。

这正好补 dicelore"无独立行动可行性门禁"缺口,**且不破坏掷骰驱动**——门禁判可否,骰判判成败,各司其职。

### 6.2 可借鉴方向(按性价比排序)


| # | 借鉴点                             | 落地方式                                                                                                                              | 取舍                                                                                                                          |
| - | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 1 | **难度预设系统**                   | build 端 manifest 富字段补`difficulty`(星级+四模块禁令);顺势修 `flows/clock/entry` 断链。引擎在 `resolve_outcome` 时按难度调 DC/bands | **最高性价比**。manifest 富字段本就设计好但断链,补全即两得。难度预设可结构化存,不必堆 prompt                                  |
| 2 | **可行性门禁层**                   | dicegm 闸B 前加一道门禁检查(物理可行+难度合规),不通过 narrate 失败。落成 L1 工具(如`gate_check`)或闸B 子步骤,非 prompt 自律           | 高价值。但要警惕:门禁判据(物理可行/难度合规)需结构化输入(境界/能力/难度基线),否则又退回 LLM 主观判——需配合 #1 难度预设      |
| 3 | **字面原则上升为 Agenda 一条**     | gm-core Agenda 加北极星条:"玩家没写=没发生,行为即字面,不替玩家决定/加戏"                                                              | **低成本高收益**。纯 skill 文本改动,不涉及工具                                                                                |
| 4 | **关键词触发场景条目**             | build 端 lore 域加`keys` 字段(或第四类内容 keyword-scene),运行时按对话关键词激活注入                                                  | 中。需权衡:dicelore 已有 FTS5 检索 lore,是否还需显式 keys?keys 适合"硬触发"(必注入),FTS5 适合"软检索"(按相关度)。可能两者都要 |
| 5 | **系统化自检环节**(形势/时空/终检) | dicegm references 补"回合末自检 checklist""形势判断"手法                                                                              | 低性价比,慎做。dicelore 走 L1 工具强制,**不该堆 prompt 自检**(违背哲学)。可做薄 checklist 但别学万域COT 的超长协议            |
| 6 | **JSON Patch 操作语义**            | `sheet_update` 支持 replace/delta/insert/remove 显式 op(部分已有)                                                                     | 低。当前 sheet_update 已能覆盖,显式 op 是锦上添花                                                                             |
| 7 | **叙事裁决派团本类型**             | build 端团本声明 `drive_mode: narrative_gate                                                                                          | dice_driven`,dicegm 切对应模式                                                                                                |

### 6.3 关键张力

- **prompt 堆叠 vs 工具强制**:万域界用超长 COT 换决策严谨,dicelore 用工具强制换抗编造。dicelore 借鉴万域界的"门禁/难度/自检"时,**不能照搬 prompt 堆叠**(违背 L1 原则),必须落成结构化字段+工具。例:难度预设→引擎按难度调 DC/bands(结构),而非 prompt 里写"请你严格"(自律)。
- **掷骰密度矫正 vs 叙事裁决派**:当前 corpus-distill 裁决正在把 dicegm 往"更高骰密度"矫正(实跑偏保守)。吸收万域界门禁/叙事裁决要避免开倒车——门禁应是骰判**之前**的预处理(过滤不可执行),不是替代骰判的叙事裁决。

## 七 结论

**差距本质**:dicelore 缺的不是"万域界那一套 COT",而是 COT 背后三类**结构化能力**——① 团本声明难度/门禁的字段(manifest 断链)② 闸B 前的可行性门禁层 ③ 字面原则的显式地位。这三类都可落成结构化字段+工具,不需要照搬万域界的 prompt 堆叠。

**兼容性结论**:两侧**互补大于冲突**。dicelore 的工具强制骨架是优势(抗编造/可审计/状态可靠),万域界的决策链系统化是参考(门禁/难度/自检)。最佳路径是**在 dicelore 工具骨架上吸收万域界的结构化决策层**——优先做 #1 难度预设(顺修 manifest 断链)+ #3 字面原则(低成本)+ #2 可行性门禁(高价值但依赖 #1),其余按需。

**最大启示**:万域界证明"纯叙事裁决+门禁"能撑长篇 RPG,但缺随机张力;dicelore 掷骰驱动有随机张力但缺可行性门禁。**门禁+骰判合一**可能是比"二选一"更优的形态——这正是 §6.1 的核心洞察。
