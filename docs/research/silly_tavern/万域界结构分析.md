# 万域界修仙手札 —— silly_tavern 预制团本结构分析

> **性质**：研究产物(长存)。拆解一个工程级 silly_tavern AI 跑团团本,映射 dicelore 概念,对比 nmbxd 真人串楼。
> **对象**：[`万域界修仙手札.json`](万域界修仙手札.json) (2.0M, chara_card_v3, 修仙品类, 2026-07-08 创建)
> **本文只研究、不改代码**。

---

## 〇 概览

| 项 | 值 |
|----|----|
| 格式 | silly_tavern 角色卡 v3 (`spec: chara_card_v3`) |
| 品类 | 修仙/玄幻 RPG |
| 设定 | `description` 5412 字 YAML(世界观/修炼系统/法则) |
| 世界书 | 213 条目 = 49 常驻 + 164 触发 |
| GM 协议 | `万域COT` 强制执行协议(Phase/Step 决策链) |
| 前端解析 | 9 个 regex_scripts(开局选择器/变量提取/状态占位) |
| 扩展 | `depth_prompt`(depth=4 注入) + `xiaobaix-tasks` + `tavern_helper` |

**一句话**：这不是"角色卡",是一个**作者预制完整、AI 按协议跑的修仙 RPG 团本**——设定 + 机制 + 场景 + GM 决策协议 + 状态变量,五层齐备。

---

## 一 五层架构

### 1.1 设定层(`description`, 5412 字 YAML)

YAML 结构化世界观,非散文:
```
game_info: {name, genre:[修仙,玄幻,RPG], description}
world_setting:
  cultivation_systems: {灵修, 魔修, ...}
  core_laws: [灵气修炼, ...]
  ...
```
**特点**:设定是**数据**(可解析),不是 prompt 散文。对应 dicelore build 端"团本设定结构化"方向。

### 1.2 COT 协议层(`万域COT` 常驻条目, 强制执行协议)

团本核心。分 Phase/Step 的 GM 决策链,**铁律**:
- `世界法则 > 境界体系 > 难度规则 > 角色认知 > 随机事件`(优先级)
- `系统机制通过叙事呈现(Show, Don't Tell),禁止系统提示暴露`
- `难度裁决在创作前完成,不可逆`

决策流程:
- **Phase 0 基础加载**:路由(开局/后续)→输出规范→文风→**难度加载**(四模块:天道衍算/因果契约/众生心智/越阶天堑,逐条摘录禁令缓存为【难度基线】)
- **Init-A 设定提取 / Init-B 模块初始化**:从上下文提取世界观/难度/主角/宗门,仅未提供项合理生成
- **Step 1 状态快照**:七模块盘点(万域/主角信息/交际网络/修仙万象/宗门/地图/动态面板),标注临界项(灵力逼近突破线?寿元逼近上限?业力极端?),仅输出自然语言摘要
- **Step 2 意图解析与难度门禁**:
  - 2.1 user 输入拆解(对话/行动/目标/OOC)
  - 2.2 **字面原则**:`用户没写的就是没发生`、`行为即字面意思,禁止附加权力/情感/策略解读`、`仅新行动过门禁,既成事实不过`
  - 2.3 **难度门禁(三层裁决,创作前强制)**:第一层物理可行性(境界能力/灵力消耗/前置条件)→第二层难度合规性(逐条比对四模块禁令)→第三层(后续)。**任一层否→标记不可执行,禁止进入创作。不可跳过/合并/事后追认。**
- (后续 Step:创作 + 变量更新)

**对 dicelore 的映射**(高度同构):

| 万域COT | dicelore dicegm skill |
|---------|----------------------|
| 铁律优先级(世界法则>...) | Agenda/Principles 优先级 |
| Show, Don't Tell | 叙事呈现(机制不暴露) |
| 难度裁决创作前完成不可逆 | 闸B 该骰处必骰 + anti-讨好 |
| 字面原则(用户没写=没发生) | 玩家能动性(不替玩家决定) |
| 三层难度门禁 | 闸A(该不该骰)→闸B(谁掷)→形状 |
| Step 1 七模块状态快照 | sheet/state 快照 |
| 变量更新放正文末尾 | sheet_update / state 更新 |

→ **万域COT 是 dicegm skill 的"民间工业实现版"**——同一个 GM 决策问题,两种实现路径。COT 的三层门禁 + 字面原则 + 七模块快照,可作 dicegm skill 的对照参考(尤其"门禁三层裁决"比 dicegm 当前闸A/闸B 更细化)。

### 1.3 常驻机制层(49 常驻条目, 始终注入)

24 个 `✅` 前缀机制条目 + COT + 数值/格式规范:

- 修炼体系:`✅境界` `✅灵根` `✅功法` `✅法则` `心魔详解` `境界突破属性判断`
- 资源经济:`✅丹药` `✅符箓` `✅法宝` `✅材料` `✅经济` `✅妖兽`
- 社会结构:`✅宗门` `✅社交` `散修与地下势力` `特殊职业体系`
- 系统规范:`✅纪元` `✅地图及坐标区域规范` `✅✅数值计算与生成` `✅✅✅[mvu_update]变量输出格式` `✅规范提示句`

**特点**:机制层**常驻**(constant=True),始终在 prompt 里——这是"团本的规则手册",AI 每回合都可引用。对应 dicelore build 端"团本机制文档"。

### 1.4 触发场景层(164 触发条目, 关键词激活)

按对话关键词激活,非始终注入:
- **DLC 扩展区域** 70 个:`【DLC】琼华京-中央皇城区-皇宫-承天门/承乾殿/文华殿/...` 细到每个殿
- **宗门** 10+:天霄宗/丹心谷/天机楼/琴心门/御兽宗/大慈恩寺/合欢宗/魔域...
- **秘境**:万兽林/炎火炼狱/...(含推荐配置/核心收益/致命危机/成功案例)
- **纪元**:鸿蒙纪元/天命纪元
- **境界详解**:练气/筑基/金丹/.../圣人(每境界独立条目)
- **妖兽/灵宠/职业详解**

**特点**:场景层**按需展开**(constant=False, keys 触发)——避免 prompt 塞满,只在相关时注入。对应 dicelore build 端"场景/NPC 按需检索"。`秘境·万兽林` 条目示例(推荐配置/核心收益/致命危机/成功案例)是很好的场景条目范式。

### 1.5 前端解析层(9 regex_scripts)

用正则在 AI 输出上做后处理,实现"前端解析机制":
- `[选择]开局选择器`:提取 `[开局]...[/开局]` 内容做选项
- `[去除/不隐藏]UpdateVariable` / `[去除+隐藏]UpdateVariable`:提取/隐藏 `<UpdateVariable>` 变量更新块
- `<StatusPlaceHolderImpl/>`:状态占位符替换
- `[前端]万域`:提取 `<content>...</content>`

**特点**:**变量/状态不靠 AI 自觉,靠 regex 从输出里提取**——AI 在正文末尾输出 `<UpdateVariable>...`,前端正则解析更新状态。这是"AI 输出 → 结构化状态"的一种实现(另一种是 dicelore 的工具调用 sheet_update)。

## 二 与 nmbxd 真人团本对比

| 维度 | silly_tavern 万域界(本团) | nmbxd 真人串楼 |
|------|---------------------------|----------------|
| 形态 | 作者预制完整团本,AI 按协议跑 | 真人 GM 即兴带团,串楼记录 |
| GM | COT 协议(AI 执行) | 真人 GM |
| 机制 | 预声明(常驻条目+门禁) | 真人现场裁决 |
| 状态 | regex 提取变量(结构化) | 串楼文本(无结构) |
| 场景 | 世界书条目按需激活 | 真人即兴叙述 |
| 随机 | COT 内置(未明骰,偏叙事裁决) | 明骰 r / 投票 fy / 选项菜单 |
| 品类 | 修仙 | 西幻/抽卡/苗床/现代/机甲/赛博... |

**关键差异**:万域界是**叙事裁决派**(COT 门禁判断可否执行,无显式骰),nmbxd 是**随机驱动派**(明骰 r / 投票 fy 推进)。这正对应 dicelore 的两条路线之争——`resolve_choice`/掷骰 vs 叙事裁决。万域界证明"纯叙事裁决+门禁"也能撑起长篇 RPG,但它**没有 nmbxd 的随机张力**(dnd 每行动一掷、安价投票定方向)。

## 三 对 dicelore 的启示

### 3.1 build 端团本结构(三层架构可借鉴)

万域界的世界书三层 = build 端团本的可参考 schema:
- **机制层**(常驻):规则手册,始终可引用 → dicelore build 的"团本机制文档"
- **场景层**(触发):地点/NPC/事件,按需展开 → dicelore build 的"场景库"
- **协议层**(COT):GM 决策链 → dicelore 的 dicegm skill(build 端可选预设 GM 风格?)

### 3.2 COT 门禁 vs dicegm 闸

万域COT 的**三层难度门禁**(物理可行性→难度合规→...)比 dicegm 当前闸A/闸B 更细化。可对照:
- dicegm 闸A(该不该骰)≈ COT 2.3 门禁(该不该执行)
- 但 COT 多了"字面原则"(用户没写=没发生)的显式表述 + 三层逐项比对——dicegm 可补这层
- COT 的"难度基线缓存"(开局摘录四模块禁令)≈ dicelore 团本难度预设

### 3.3 状态机制两种实现

- 万域界:AI 输出 `<UpdateVariable>`,前端 regex 提取
- dicelore:工具调用 `sheet_update` 直接写状态
- **取舍**:regex 方式不依赖工具调用能力(纯文本即可),但易出错(AI 漏输出/格式错);工具调用可靠但依赖模型支持。dicelore 走工具调用是对的,但万域界的 regex 思路可作 fallback 参考。

### 3.4 随机 vs 叙事裁决

万域界(叙事裁决派)成功撑起长篇修仙 RPG,但缺 nmbxd 的随机张力。dicelore 当前偏随机驱动(dicegm 强调掷骰密度),**可考虑支持"叙事裁决派"团本**——即 build 端允许团本声明"本团走门禁裁决,非掷骰驱动",dicegm 切对应模式。这呼应之前 §三 讨论的"genre-aware / 机制密度"问题。
