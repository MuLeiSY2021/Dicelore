# 探索：如何让 AI 当一个好 GM

## 核心问题

AI 的天然倾向是**讨好玩家**——给好结果、软着陆失败、跳过骰子、回避冲突。
这恰恰是坏 GM 的特征。

anko_driver 的本质不是游戏引擎，是 **GM 行为塑形系统**。
所有机制的核心目标：让 AI 克服讨好本能，按规则书当一个公正、有戏剧性的 GM。

## 我们已经有什么

Phase 0+1 已实现：

| 模块 | 作用 | 对"GM 行为塑形"的价值 |
|------|------|----------------------|
| `anko_core.dice` | 骰子纯计算 | 结构强制——AI 无法伪造骰子结果 |
| `anko_dice` MCP | 5 个骰子工具暴露给 AI | 调试/自由骰点，但不强制 AI 使用 |
| `anko_db.store` | SQLite 持久化 | 状态不可篡改——写入就是写入了 |
| `anko_db.session` | Session 生命周期 | 隔离游戏实例，状态/历史/笔记 CRUD |
| `anko_db.knowledge` | FTS5 全文搜索 | 规则书内容可检索——AI 能查到规则 |
| `anko_db.rulebook` | 规则书加载 | Markdown→知识库，YAML→结构化数据 |
| `anko_game` MCP | session 管理 + scene_narrate | 目前只有一个场景工具，骨架很空 |

**关键缺口**：AI *可以* 不调用任何工具，直接用纯文本叙事，跳过所有骰子和状态管理。
当前架构只提供了工具，没有提供**让 AI 必须使用工具的机制**。

## AI 不听话的三种失败模式

### F1: 跳过骰子
玩家："我攻击哥布林"
AI 直接写："你一剑砍倒了哥布林"——没有骰点，没有失败可能。

**可强制**：如果 Skill 强制 AI 调用 `action_resolve`，骰子一定会被掷。

### F2: 不尊重骰子结果
骰出大失败，AI 写："虽然失误了，但你还是勉强命中了"——软着陆。

**难强制**：这是叙事层面的，代码无法直接控制 AI 的措辞。
但可以通过 Hook 检测不一致（骰出 failure 但叙事写了成功），或通过 Skill 的明确指令约束。

### F3: 选错处理方式
玩家："我接近那个神秘旅人"
AI 应该给选项（搭话/绕路/偷袭），但直接骰了一个 1d4 决定结果——剥夺了玩家选择权。

**无法强制**：这需要 GM 判断力，代码无法判断"这个场景该给选项还是该骰子"。
只能通过 Skill 流程和 Guidelines 训练。

## 三层塑形模型

```
┌─────────────────────────────────────────────┐
│  L3: 事后验证 (Verification)                 │
│  Hook 检查结果一致性，审计骰子 vs 叙事        │
├─────────────────────────────────────────────┤
│  L2: 流程引导 (Guidance)                     │
│  Skill 编码 GM 决策流程，Guidelines 描述行为  │
├─────────────────────────────────────────────┤
│  L1: 结构强制 (Enforcement)                  │
│  MCP 工具内置约束——骰子必须掷，状态必须记     │
└─────────────────────────────────────────────┘
```

每层对抗不同的失败模式：

| 失败模式 | L1 结构强制 | L2 流程引导 | L3 事后验证 |
|---------|-----------|-----------|-----------|
| F1 跳过骰子 | 工具必须掷骰 | Skill 强制调用流程 | Hook 检查是否有骰子记录 |
| F2 不尊重结果 | — | Guidelines 明确禁止软着陆 | Hook 比对 outcome vs 叙事 |
| F3 选错方式 | — | Skill 决策框架 + 范例 | — |

**核心洞察**：L1 能防 F1，L3 能抓 F2，但 **F3 只能靠 L2**。
而 L2 的主要载体是 **Skill**，不是 MCP 工具。

## 重新理解各组件的角色

### MCP 工具 = 结构强制的手段

MCP 工具的价值不在于"功能"，在于**不可绕过性**。

- `action_resolve` 的价值不是"帮你掷骰子"，而是"你调了这个工具，骰子就一定会被掷，结果就一定会被记录"
- `state_set` 的价值不是"帮你存数据"，而是"你不调这个工具，状态就不会变"

所以工具设计的原则：**凡是不该由 AI 自由决定的事，都锁在工具里**。

### Skill = GM 训练的主要载体

Skill 是 L2 的核心。它编码的是 **GM 操作流程**，不是"便利功能"。

一个好的 Skill 应该像 RPG 规则书的 GM 章节一样：
1. 告诉 AI 在什么情况下该做什么
2. 给出明确的步骤流程
3. 提供判例和边界

### Guidelines = 规则书的 GM 约束

`ai/guidelines.md` 不是"建议"，是**行为规则**。应该像真正的 RPG 规则书一样具体：
- "当玩家执行有风险的行动时，你必须调用 action_resolve"
- "当骰子结果为大失败时，你必须引入新的困境，而不是让玩家侥幸逃脱"
- "当玩家在探索新场景时，你应该先调用 explore_options 展示选项"

### Hook = 合规审计

Hook 是 L3。它不阻止 AI 犯错，但能**发现**错误：
- 检查 action_resolve 返回 failure 时，后续叙事是否尊重了失败
- 检查 state_changes 是否被正确应用

## 开放问题

### Q1: Skill 优先还是工具优先？

**原设计**：Phase 2 先做工具，Phase 3 再做 Skill。
**新思路**：Skill 是 GM 行为塑形的主载体，是否应该先做 Skill？

如果先做 Skill，AI 已经有了 GM 操作流程的指导，即使工具接口还不够完善，
AI 也能遵循基本的 GM 行为。工具可以逐步完善，Skill 是骨架。

### Q2: 工具粒度——5 个场景驱动工具还是更少？

原设计有 5 个工具，每个对应一种范式。但如果范式选择归 AI，
工具是否可以更简化？

**方案 A**：保留 5 个工具，每个有明确的语义（action_resolve = 对抗，explore_options = 选择...）
**方案 B**：合并为 2-3 个工具（`resolve` 统一处理骰点/选项/分支，`generate` 处理随机生成，`narrate` 处理纯叙事）
**方案 C**：1 个核心工具 `act`，范式作为参数——AI 决定范式，工具执行

### Q3: Skill 如何编码 GM 决策流程？

Skill 是 Markdown 文件，通过 prompt 注入给 AI。但它如何编码"决策流程"？

**方式 1**：步骤式——"第 1 步判断场景类型，第 2 步选择工具，第 3 步调用工具，第 4 步按结果叙事"
**方式 2**：判例式——"案例：玩家攻击 → 调 action_resolve → 骰出失败 → 叙事引入困境"
**方式 3**：检查清单式——"每次玩家行动后检查：骰子是否掷了？结果是否尊重了？状态是否更新了？"

### Q4: 规则书的范式相关内容如何组织？

既然范式选择归 AI，规则书关于范式的部分应该：

**方案 A**：完全放在 `ai/guidelines.md` 中，作为 GM 操作指南的一部分
**方案 B**：保留结构化数据（骰子配置、暴击阈值），但场景绑定变为建议文本
**方案 C**：规则书定义"场景模板"——不是强制映射，而是"典型处理方式的参考"

### Q5: 如何防止 AI 绕过 Skill？

如果 Skill 只是 Markdown prompt，AI 完全可以忽略它。
如何确保 AI 遵循 Skill 的流程？

**思路 1**：Hooks 检查——如果 history 里没有对应的工具调用记录，报警
**思路 2**：Skill 作为强制入口——用户输入 `/anko-action`，Skill 接管整个流程
**思路 3**：接受不完美——Skill 是最佳实践，AI 偶尔偏离可以接受，Hook 负责纠偏

### Q6: 当前该保留什么，该重建什么？

**保留（已实现且方向正确）：**
- anko_core.dice — 骰子引擎
- anko_db — 持久化层
- anko_dice MCP — 骰子工具
- 规则书加载框架 — Markdown + CSV + YAML

**需要重新设计：**
- anko_game MCP — 工具集和接口
- 范式系统 — 从代码强制改为 AI 决策框架
- Skill 体系 — 从便利功能升级为核心机制
- Guidelines 规范 — 从空泛建议改为具体 GM 规则

**可能需要新建：**
- Skill 编写规范 — 如何写一个有效的 GM 训练 Skill
- Hook 审计体系 — 如何检测 AI 不听话
