## 一、起因：构建侧类型无单源（已实测的漂移）

Review `backend/src/build/`(Draft/buildMcp/pack)+ `harness/eval-loregm/build-mcp.ts`(MCP)+ `frontend/src/features/build/`(前端)三层后，发现团本包(pack)/Draft 领域类型分散定义、无单一权威源：

| 类型 | 定义点数 | 现状 |
|------|---------|------|
| `StateCell` | 3 处 | `@dicelore/interface/domain.ts`(canonical，但**没人用**)+ 后端 `draft.ts` + 前端 `api.ts`(已漂：`value` 类型不同) |
| `FrontSpec` | 2 处，**不同构** | 后端 `{name,stakes,clock_attr,clock_min/max,clock_mode,omens[{threshold,payload}]}`(正典 md 格式) vs 前端 `{title,clockName,clockMax,steps[{at,text}]}`(**旧格式，已废弃**) |
| `DraftSnapshot` | 1 处(前端手抄) | 手抄后端 `Draft.snapshot()` 的匿名返回类型，后端连命名导出都没有 |
| `PackFile` | 2 处 | 后端 `catalog.ts` + 前端 `catalog/api.ts` |
| `ValidateIssue` | 2 处，**不同构** | 后端 `{file,msg,hint}` vs 前端 `{path,msg}`(丢了 hint) |
| MCP author 侧 | 0 类型 | `build-mcp.ts` 的 `get_draft` 返回 `unknown` |

**最能说明问题的实证**：前端 build 页渲染 front 卡片读 `f.title/f.clockMax/f.steps`(`BuildPage.tsx:473-474`)，但后端真实数据键是 `name/clock_max/omens`——**前端 front 卡片现在渲染出来全是空的**。

还发现一条更深的病根：`validatePack` 的 manifest 富校验(Rule 1-4)只认 `manifest.yaml`，但 `toPackFiles()` 产出的是 `manifest.md`——Draft/构建 GM 路径造出来的包，manifest 校验规则**从不触发**。

`import.ts` 侧交叉验证：域级别的构建→导出→回放链条(lore/rules/pools/state/sheets/fronts/plotlines/foreshadows/anchors/manifest.md/prologue/tools)其实是齐的，**漂移集中在"类型定义"层，不在"运行时消费"层**。

## 二、ST 架构调研的论文结论（既有产物，本轮直接复用）

已有 7 篇调研文档（`docs/research/silly_tavern/架构调研/00~06`），核心结论：

1. **分层混合，非二选一**：结构化只覆盖"一致性不变量"(位置/库存/状态标记/骰值/HP/回合阶段)，叙事细节留自由文本(PAYADOR balanced approach + Labyrinth FG-gen 0.267 最差 + PTCG 结构化观测仅 −33)。
2. **引擎强制门禁是最大杠杆**：PTCG legal-action masking −118(2.1× 相邻模型代差)，远超状态格式本身(−33)。合法性要从"模型自觉"移到"引擎强制"。
3. **function calling 优于纯文本生成状态**：Labyrinth FG-all(工具调用)单测正确率 0.422 vs FG-gen(纯文本生成)0.267，p=0.0025 显著。
4. **万域界修仙手札(ST 角色卡实证)是"方向对、接口错、层级错"的反例**：状态层靠 regex 提取 `<UpdateVariable>` + 外部 MVU JS 引擎应用 JSON Patch(FG-gen 变体，被论文否定)；门禁靠 L2 prompt 教条(万域三层门禁)而非引擎强制；无 MVU 引擎则状态系统全瘫(强外部依赖)。dicelore 现有的工具调用路线(`sheet_update`)已经是更优路线，不应退化。
5. **dicelore 承重墙方向对**：L1 工具强制(随机真值在引擎)/ visible 三态横切 / 版本化 Catalog / watcher 自驱 / 声明式 toolgen / 明暗骰玩家亲手掷——六处既有优势，迁移/改进不可丢。
6. **06 已给出 P0/P1/P2 分批建议**（机制下沉→引擎门禁→团本级 skill 出口），本轮设计到"架构骨架"级别为止，字段级细节按此分批走 `/idea-to-roadmap`。

## 四、已定的架构决策

### 4.1 单元模型：DLC = 无头团本(headless pack)

- **头** = `prologue`(开场白) + 开局/主角选择 + `entry` 锚点。
- **DLC** = 团本(pack)去掉头的全集——同一套 canonical schema，只差"有没有头"这一个判据。**DLC 可大可小**(小到一个 NPC/一条机制，大到一整个世界)，可复用、可被团本引用组合。
- 团本 = DLC 内容全集 + 头。ST 的"角色卡+可挂世界书"在此同构为"有头 pack + 可组合的无头 pack"，但运行态统一为 pack。
- **ST 世界书 = DLC**：ST 的世界书概念对应到 dicelore 就是"无头团本"——包含所有域(sheets/rule/pool/front/plotline/skill/mcp 声明式工具等)但没有开场，不能独立开局。

### 4.2 预设三拆（不做一等单元）

ST 的"预设"实际是三样东西混在一起，拆开后各有归宿，不新增顶级单元：

| ST 预设含义 | dicelore 落位 | 理由 |
|---|---|---|
| 采样参数(temperature/top_p/max_tokens…) | **Claude Agent SDK / 模型层**，不做团本可编辑单元 | DnD Reasoning(2510.18112)论文：结构化输出任务里 instruct+精确 schema > 调参/推理深度，采样调参是低杠杆 |
| prompt 结构/系统提示/上下文模板 | **gm-core skill(框架单源) + flow skill(团本编排)** | PTCG −118 指向 L1 引擎强制，prompt 堆叠优先级不是正解 |
| 输出规范/文风(ST roleplay_settings：语言/人称/字数/文风) | **narrative MCP**(`backend/src/stdlib/narration.ts`，与 front/plotline/foreshadow/tension_board 同一工具面) | 工具调用可查可改，比塞进静态 manifest 字段更贴合论文"工具接口优于静态注入"；且顺手补上这个已知的"stdlib 声明未接生产"断节(narration.ts 现有 front_open/plotline_*/foreshadow_*/tension_board 已声明未接线) |

### 4.3 Core skill 分槽 + overlay + 团本专属 skill

**背景**：需要允许团本"覆写" core skill 的某些教条，同时允许团本添加专属 skill——但团本级完整协议压过框架(万域 COT 反例)会破坏 L1 承重墙，需要有界覆写。

- **锁定槽(承重墙，不可覆写)**：L1 工具强制(随机真值在引擎)、明暗骰主权、visible 三态、F1 必掷骰/F2 双边护栏底线、门禁 legality mask 的引擎强制性。
- **开放槽(可覆写/扩展)**：品类禁令、叙事偏好、Moves/形状表扩展、难度基调等 L2 教条。
- **overlay**（术语已定名）= 团本对 core 开放槽的覆写物。文件约定：`overlay/<name>.md` + `sequence.json`(声明多 overlay 叠放顺序，开一个团时可能叠多个)。
- **不建议写 overlay**——优先建**团本专属 skill**(纯增量，编排已有工具，不碰 core，05 已有此结论)。overlay 是真需要覆写开放槽时才退到的逃生阀，不是默认路径。

### 4.4 ST 能力缺口 → 现有骨架内的最佳实践（零新增 schema，用户纠正版）

> 这两点是用户对我最初"新增 persona 域"+"prologue 变体列表"两个提案的纠正——都不需要新结构，现有骨架已经能表达。

1. **Persona 富字段(性格/外观/说话风格/初始好感等)** → **全部落 npc(或 player)sheet 的 state cells**，走既有 `set_state`/`sheet_update`/`npc.ts` stdlib(`npc_register`/`npc_set_identity`/`npc_update_affinity` 等)。**不新建 persona 域**、不引入 `personas/<entity>.md` 两级物化格式。唯一要补的是**最佳实践约定**：build skill 教条里定一套 canonical attr 命名(性格/外观/说话风格/初始好感…)，避免不同团本各写各的字段名导致语义漂移。
2. **多开局变体(对应 ST `alternate_greetings`)** → **prologue 保持单值不变**；变体内容落 **pool**(`add_pool`/`pools/*.csv`，如 `pools/开局变体.csv`)；prologue 正文指示 GM 在开局时调用已有的 `world_sample`(`poolSample`)从该池取一条决定具体开场。**已验证** `world_sample` 运行时工具已存在(`backend/src/stdlib/world.ts` + `backend/src/store/world/world.ts poolSample`)，**零新增**。

## 五、大规模叙事知识存储/检索——数据层分类总纲

> 这部分是本轮最大的方向修正：最初按"检索机制"(grep/RAG/skill)分析被用户否定，重新按"数据天生的形状"分类，检索机制是形状的结果，不是独立选择。

### 5.1 先被否定的框架（记录以防重复）

最初我按"检索机制"分析(build-mcp的agentic读取/lore的FTS5词法检索/skill的progressive disclosure/vector RAG)，援引：
- Anthropic 把 Claude Code 的 RAG pipeline 换成 agentic search(Boris Cherny："outperformed everything, by a lot")，独立复现研究(42测试)显示向量 RAG 的结构性死穴是**跨文档综合/交叉引用**(top-k 相似度检索会整体聚拢到语义最强的单一文档，另一份内容零命中)。
- Amazon Science(AAAI 2026)：agentic 关键词检索达到向量 RAG 忠实度的 94.5%，零向量库。
- Claude Skills 官方 progressive disclosure(SKILL.md + reference/*.md 分域，agent grep 域内文件)。

**结论曾是**："dicelore 现有的 FTS5+jieba 词法检索+主动召回路线已经是对的，不要引入向量 RAG"。

**用户反驳**：如果 Claude Code 的发现是真的，那这就不是"该用哪个检索工具"的问题——检索机制的选择应该是数据形状的**结果**，不该脱离数据形状单独讨论。这个反驳成立，推翻了"先选机制"的分析顺序。

### 5.2 最终版：按数据形状分类(13 类 + 2 条正交轴)

| # | 类别 | 现有/建议落点 | 形状特征 | 天然访问方式(形状决定) |
|---|------|---------|---------|---------------------------|
| 1 | 常变数据 | `state`/sheet 表(entity+attr+value+visible, kind=player/npc/world/flag/clock) | 小基数(每会话几十~几百行)，逐轮可能变 | 永远全量在场，不检索——直接读 |
| 2 | 关系图 | `anchor` 表(owner→target:role) | 小基数，图形状(节点+边)非表格 | 图遍历查询(从某节点找边)，非搜索 |
| 3 | 教条数据 | **文件**(pack 内 `rules/*.md` 等，canonical 路径)+ gm-core skill(框架级 Agenda/Moves/Principles)+ overlay/团本专属skill(同形状，作用域降到团本级)——**已决定，见 §5.4** | 小~中基数，静态(build 写/play 只读) | play 期 GM 用 `Grep`/`Glob`/`Read` 原生探索 checked-out 文件目录(非 MCP 工具包裹) |

### 5.3 待理顺的缝

**关系的"结构"和"强度"现在分裂在两张表**：`anchor` 表只有 `role`(标签)，无强度数值列；"好感度"这类关系强度实际存在 `state` 表(`npc_update_affinity` 写的是 state 的"好感"attr，靠 entity 名字符串关联，不经过 anchor 的 owner/target_id)。图的"边"和"边的属性"现在没有统一概念绑住，是两套机制凑巧描述同一件事。**需要决定**：把 anchor 收成真正带权图(边可带数值属性)，还是保持现状(结构靠 anchor、强度靠 state，靠约定对齐)。

### 5.4 三模块职责分工 + 逐类落点（用户提出的第三条轴）

> 用户先明确三模块目标，再据此给每类数据切 backend/harness/前端 三列落点：
> - **harness**：为 agent 提供友好、满足业务需求的数据和信息
> - **前端**：为用户提供友好可见视图，并正确反映 agent 侧行为
> - **backend**：为 harness 提供数据持久化，为前端提供 API 支持

**通用原则（用户最终定稿，贯穿全文档，比下表任何一行都更重要）**：这张表的目的不只是看"物理形态"（表还是文件），更重要的是——**`shared`/canonical 包的代码也该按这套类别分类，不是按物理存储分类**。API 形状要跟着数据的天然形状走，backend 负责把物理实现(SQL/文件/etc.)包装成这个形状，不能反过来让调用方直接摸物理结构。举例：#2(关系图)即便底层是 SQL 表(`anchor`)，`shared` 对外暴露的必须是**图 API**(节点/边/遍历方法)，不能叫"表"、不能长"CRUD一张表"的样子。同理：#1(常变数据)该暴露"账本(ledger)API"；#3/#4(教条+叙事语料)该暴露"文档(doc)API"(path/category/id+content，Grep/Glob 友好的形状)；#9(能力声明)该暴露"capability API"；#12(视觉资产)该暴露"asset API"(uri+渲染钩子)。**这也是 §六 待决问题#1(canonical schema 单源化技术方案 A/B/C)被有意延后的原因**：技术选型要服务于"每个类别该长什么API形状"这个更上游的决定，必须等这张表(逐类落点)完全敲定后才能定，不能反过来先选技术再套形状。

| # | 类别 | backend 存储 | harness 落点 | 前端落点 |
|---|------|---------|---------|---|
| 1 | 常变数据 | sqldb(`state`表) | mcp | dock-card ✅确认无异议 |
| 2 | 关系图 | sqldb(`anchor`表，非图数据库——会话级别边数量几十~低百条，SQLite关系表+递归CTE足够，图DB过度设计)。**`shared` 对外暴露"图"形状 API(节点/边/遍历)，不是"表"形状 API——即便backend实现是SQL表**(通用原则的示例本体)。§5.3的"强度要不要收进anchor"是SQL层实现细节，可推给backend实现阶段，不阻塞图API形状这个已定原则 | mcp | 关系卡片列表(复用dock-card模式，不需要力导向图可视化) |
| 3 | 教条数据 | **文件**(pack内`rules/*.md`+ gm-core/overlay/团本skill文件，canonical路径) | **Grep/Glob/Read 原生探索**(非裸MCP文本工具)，见下方"3/4的落地机制" | rule检索(后端对checked-out文件做grep/索引，API化给前端搜) |

**#3/#4 的落地机制（已决定，非"待定"）**：

1. **play 期开会话时，把该团本 checkout 出来的 lore/rule 文件物化到一个只读的会话工作目录**——复用 `catalog.ts` 已有的 `checkout()`(已返回 `PackFile[]`)，新增一步把这些文件写到磁盘。这不是新建检索机制，只是"写文件"这一步，且与 build 侧`workspace/materials/`是同一套思路的 play 侧镜像。
2. **`harness/src/dicegm/gmAssembly.ts` 的 `allowedTools` 从两档扩为三档**：
   - build/loregm(现状)：`["mcp__dicelore","Read","Bash","Grep","Glob","Write","Edit"]`
   - **play 期新增一档**：`["mcp__dicelore","Read","Grep","Glob"]`(能探索/搜索，**不给 Bash/Write/Edit**——不能改团本canon、不能任意执行)
   - play 期无 workspace(兜底)：`["mcp__dicelore","Read"]`
3. **审计边界精确圈定在"状态变更"而非"是否走MCP工具"**：`sheet_update`/骰判/front·plotline·foreshadow变更等一切改状态的动作依然100%走MCP工具，不受影响；只有"查lore/rule知识"这个纯只读动作从"MCP工具返回文本"换成"Grep/Glob原生探索"。Grep/Glob调用完整进 session transcript(build-eval读`<id>_session.jsonl`的机制已在用)，但不必也不该进 dicelore 自己的游戏内 event log(narrate/verdict/mutation记的是"故事里发生了什么"，GM翻自己的备注不算游戏事件)。
4. **为什么不是"MCP工具包一层ripgrep"（曾经的中间方案，已被推翻）**：那个方案要手工维护搜索结果的返回契约(uri/category/id + content)才能不丢结构化身份信息；而 Grep/Glob 是 Claude Agent SDK 原生能力，天然吐 `path:line:content`，uri 问题被工具自身输出格式免费解决，不需要额外维护返回形状。**这也把"叙事语料的canonical命名分区"从"字段级细节"提升到"架构骨架前提"**：没有canonical路径，Grep结果就没有可携带的结构化身份。

## 六、待决问题（未最终定稿，需用户拍板）

1. **Canonical schema 单源化技术方案——有意延后，非悬而未决**：用户明确表态：不是没想清楚，是**要等 §5.4"逐类落点"完全敲定后才考虑**——技术选型(A/B/C)必须服务于"每个类别该长什么API形状"这个更上游的决定，不能反过来先选技术再套形状。三个候选保留供到时候用：
   - 方案 A(推荐)：新包 `@dicelore/pack`，zod-first——pack/DLC 域全部类型定义为 zod schema，`z.infer` 派生 TS 类型；`buildMcp.ts` 现有 `BUILD_SCHEMAS`(已是zod)改为从此包 import；解决 build-mcp stdio 边界 + `/sessions/loregm/:id/draft` HTTP 边界"运行时JSON边界零类型保护"的真空。
   - 方案 B：纯 TS interface 挪进 `packages/shared`，不引入校验技术——只治类型漂移，治不了运行时边界校验缺口。
   - 方案 C：JSON Schema 单源生成 TS+zod——过度设计(项目全 TypeScript，无跨语言诉求)。
2. **anchor 图边 vs state 强度分裂 + 更根本的原则(已定)**：§5.4 已敲定通用原则——**`shared` 包按 §5.4 的数据形状分类切 API 模块，不按物理存储分类**；即便 anchor 底层是 SQL 表，`shared` 对外必须暴露"图"形状的 API(节点/边/遍历)，不能长"表"的样子；backend 负责把 SQL 实现包装成这个图 API 形状。§5.3 原本悬而未决的"强度要不要收进 anchor 加权重列"，现在降级为 SQL 层实现细节，可推给 backend 实现阶段决定，不阻塞已定的图 API 形状原则。
3. **视觉/媒体资产缺口**(§5.2 #13 / §5.4 #12)怎么补——已明确**阻塞依赖**：`PackFile`现纯文本(`{path,content:string}`)，Catalog commit/checkout全链路无二进制支持，需先扩展才能落地。是否在本轮设计范围内排期，还是单独立项。

## 八、不做什么（承接研究 05§2.3/06§4）
