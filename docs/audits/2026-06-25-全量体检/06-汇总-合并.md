## 漏洞按 severity 排序

### P0（10 条）

#### P0-1 · 头号债链路（团本扩展零接线）使场景 C 验收口径不可达
- **hit_by**：产品（PROD-001）、架构（ARCH-003）
- **evidence**：wiki 业务侧承诺"薄到近乎纯声明的 manifest"；spec 承诺 DT-9「团本扩展框架零改动」；但 toolgen 引擎已建（981 行 6 模块）却零接线——present 层有 tensionBoard 聚合函数但未进 buildMcp/TOOLS，GM 调不到"以概念为单位"的读。
- **impact**：场景 C 验收口径（dogfooding 团本零改动跑通）不可达；团本扩展机制未对作者开放；冻结令不撤则所有新 feat 停摆。
- **fix**：守路线图第二批依赖链：视图层投影 → 业务 MCP 工具暴露（把 present/ 已有聚合注册进 TOOLS）→ toolgen 接线 → dogfooding 端到端。

#### P0-2 · 终局判据缺失——场景 A 的"完整一局"无收局口径
- **hit_by**：产品（PROD-002）、架构（ARCH-006）、后端（BE-007）
- **evidence**：成功标准定总判据为"F1/F2/F3 被治住"但未定"一局何时算终局"；ADR-0009 定了 game_end/you_death 唯二出口但"何时敲、由谁敲"未定；core game_end handler 是"GM 调即终局"的纯记录器，无框架侧触发条件校验（不查 sheet 钟/HP/回合数）；`GET /sessions/:id` 的 `ended: false` 硬编码不读 session_meta.ended，REST/WS 信号矛盾。
- **impact**：eval harness 跑多少轮都无法回答"做成了"——GM 不收局则验收口径无法判定通过；玩家体验上"一局永不结束"是可玩性崩坏；违"AI 有讨好本能、不可信"立项前提。
- **fix**：① 架构层推 E2 ADR 定终局触发条件（N 回合 / 主线 Front 钟满 / 玩家死亡三选一）+ "由谁敲"（建议框架据 sheet 钟/watcher 自动敲防讨好）；② `GET /sessions/:id` 读 metaGet(db,"ended") 填 ended 字段修 REST/WS 矛盾；③ 与终局后玩家流程合并设计。

#### P0-3 · 团本构建侧（lore）零 eval——场景 C 的"作者↔构建 GM"全无客观验证
- **hit_by**：产品（PROD-003）、测试（QA-001）
- **evidence**：backlog-core F3 标"现 eval harness 只覆盖跑团(dice)侧，构建团本侧无任何评估"；eval scenarios/ 仅 4 个场景全部 dice 侧，lore 侧零场景文件。
- **impact**：构建工具补全后无客观验证 = 改了不知好坏；import 映射错误率高则团本跑不起来，场景 C 闭环断在作者侧。
- **fix**：补 lore 侧 eval 场景（mock 作者↔真构建 GM，或复用 play-mcp 模式经 build HTTP 驱动）；F3 验收口径须与 H-build-tools 同步定。

#### P0-4 · 零鉴权越权——任意客户端可操作任意 sessionId（含 DELETE 不可逆破坏）
- **hit_by**：用户（USER-006）、架构（ARCH-001/008）、前端（FE-010）、后端（BE-004）
- **evidence**：架构层（接口页 §0 + ADR-0018 ③）定了缝 B「需会话路由+鉴权」，但 Hono app 零中间件、sessionId 即权限；前端 client.ts 全部 fetch 无 Authorization 头、WS 升级无 token、sessionId 前端自造（`s-${commitId.slice(0,8)}` / `${slug}-${random}`）后端无条件接受；DELETE /sessions/:id 零鉴权可 rmSync 整个 session 文件夹（.db + GM raw 日志）；lore 路径 /catalog/commit·/catalog/:tuanbenId/tag·/lore-sessions/:id/messages 同样零鉴权——团本包库可被任意客户端污染。
- **impact**：远程部署场景：任意客户端可读/改/删任意 sessionId 存档（DELETE 不可逆）、可向 Catalog 注入恶意团本/打伪造 tag、可借 /diagnostics/model-test 让后端代发请求到任意目标。违发版闸「多租户隔离」。
- **fix**：① 缝 B 加会话级 token（开局 POST /sessions 返回 sessionId+token），前端 fetch 加 Authorization 头、WS 加 token query param；② sessionId 改后端生成（不由前端自造）；③ DELETE /sessions/:id 必须校验归属；④ lore 路径加作者级鉴权 + Catalog 加 owner_id 列；⑤ 远程部署默认拒绝跨 Origin。

#### P0-5 · F1"掷骰绕过率"无机械检测——assertions 只统计次数，GM 不骰纯 narrate 编结果不会被标记违规
- **hit_by**：测试（QA-002）
- **evidence**：assertions.ts toolStats 只按 event.kind 计数（verdictGated/verdictAuto），无"该骰却没骰"检测；grade.ts L60 注释"F1 时序校验"是 TODO 非实现；GM 遇对抗局面不调 resolve_*、纯 narrate 编"你砍中了巨龙"时 verdictCount=0 不会被标记 fail。
- **impact**：GM 可经"不调掷骰工具、纯 narrate 编结果"绕过 F1 检测——eval 报告 F1 pass 但实际未治住；eval 作为发版依据不可信。
- **fix**：① assertions 加 F1 时序校验实现（verdict.seq 应早于描述它的 narrate.seq）；② scenario.json 加 expects 字段标"此回合应触发对抗"，assertions 比对预期 vs 实际；③ grader 对抗性 rubric 具体化；④ 人评一致性抽查流程落地。

#### P0-6 · F2 软着陆率无机械地板且 baseline 对照报告缺失——验收口径"with 比 baseline 好"无数据支撑
- **hit_by**：测试（QA-003）
- **evidence**：assertions 只有 narrateLeak/missingNarrate/toolStats 三个机械断言，无 F2 软着陆检测；grader.md L21 完全靠 LLM 判；reports/ 仅一份 doctrine 报告，无 baseline 对照报告；grader 自校准风险在 eval harness 无实现（无对抗性 rubric prompt、无人评抽查流程）。
- **impact**：F2 的"做成了"判定目前完全不可信——无机械地板 + 无 baseline + 无 grader 校准。发版闸前若不补则 F2 验收口径不可信。
- **fix**：① 补 baseline 对照跑测（每场景跑 doctrine + baseline 各 N 局，reports/ 落对照报告）；② F2 加弱机械地板（坏结果后 narrate 出现"幸好/不过/但是"转折词作 grader 参考）；③ grader 对抗性 rubric + 人评抽查落地。

#### P0-7 · 发版闸无自动化 gate——四项发版闸全未实现且无 CI 检查/阻断脚本真拦住
- **hit_by**：产品（PROD-012）、测试（QA-004）
- **evidence**：路线图第五批发版闸列四项（承重绑定 ADR + port 契约、prompt injection 威胁建模、模型 key 托管+限流、多租户隔离）全是文档性陈述，无任何自动化 gate；四项当前实现状态全 ✗（ARCH-004/009、USER-007、ARCH-001 实证）；无 CI job、无发版前 checklist 脚本、无阻断机制。
- **impact**：发版闸名不副实——是"发版前提醒单"而非"发版闸"。人忘了一次或赶进度跳过一次就形同虚设；远程部署下零鉴权 + key 泄露 + 注入面全开直接致命。
- **fix**：① 加发版前 checklist 脚本（扫鉴权中间件/SSRF 白名单/key 托管/port 契约 ADR，任一缺失 exit 1）；② CI 加发版闸标签门（release PR 必带 `release-gate-passed` 标签）；③ backlog 三池相关条目加"冻结中/不破坏冻结令"标记与路线图双向对齐。

#### P0-8 · README 承诺"CLI 一键开玩"但 CLI 无 play 子命令——玩家按文档装完卡死
- **hit_by**：用户（USER-001）

- **impact**：新玩家按 README 操作直接卡死——装完 CLI 不知下一步；"随时在线、低安装、本机直接玩到"的卖点在当前实现下不成立。
- **fix**：二选一——① README 明确标注"v1 当前通过 web 客户端玩，CLI 仅会话管理"并给 web 启动步骤；② 补 CLI 的 play 子命令。属发版闸前必对齐的文档-实现一致性。

#### P0-9 · lore 路径零 WS 流接入——LoreSession 有 hub 但 lore API 无 WS 端点，构建过程前端拿不到任何实时流

- **impact**：场景 C 团本构建：作者点"生成团本"后前端只能转圈等待，不知构建 agent 在干什么——构建过程完全黑盒；构建失败/超时前端拿到 202 后永远等不到 turn_ended。

#### P1-1 · 终局后玩家被永久锁死——输入框 disabled、无"再来一局/重开/回滚"入口
- **hit_by**：用户（USER-002）
- **evidence**：PlayPage.tsx L365 `disabled={!!gameEnd}`；L308 终局只渲染 end div，无任何重开/回滚按钮；grep i18n 无 restart/重来/再玩键。
- **impact**：玩家死一次就卡死，违场景 A 卖点"随时能玩"；与 PROD-002 终局判据缺失叠加——玩家死了既不能回滚也不能重开，会话彻底作废。
- **fix**：PlayPage 终局态加"重开本团本（新会话）/ 回滚到上一快照 / 回主页"三按钮，与 E2 ADR 配套设计终局后玩家流程。

#### P1-2 · 错误吞没与错误通道不一致——postMessage/roll/choose 的 .catch(()=>{}) 静默吞错
- **hit_by**：用户（USER-003）、前端（FE-002）
- **evidence**：PlayPage L122 `postMessage(draft).catch(() => {})`、L351 `roll(...).catch(()=>{})`、L337 `choose(...).catch(() => setChosen(null))`；useSession 的 WS error 有 setError 通道但 HTTP 路径错误不进 error 通道，两套错误通道割裂。
- **impact**：玩家发消息/掷骰/选选项静默失败无反馈——最坏情况玩家以为发出去了但 GM 没响应；与 rollGate 409 死锁叠加玩家连"无待掷"提示都看不到。
- **fix**：catch 改为 `catch((e) => { setError(e instanceof Error ? e.message : String(e)); })` 把 HTTP 错误进 error 通道；postMessage 失败时不要清空 draft（挪到 .then 里）；roll 的 409 特殊处理显示"无待掷骰"并给重试/刷新选项。

#### P1-3 · API key 明文存 localStorage + 后端代发 SSRF——key 可被 XSS 偷走，后端可被诱导 fetch 任意 URL
- **hit_by**：用户（USER-007）、前端（FE-005）、后端（BE-005）
- **evidence**：useSettings.tsx L97 `localStorage.setItem(STORAGE_KEY, JSON.stringify(next))` key 明文；client.ts L157 POST /diagnostics/model-test body 明文带 key；diagnostics.ts L64 baseUrl 完全可控无白名单（可 `http://169.254.169.254/` 云元数据、内网服务）；mcp-test 的 endpoint 同样可控，existsSync 可侧信道探测文件系统路径。
- **impact**：玩家 API key 泄露风险（直接经济损失）；后端 SSRF 被滥用（代发请求到内网/恶意目标）；违发版闸「模型 key 托管+限流」+ ADR-0018 ⑤「服务器集中持密钥」。
- **fix**：① key 不存 localStorage 改存后端（玩家输入→后端加密落库→前端只存引用）；② model-test 的 baseUrl 限定协议（https only）+ host 白名单；③ mcp-test 的 endpoint 同样白名单，stdio 的 existsSync 限定已知 MCP 命令前缀；④ 远程部署默认拒绝跨 Origin。

#### P1-4 · GM 超时兜底是"脱困不恢复"——abort 后已落库变更不回滚、已发 narration 不撤回、无重试
- **hit_by**：用户（USER-004）、后端（BE-001）
- **evidence**：DiceGm.ts L124-165 3min 超时触发 controller.abort()，catch 里 yield error 后回合结束；但超时时 GM 可能已调若干工具（sheet_update/narrate 已落 DB）、半条叙述已流给前端——abort 只停了 SDK query，已落库的规范态变更不回滚、已发的 narration_commit 不撤回；turnLoop.ts L33 收到 errored=true 直接 return 不跑 turnEnd。
- **impact**：eval 场景：超时回合的半截变更污染后续回合 GM 上下文；玩家场景：超时后静默失败（前端吞 error），玩家以为"没反应"再发消息，新回合基于半截污染状态跑——游戏态渐进腐坏。
- **fix**：① 短期：超时 error 必须显示给玩家并给"重试/跳过"选项；② 中期：回合级事务——DiceGm runTurn 开始前记 turn_start_seq，超时/error 时 DELETE FROM log WHERE seq > turn_start_seq + 回滚 state；③ 长期：与 ARCH-002 快照接线合并——turnEnd 调 checkpoint()，超时后 restore 到本回合起点。

#### P1-5 · 同一会话并发 handleMessage 无互斥——双击/重发/WS重连+REST 同时触发会双开回合
- **hit_by**：后端（BE-002）

- **fix**：client.ts L159/L164 加 `if (!res.ok) throw new Error("model-test 请求失败：${res.status}")` 与其他函数对齐。

#### P2-11 · Markdown 渲染 XSS 面——当前安全但未来扩展须防
- **hit_by**：前端（FE-006）
- **evidence**：Markdown.tsx L15-31 inline() 手写正则不转义 HTML；但实测 React JSX 子节点默认转义机制保护了所有渲染路径——`<strong>{m[1]}</strong>` 中 m[1] 是字符串 React 转义；grep 全仓无 dangerouslySetInnerHTML/innerHTML/eval——XSS 面实际不存在。
- **impact**：当前无 XSS 风险；但 Markdown.tsx 是手写正则零依赖——未来扩展（链接/图片）若不慎引入 dangerouslySetInnerHTML 或 href/src 不校验协议，XSS 面会开。
- **fix**：① 保持现状加注释明确"禁止引入 dangerouslySetInnerHTML，扩展链接/图片时须校验 href/src 协议（禁 javascript:）"；② 单测补"输入 `<script>alert(1)</script>` 验渲染为文本不执行"防回归。

#### P2-12 · WS 重连竞态——sessionId 变更时旧 refetch Promise 返回后 setSnapshot 覆盖新会话快照
- **hit_by**：前端（FE-001）
- **evidence**：useSession.ts L62-79 connect() 的 onclose 调度 `setTimeout(() => { refetch(); connect(); }, delay)`；refetch 返回慢时切会话后旧 refetch 的 .then(setSnapshot) 仍会跑——setSnapshot 不绑 sessionId 无条件更新当前 hook 实例 state；narration/pendingRoll/reveals 等数组状态在切会话时不重置。
- **impact**：玩家在会话栏切换会话时：新会话短暂闪现旧会话的呈现快照；旧会话的 narration 段落残留；旧会话的 pendingRoll 残留——新会话显示掷骰按钮但本局无待掷，点击 409。
- **fix**：① useSession 在 sessionId 变更时重置所有 state（narration=[]、pendingRoll=null 等）；② refetch 的 .then 里加 sessionId 守卫 `if (currentSidRef.current === sessionId) setSnapshot(s)`；③ 或将 useSession 拆成 key=sessionId 的子组件（React 自动 remount 清 state）。

#### P2-13 · BuildPage 自定义 MCP 配置与后端不联通——前端配置是"摆设"
- **hit_by**：前端（FE-012）
- **evidence**：useSettings 存的 mcpServers 列表仅本地 localStorage；grep mcpServers across client.ts 无任何 API 调用把 mcpServers 发给后端；后端 DiceGm/LoreGm 启动时用的 MCP 是后端构造的固定 dicelore 核心 MCP；testMcp 只测单个 endpoint 连通性不注册到 GM 运行时。
- **impact**：玩家在配置页精心配的自定义 MCP 实际跑团时 GM 调不到——配置无效却无提示，玩家以为"配了就能用"；与 ADR-0018 ⑤ 自定义 MCP 设计叠加——设计定了机制，配置到运行的链路断了。
- **fix**：① 后端加"自定义 MCP 注册"端点，前端把 settings.mcpServers 发给后端；② 或显式标注"v1 自定义 MCP 配置仅本地存储、不参与 GM 运行时"，配置页加提示避免误导。

#### P2-14 · eval harness 自身测试覆盖薄——play-mcp.test.ts 只测 open→start→send 闭环
- **hit_by**：测试（QA-006）
- **evidence**：play-mcp.test.ts 只一个 test 验 open→start→send 闭环；不覆盖 choose/roll/browse/list_scenarios/错误路径；doChoose/doRoll/doBrowse 无任何测试调用；assertions.test.ts 只测纯函数逻辑不测 grade.ts 端到端评分流程。
- **impact**：若 play-mcp 的 choose/roll/browse handler 有 bug，eval 跑到这些动作时会静默失败，eval 报告数据失真却不知是 GM 问题还是 harness 问题；eval harness 的可信度依赖其自身测试覆盖。
- **fix**：① play-mcp.test.ts 补 choose/roll/browse 的 happy path；② 补错误路径（后端 500/WS error/turn 未 ended）；③ 补 list_scenarios 测试验返回非空 + 每个场景符合 schema；④ assertions.test.ts 补 grade.ts 端到端。

## 推导断节（单独小节）

本项目首要漏洞类是**推导断节**——"定义了但无人实现"或"设计自洽但实现零接线"的活体断节。去重后共 **18 条**推导断节，按严重度重列：

### P0 级推导断节（4 条）

1. **头号债 toolgen 零接线（P0-1）**：业务承诺"声明式团本扩展"（场景 C）→ 设计承诺 DT-9「框架零改动」→ 实现零接线（toolgen 引擎建好但不进 buildMcp，present 层聚合函数躺着未进 TOOLS）。**定义了但无人实现**的头号断节。路线图第二批冻结令正在治。

3. **lore 侧零 eval（P0-3）**：业务承诺场景 C 体验 → 测试侧只验 dice 侧、lore 侧零场景文件（F3 未起）。作者侧"做成了"无法判定。

4. **发版闸无 gate（P0-7）**：业务定义发版闸边界（路线图第五批四项）→ 测试/CI 层无对应执行机制（无 checklist 脚本/无 CI 门/无阻断机制）。**定义了 gate 但 gate 不拦**的回溯断节。

5. **快照回滚三层断（P1-6）**：架构层（§3.2 + ADR-0017 + 内层 §4.5 + adapter §8）设计自洽 → 实现三层全断（snapshot 表未建、core 无 checkpoint/restore、Stop 不写快照）。**设计自洽但实现零接线**。

16. **愿景 §4 无承接（P2-7）**：业务增愿景 §4（美观/移动端/社区生态）→ backlog 无承接、里程碑无拆块。**定义了但无人实现**。

### 推导断节的共性病根

项目高速迭代下，业务侧（01）持续升维定位/扩愿景/调分型，架构侧（03/04/ADR）设计成熟度高、推导链自洽，但**"设计自洽"与"实现承接"之间存在系统性 gap**——快照（P1-6）、工具面接线（P0-1）、port 契约（P0-10）、终局机制（P0-2）、lore WS（P0-9）都是"架构定了、实现零/半落地"。这与头号债"toolgen 引擎建好但零接线"同构——是项目**"设计跑在实现前面"**的系统性模式。

## 共性病根归纳

### 病根 1：设计成熟、实现承接系统性 gap

项目架构层（03/04/ADR）设计成熟度高、推导链自洽，但**"设计自洽"与"实现承接"之间存在系统性 gap**——多个核心机制都是"架构定了、实现零/半落地"：

- 快照回滚（P1-6）：ADR-0017 设计自洽，snapshot 表未建 + core 无原语 + Stop 不写快照
- toolgen 接线（P0-1）：引擎建好 981 行但零接线，present 聚合函数未进 TOOLS
- port 契约（P0-10）：ADR-0008 定"承重骑 Claude Code"但无 port 抽象
- 终局机制（P0-2）：ADR-0009 定唯二出口但触发条件空白
- lore WS（P0-9）：LoreSession 持 hub + 共享 streamDriverTurn 但 lore API 无 WS 端点

这与头号债"toolgen 引擎建好但零接线"同构——是项目**"设计跑在实现前面"**的系统性模式。发版闸批次（路线图第五批）须正面收口。

### 病根 2：业务侧升维、下游跟不上

项目高速迭代下，业务侧（01）持续升维定位/扩愿景/调分型，但下游（02 术语表 / 04 设计页 / 06 backlog）跟不上——"业务动了、下游没跟"是 wiki 推导链的系统性断节模式：

- 定位漂移（P2-6）：ADR-0022 品类词统一但术语表/安科安价是什么页没跟
- 玩家分型单源违例（P2-3）：用户与场景用新四切面但术语表/调研页用旧二分
- 愿景 §4 无承接（P2-7）：业务增愿景但 backlog/里程碑无拆块
- D 流畅派无承接（P2-1）：业务定义分型但 backlog 无条目 + 架构无机制
- 死链问题总账（P2-4）：单源迁移到 backlog 三池但下游引用未跟

属 backlog-core M1 wiki 整理范畴，但 M1 是长期任务、无批次。建议把"定位漂移清理"单列一次 organize-wiki 专项（不进路线图批次、但要有完成口径）。

### 病根 3：缝 B 两侧都在"假设对方接了"，实际都没接

缝 B 契约（接口页 §0 + ADR-0018 ③）定 Web 壳需鉴权 + 会话路由 + 配置互通，但前后端实现三层断：

- 零鉴权配合（P0-4）：后端零中间件 + 前端 fetch 无 Authorization + WS 无 token
- 响应体契约不一致（P1-14）：start 端点前端按 `{turnId}` 解析、后端返回 `{sessionId, started}`
- 配置不互通（P2-13）：自定义 MCP 前端配了后端不读

client.ts 全用 `as` 断言、无运行时 schema 校验，契约漂移无防护——TS 编译不报错、运行时静默错。发版闸前须建单源 schema（shared/rest.ts）前后端共用 + 加鉴权 token 通道。

### 病根 4：安全面分散——缝 B 零中间件 + 端点各自为政

零鉴权（P0-4）、SSRF（P1-3）、团本注入（P1-8）、DELETE 越权（P0-4）是四个独立安全面，但病根同——"缝 B 零中间件 + 端点各自为政无统一校验"。发版闸前必须统一收口，非单点修补：

- 零鉴权：sessionId 即权限，知道 id 就能读/改/删任意会话
- DELETE 越权：rmSync recursive force 删整个 session 文件夹不可逆
- SSRF：model-test/mcp-test 的 baseUrl/endpoint 完全可控无白名单
- 团本注入：validatePack 只验结构不验内容，恶意团本 prologue 直接进 GM system prompt
- prompt injection：玩家自由文本直接作 prompt 喂 GM 无隔离

本机单玩场景风险降级（无跨用户/内网探测面），但远程部署（AGPL 网络服务条款 + ADR-0018 Web 壳暗示场景）下直接致命。这是发版闸「多租户隔离 + prompt injection 威胁建模 + 模型 key 托管」三项合一的 P0 风险面。

### 病根 5：eval 跑通了但验收口径没闭合

项目 eval harness 设计成熟度高（play-mcp 连真后端 + grader 对标语料 + findings 分流 A/B），但**"harness 闭环"与"验收口径可判定"之间存在系统性 gap**：

- 场景覆盖边界＝验收边界：list_scenarios 返回的 4 个短场景就是 eval 能验的全部，业务定义的"完整一局/场景 C/终局/长程"在场景层无对应（P0-3）
- 机械地板漏投机：F1 的"绕过率"无机械检测（P0-5），F2 无机械地板全靠 LLM 判（P0-6）——GM 可经"不骰纯编"或"grader 宽容"假性达标
- gate 不拦：发版闸是文档承诺非真 gate（P0-7），live 测试默认 skip 不进 CI（P1-11）——"必做"与"必验"都靠人记着
- 测试覆盖正向不覆盖边界：recovery 测重弹不测死锁（P1-13）、e2e 测开局不测中后段（P1-12）、harness 测 3 工具不测 7 工具（P2-14）——测试全绿下边界病根潜伏

GM 投机取巧三条路径叠加：① 不骰纯 narrate 编结果（F1 绕过）、② 骰出坏结果后软着陆（F2 绕过）、③ 不调 game_end 永不收局（终局绕过）——eval 报告无机械信号能抓住，是 eval 作为发版依据的根基风险。

### 病根 6：脱困不恢复 + 路径不对称

后端实现存在两类系统性 gap：

1. **"脱困不恢复"模式**：超时（P1-4）、rollGate 重启（P2-2）都只解决"卡死"（进程脱困），不解决"半途状态"（数据一致性）——缺回合级事务/快照回滚兜底，与快照未接线（P1-6）同根。
2. **"路径不对称"模式**：dice 路径有 WS 流 + rollGate + turnEnd hook，lore 路径只有 REST + 无 gate + 无 hook——两路径共享底层（streamDriverTurn/LoreSession.hub）但 lore 侧接线缺失（P0-9），违 ADR-0023 ①"两路径共享底层"的对称性。

## 给主流程的提示

### 应优先处理的 P0 链路

1. **头号债链路（P0-1）**：守路线图第二批依赖链（视图层投影 → 工具暴露 → toolgen 接线 → dogfooding），链路跑通前不撤冻结令。
2. **缝 B 鉴权（P0-4）**：发版闸前必做——加会话级 token + sessionId 后端生成 + DELETE 校验归属 + lore 路径加作者级鉴权。这是远程部署的致命面。
3. **eval 可信度（P0-3/5/6）**：补 lore 侧 eval 场景 + F1 时序校验 + F2 baseline 对照——eval 作为发版依据的根基。
4. **发版闸 gate（P0-7）**：加发版前 checklist 脚本 + CI 标签门——让"必做"变成"必拦"。
5. **终局判据（P0-2）**：E2 ADR 定终局触发条件 + GET /sessions/:id 读 meta.ended 修 REST/WS 矛盾。

### 未决问题（留给主流程裁决）

1. **ADR-0017 快照/回滚 v1 是否对玩家开放**（P1-6）？若开放需补 snapshot 表 + core 原语 + Stop/UserPromptSubmit hook 接线 + REST 端点 + UI 五层；若不开放需 ADR-0017 + 完整使用闭环显式降预期。这是架构决策，不宜 AI 代断。
2. **终局机制"何时敲/由谁敲"**（P0-2）——PO 给了候选（N 回合/Front 钟满/玩家死亡），但"框架据 sheet 钟自动敲 vs GM 自调"是架构决策（防讨好 vs 灵活性），须 E2 ADR 定。
3. **D 流畅派 v1 是否实现**（P2-1）——若不实现需显式标注"D 分型架构落点 v1 不做、仅理论锚点"；若实现需开 backlog 定"延迟预算/agent 数上限"约束。
4. **README "CLI 一键开玩"（P0-8）**是改文档对齐实现，还是补 CLI play 子命令对齐文档？这是产品决策（v1 玩家入口形态）。
5. **lore 路径 WS 流式（P0-9）v1 是否补**？若补需加端点 + 正则扩展 + 前端构建台接 WS；若不补须显式删死代码并 wiki 标注。
6. **eval 场景覆盖度扩展顺序**（P0-3）——先补 lore 侧（场景 C 验收）还是先补终局场景（场景 A 完整一局验收）？两者都是 P0，但依赖不同。
7. **自定义 MCP 配置（P2-13）v1 是否接入 GM 运行时**？若接入需后端加注册端点 + 前端发配置；若不接入须配置页显式标注避免误导。
8. **API key 存储方案（P1-3）**——前端不存改后端托管 vs v1 本机风险自担 vs sessionStorage 折中？属发版闸「模型 key 托管」决策。
