## 锚定单

### 用户画像

**两类直接用户 + 一类塑形对象**（据 [用户与场景 §三类角色](../../wiki/01-业务分析/用户与场景.md)）：

| 角色 | 是谁 | 核心诉求 | 痛点 |
|------|------|---------|------|
| **玩家** | 想玩文字冒险游戏的人（单人） | 尊重骰子的真实感、剧情不崩坏、随时能玩 | 人类 PO 一咕就掉线整局死掉；裸 AI 当 GM 会跳骰/软着陆/替玩家选（F1/F2/F3） |
| **团本作者** | 设计一个团的人（世界观+机制+卡池/随机池数据） | 专注"我这个团特有的东西"，主持逻辑别从零造；规则能运行时调整 | 只能手搓 system prompt 不可移植；规则书膨胀到十几万字爆上下文 |
| **被塑形的 AI-GM**（塑形对象，非用户） | 被嫁接框架的那个 agent | 需被兜住"骰子/随机不可绕过"、被教会"别讨好、声明后果在先" | 凭本能就 F1/F2/F3 |

**接入方 = 团本作者**：拿 Dicelore 嫁接到自己的 agent 上、填一个团、开跑。玩家通过这个团间接消费框架。v1 不预设商业角色、不走商业化（AGPL-3.0-or-later，[ADR-0022](../../wiki/05-决策记录-ADR/README.md)）。

**玩家需求四类切面**（2026-06-25 多源调研归纳，[报告](../../../reports/2026-06-25-玩家分型调研.md)；非互斥人口 segment、可叠加；中国侧证据偏弱，落地前须在欧美+中国玩家中做定性验证）：

| 分型 | 核心诉求 | 架构落点 |
|------|---------|---------|
| **A 单人沉浸派** | 一个人也能玩到 D&D 式体验 | 行为塑形层（让 AI 独立顶住骰子、不崩） |
| **B 权威保真派** | 规则不糊弄、状态不脱轨、GM 不讨好 | 行为塑形层 + 数据层 `rule` 域只读契约 |
| **C 协同减负派** | AI 帮人记账/补即兴/管整合性，主持权仍在人 | 数据层（sheet/event/world）+ 部分行为约束 |
| **D 流畅体验派** | 交互不掉帧、回合顺滑 | 约束行为塑形层的多智能体编排深度（agent 数/反馈轮次 vs 延迟 trade-off） |

**D 是旧分型（替代派/相棒派）缺失的维度**——多智能体架构使响应延迟翻倍、>75% 玩家嫌慢（Minori 等 2025），延迟与规则保真是一等 trade-off。旧二分已被本四切面取代（见本领域漏洞 PROD-006 单源违例）。

### 核心场景

**场景 A — 单人文字冒险游戏（现在做，路线图唯一主线）**
玩家本机装 Claude Code（可接各种大模型含国产）+ `dicelore` CLI 一键脚手架即开玩。玩家选方向，框架强制掷骰/给选项、维护人物卡与剧情状态，AI 据结果叙述且不软着陆。形态接近"可交互的个人小说 / ソロ TRPG"。**卖点不是人味，是"随时在线 + 尊重骰子 + 不崩坏"的可靠 GM**；跨端（Win/Mac/Linux）、低安装、本机直接玩到。

**场景 C — 团本作者准备一个团（贯穿始终）**
作者不写主持逻辑，只做"我这个团特有的"三件事：世界观设定、机制差异、随机池/卡池数据。理想形态是薄到近乎纯声明的 manifest（选 skill + 填数据），框架负责把它跑起来。

**完整使用闭环（从头到尾）**：
1. 作者侧（场景 C）：作者经构建台（Web 门面 + 构建 MCP/Skill）声明团本 → import 物化为四域 store + 叙事域（front/plotline/foreshadow/anchor）→ Catalog 落库。
2. 玩家侧（场景 A）：玩家 `dicelore play` 起 play 会话 → GM agent 经 MCP 读团本 + 塑形 skill 主持 → 每轮：玩家行动 → 塑形层判断该给选项/掷骰 → resolver 产出叙述者无法伪造的结果 → narrate 散文 → 状态写入 sheet/event → 推进剧情。
3. 终局：game_end / you_death 出口收局（[ADR-0009](../../wiki/05-决策记录-ADR/README.md)）。

**场景 B — 多人论坛形态（已弃，2026-06-25）**
原"AI 当服务器/裁判、玩家间起哄玩梗"形态**已从路线图移除**。安价作为"单人 TRPG 最简实现 + AI 规划语料"身份保留（见本领域漏洞 PROD-005 弃用场景下游未清）。

### 需求边界

**承诺做（v1）**：
- **行为塑形**：用 L1 工具强制（必掷骰、声明后果在先、给选项/掷骰分流）+ L2 skill 教（Agenda/Principles/Moves）+ L3 Hook 审计（绕过率/后果-叙事比对）对抗 F1/F2/F3。
- **记账担子**：四业务域（sheet/event/world/rule）SQLite + FTS5 朴素兜底；可见性 deny-by-default + show/reveal_once。
- **团本构建**：构建台（Web 门面 + 构建 MCP/Skill）让作者声明式造团；manifest 薄到近声明。
- **跨端分发**：npm 包 + `dicelore` CLI、跨端预编译、骑 Claude Code 作基底（[ADR-0008](../../wiki/05-决策记录-ADR/README.md)）。
- **界面/平台层通用**：可对接多 agent/model（[ADR-0022](../../wiki/05-决策记录-ADR/README.md)），跑团套件特化但不拒绝陪伴品类的界面接入。

**承诺不做（明确非目标，[成功标准 §3](../../wiki/01-业务分析/成功标准.md)）**：
- **不追求取代人类 GM 上限**：即兴惊喜、情感共鸣、赛后感想戦、玩家间人际碰撞——只稳住下限。
- **跑团套件不为纯陪伴/角色拟真优化**：强制掷骰/外置状态在纯陪伴场景是套件取舍（非产品级品类拒绝，界面层通用）。
- **不造游戏引擎**：状态机/约束验证器/效果调度器不进主线；记账用最朴素数据层兜住。
- **单人阶段不追求玩家间碰撞**（多人论坛已弃）。
- **不追求叙事质量绝对值**：主判据是"讨好本能被压下去"，叙事质量只在"明显劣化影响可玩性"时作护栏。
- **v1 不解长程语义召回**：软伏笔的"相关时自动想起"是 RAG 难题，v1 用 FTS5 + tag + agent 自己读凑合；结构化部分（目标/未结线索/世界硬状态）照常显式追踪。
- **不商业化**：AGPL-3.0-or-later、无双授权。

### 验收口径

**总判据**（[成功标准](../../wiki/01-业务分析/成功标准.md)）：成功 = 相比"裸 agent"，AI 的讨好本能被压下去、稳定地不犯 F1/F2/F3。**不是"叙事多精彩"，而是"该让玩家失败时它真的让失败发生"**。

**每个核心场景的"做成了"判定**：

| 场景 | 验收口径 | 当前状态 |
|------|---------|---------|
| **场景 A · 单人跑团** | with/without baseline 范式：F1 掷骰绕过率（脚本可测，with 应趋近 0）、F2 不利结果后叙事软着陆率（人评/裁判 agent，with 显著低于 baseline）、F3 工具选择与情境匹配率（人评，with 显著高于 baseline）。**相对判据非绝对分数**。 | 🟡 eval harness 已闭环（play-mcp，[ADR-0025](../../wiki/05-决策记录-ADR/README.md)），F1 闭环；F2 终局观测待多轮跑测。裁判 agent 自校准风险已显式防（对抗性 rubric + 人评一致性抽查） |
| **场景 C · 团本构建** | 存在一个 dogfooding 团本，仅靠"声明 + flow"跑通完整一局（含 front/plotline/foreshadow/tension_board 的读），**框架 core 代码零改动**（守 spec DT-9 契约）。 | 🔴 未达——toolgen 引擎已建（981 行）但零接线：视图层未投影 / 业务 MCP 工具未暴露 / NPC 无一等抽象。spec 承诺的"新剧本=新声明+新 flow"目前跑不通（[backlog-core 主题A′](../../wiki/06-里程碑与问题/backlog-core.md)）。**这是头号债** |
| **团本构建侧 eval** | 构建流程"作者↔构建 GM"交互质量、import 映射正确性、构建工具可用性有客观验证（mock 作者↔真构建 GM 或复用 play-mcp 经 build HTTP 驱动）。 | 🔴 零 eval——现 harness 只覆盖跑团 dice 侧，lore 侧全无评估（[backlog-core F3](../../wiki/06-里程碑与问题/backlog-core.md)） |
| **终局判据** | gm-core 教"别朝结局叙事"也教"何时收局正当"；团本作者能在 rule 里配置终局条件（含与复活/快照回滚的交互）。 | 🔴 E1（终局判据话术）待 F2 harness 多轮数据再动；E2（终局机制数据层）需 ADR（[backlog-core E1/E2](../../wiki/06-里程碑与问题/backlog-core.md)） |

**发版闸（里程碑三前必做，[路线图第五批](../../wiki/06-里程碑与问题/路线图.md)）**：承重绑定 ADR + port 契约、prompt injection 威胁建模、模型 key 托管+限流、多租户隔离。
