# 前端 · 落地
> 上游文档：02-设计-架构.md
> 本文档：作为前端工程师（对 apps/web 负责），评估实现是否落地了架构契约——主扫状态管理 bug、错误处理缺失、缝 B 契约两侧一致性、测试缺口与极端输入、XSS 与敏感信息前端泄露。

## 上游回放

接住 02-设计-架构.md 的关键要点（前端相关投射）：

- **ARCH-001/008（零鉴权 P0）**：缝 B（后端↔web）架构层定了「需会话路由 + 鉴权」但实现零中间件——sessionId 即权限（知道 id 就能读/改/删任意会话）、lore 路径团本包库可被任意客户端 commit/tag。给前端的提示：开局须拿 session token、后续 REST + WS 升级带 token；`useSession.ts` 的 fetch/WS 连接加 Authorization 头。
- **ARCH-002（快照未接线 P1）**：ADR-0017 快照/回滚设计自洽但实现三层全断——snapshot 表未建、core 无 checkpoint/restore、orchestrator Stop 不写快照。给前端的提示：PlayPage 回滚 UI 的接线依赖后端 snapshot 表 + REST 端点（`POST /sessions/:id/rewind`），架构侧确认底下没有——前端别先做回滚按钮（做了也点不动）。
- **ARCH-006（终局机制 P1）**：game_end 后"重开/回滚/主页"三按钮——"回滚"依赖 ARCH-002，"重开"用同团本开新 session。
- **ARCH-007（rollGate 重启死锁 P2）**：后端重启后 pending roll 卡死直到超时；roll 的 409 错误须显示给玩家（与 [01 USER-003] 错误吞没合并修）。
- **ARCH-009（prompt injection P1）**：玩家自由文本直接进 GM 上下文。给前端的提示：玩家输入与 GM 上下文结构化分隔属后端职责，但前端可在 UI 上标注"你的输入会原样发给 GM"提示。

接住 01-验真-用户.md 的前端相关要点：USER-002（终局后玩家锁死无重开）、USER-003（错误被 `.catch(()=>{})` 静默吞）、USER-005（存档/回滚无 UI 入口）、USER-007（API key 明文存 localStorage + 后端代发 SSRF）、USER-008（e2e 只覆盖开局段）、USER-009（长对话上下文爆炸无前端护栏）、USER-010（rollGate 重启死锁）。

我作为前端工程师，只验「状态管理 bug / 错误处理缺失 / 缝 B 契约两侧一致 / 测试缺口 / XSS 与敏感信息前端泄露」，不验后端单接口实现（属后端）与架构层契约设计（属架构）。

- finding_id: FE-001
  role: 前端
  title: WS 重连竞态——sessionId 变更时旧 connect() 的 setTimeout 已调度但 cleanup 只关 ws、不清在途 timer 之外的新 timer；reconnect 期间新 WS 与旧 WS 可能并存触发重复消息
  kind: 领域漏洞
  layer: 实现
  severity: P2
  evidence: >
    `apps/web/src/live/useSession.ts` L62-79 connect()：`ws.onclose = () => { ...; timer = setTimeout(() => { refetch(); connect(); }, delay); }`，
    cleanup L78 `return () => { closed = true; if (timer) clearTimeout(timer); ws?.close(); }`。
    问题路径：① sessionId 变更触发 effect 重跑→cleanup 调 `ws?.close()`，但 `ws.close()` 是异步的（触发 onclose），onclose 里的 `if (closed) return` 守住了不再重连，OK；
    ② 但若 onclose 在 cleanup 之后才触发（close 是异步事件），`closed` 已 true 故不会重连——这条 OK；
    ③ 真正的竞态：网络抖动导致 onclose 触发→`timer = setTimeout(connect, delay)` 调度重连→在 delay 到期前用户切会话（sessionId 变）→cleanup 跑 `clearTimeout(timer)`（此时 timer 有值，清掉了）+ `ws?.close()`（但 ws 已经是 close 状态的旧 ws）→effect 重跑新 connect() 创建新 ws。
    表面看 OK，但 `timer` 变量在 closure 内被 onclose 写、cleanup 读——若 onclose 在 cleanup 之后异步触发（浏览器 close 事件延迟），onclose 会读到 `closed=true` 提前返回，不会写新 timer，OK。
    实际竞态点：refetch() 在 onclose 调度前无脑调（L73 `timer = setTimeout(() => { refetch(); connect(); }, delay)`——refetch 在 setTimeout 回调内），但若重连期间 refetch 返回慢（网络差），切会话后旧 refetch 的 .then(setSnapshot) 仍会跑——setSnapshot 是闭包绑死旧 sessionId 的，但 setSnapshot 会更新 state（React 不区分是哪个 sessionId 的 snapshot），导致新会话闪一下旧会话的呈现快照。
  description: >
    useSession 的 WS 重连逻辑在「重连 in-flight 期间切会话」时存在状态污染：旧 sessionId 的 refetch Promise（GET /sessions/旧id/presentation）返回后 setSnapshot 会覆盖新 sessionId 的快照，
    因为 setSnapshot 不绑 sessionId（React state setter 无条件更新当前 hook 实例的 state）。
    同时 narration/pendingRoll/reveals 等数组状态在切会话时不重置——useSession 的 useState 初始化只在首次 mount 跑，sessionId 变更时 effect 重跑但 state 不清零，旧会话的 narration 段落会残留到新会话视图。
  impact: >
    玩家在会话栏切换会话（PlayPage L224 `onChange={(e) => navigate(/play/${e.target.value})}`）时：
    ① 新会话短暂闪现旧会话的呈现快照（refetch 竞态）；
    ② 旧会话的 narration 段落残留（state 不重置）——玩家看到上一局的叙事文本混进本局；
    ③ 旧会话的 pendingRoll 残留——新会话显示掷骰按钮但本局无待掷，点击 409。
    单人本机场景切换不频繁，但会话栏切换是 PlayPage 一等功能（次级 bar 可切换/删除会话），残留会误导玩家。
  suggested_fix: >
    ① useSession 在 sessionId 变更时重置所有 state（narration=[]、pendingRoll=null、reveals=[]、gameEnd=null、error=null、generating=false）——加 `useEffect(() => { reset(); }, [sessionId])` 或在 refetch 里校验返回的 sessionId 与当前一致再 setSnapshot。
    ② refetch 的 .then 里加 sessionId 守卫：`getPresentation(sessionId).then(s => { if (currentSidRef.current === sessionId) setSnapshot(s); })`。
    ③ 或将 useSession 拆成「key=sessionId」的子组件（React 自动 remount 清 state），最干净。
  confidence: medium
  hit_by: [前端]

  suggested_fix: >
    ① PlayPage 的 postMessage/roll/choose 的 catch 改为 `catch((e) => { setError(e instanceof Error ? e.message : String(e)); })`——把 HTTP 错误也进 error 通道（复用 useSession 的 setError，或 PlayPage 本地 error state）；
    ② 或 useSession 的 postMessage/roll/choose 在 catch 里 setError，统一错误通道（HTTP + WS 都进同一 error state）；
    ③ postMessage 失败时不要清空 draft（L122 把 setDraft("") 挪到 .then 里），让玩家能重发；
    ④ roll 的 409 特殊处理：显示"无待掷骰，可能后端已重启/会话状态已变"并给"重试/刷新"选项（与 ARCH-007 修复联动）。
  confidence: high
  hit_by: [前端]

  hit_by: [前端]

- finding_id: FE-006
  role: 前端
  title: Markdown 渲染用 dangerouslySetInnerHTML 等价路径——inline() 手写正则不转义 HTML，GM 叙事/玩家输入含 <script>/<img onerror> 时存在 XSS 面
  kind: 领域漏洞
  layer: 实现→安全
  severity: P2
  evidence: >
    `apps/web/src/play/Markdown.tsx` L15-31 `inline(text, kb)`：
    ```
    const re = /\*\*([^*]+)\*\*|\*([^*]+)\*|`([^`]+)`/g;
    ...
    out.push(text.slice(last, m.index));  // 原样 push 文本片段
    ```
    L33-60 Markdown 组件把 text 按段落/标题/列表 split 后调 inline()——**未对原始 text 做 HTML 转义**。

    玩家在配置页精心配的自定义 MCP，实际跑团时 GM 调不到——配置无效却无提示，玩家以为"配了就能用"。
    与 [ADR-0018 ⑤] 自定义 MCP「产出作叙述流回」的设计叠加——设计定了自定义 MCP 机制，但前端配置到后端运行的链路断了。
    非阻塞（核心 dicelore MCP 可用），但自定义 MCP 是 ADR-0018 ⑤ 的落地项，当前是"配置摆设"。
  suggested_fix: >
    ① 后端加"自定义 MCP 注册"端点（POST /diagnostics/mcp/register 或开局时带 mcpServers 配置），前端 client.ts 把 settings.mcpServers 发给后端；
    ② 或显式标注"v1 自定义 MCP 配置仅本地存储、不参与 GM 运行时"，配置页加提示"当前版本自定义 MCP 配置暂未接入 GM 运行时"——避免玩家误以为配了能用；
    ③ 与 ARCH-001 鉴权叠加：自定义 MCP 注册端点也须鉴权（恶意客户端可注册窃密 MCP）。
    属 ADR-0018 ⑤ 落地项，发版闸前须明确 v1 是否接入。
  confidence: medium
  hit_by: [前端]
```

## 推导断节

（上方 YAML 中 kind=推导断节 的条目按严重度重列聚焦：）

- **FE-003（P1）**：缝 B 契约 `POST /sessions/:id/start` 响应体两侧不一致——前端 client.ts 按 `{turnId}` 解析、后端 dice.ts 返回 `{sessionId, started}`。当前不崩只因 PlayPage 不消费返回值，但 client.ts 的类型契约是谎言（as 断言绕过 TS）。与 ARCH-001 鉴权断节叠加：start 端点也无鉴权。**缝 B 两侧契约不一致的活体断节**——前端假设的响应形状与后端实际暴露的不匹配，前后端角色交叉命中=高置信。
- **FE-010（P0）**：缝 B 零鉴权的前端配合缺失——client.ts 全部 fetch 无 Authorization 头、useSession WS 升级无 token、sessionId 前端自造（HomePage `s-${commitId.slice(0,8)}` / CatalogPage `${slug}-${random}`）后端无条件接受。与 [ARCH-001] 同根——架构侧已标 P0，前端侧确认实现属实且三层全断（架构定了、后端零中间件、前端零配合）。**前后端角色交叉命中=高置信断节**。
- **FE-012（P2）**：BuildPage 自定义 MCP 配置与后端不联通——前端 useSettings 存的 mcpServers 列表从未发给后端，后端 GM 用固定 MCP，前端配置是"摆设"。缝 B 契约断节——前端假设"配了 MCP 后端会用"、后端"不读前端配置"。与 [ADR-0018 ⑤] 自定义 MCP 设计叠加——设计定了机制，配置到运行的链路断了。

**共性病根（前端侧）**：缝 B 契约（接口页 §0 + ADR-0018 ③）定 Web 壳需鉴权 + 会话路由 + 配置互通，但前端实现三层断——① 零鉴权配合（FE-010）、② 响应体契约不一致（FE-003）、③ 配置不互通（FE-012）。这与 [ARCH-001/008] 后端侧的零鉴权同构——**缝 B 两侧都在"假设对方接了"，实际都没接**。client.ts 全用 `as` 断言、无运行时 schema 校验，契约漂移无防护——TS 编译不报错、运行时静默错。发版闸前须建单源 schema（shared/rest.ts）前后端共用 + 加鉴权 token 通道。
