> **状态**：🟢 已成型（迁自北极星文档；2026-06-01 据 01 补强塑形层 / 担子映射 / 第三类问题对齐）。

## 0. 为什么需要这层概念

设计之所以会"零碎、混乱"，根因是**三个层被同时设计**。任何一层在另一层还没定形时去对接，都会乱。
解药：**承认有三层，一次只设计一层。**

```
③ 塑形层   skill / guideline / 反讨好（F1/F2/F3）
② 行动层   裁决工具：三种骰 + 给选项
① 数据层   三种存储：状态 / 事件 / 剧本
```

**这三层是对 [01 问题域](../01-业务分析/问题域.md) "镜像两副担子"的概念回应：**

| 01 的担子 | 由哪层接 |
|---|---|
| **记账担子**（维护状态，能力问题） | ① 数据层 |
| **失控倾向**（尊重骰子 / 不软着陆，行为问题） | ② 行动层的强制工具 + ③ 塑形层 |

记账担子用最朴素的数据层兜住；失控倾向才是重头——它落在"行动层产出不可编造的结果 + 塑形层对抗讨好本能"。下面三层按 ①→②→③ 展开。

> 这套"执行离手、状态外置、参照锁源"的取向有外部佐证：中日英独立开发者各自撞墙后收敛到同款架构，见 [调研-期待与预测](../01-业务分析/调研-期待与预测.md) 第三节。

## 1. 贯穿全项目的原则：agent 即引擎

anko_driver **嫁接在 AI agent 上**。所以：

- **agent 本身就是阅读器和召回器。** 不自建 embedding / 向量库 / ML 流水线。
- 数据层只需 **SQLite（KV + FTS5 全文搜索）** 就够 v1。
- **任何"要不要上 MongoDB / 向量检索 / RAG / ML"的冲动，默认答案是"v1 不上"。** 真有必要的单开 spike，不进主线。

## 2. 数据层：三种存储，按"访问模式"区分

> **关键认知纠偏**：区分这三张表的**不是"结构化 vs KV"**，而是 **AI 怎么访问它们**。

| 存储 | 本质（存什么） | 访问模式（怎么用） | 形态 |
|------|----------------|--------------------|------|
| **状态表** | 此刻为真的事实：人物卡、货币、计时器、商店库存 | **精确 get / set**，可被骰子读写 | KV / 文档（值为 JSON） |
| **事件表** | 发生过 / 将发生的记忆：剧情、伏笔、走向 | **追加 + 召回**（按相关性取回） | 带标签的叙事条目 |
| **剧本表** | 作者写死的世界：设定、门派、NPC、卡池 | **只读，搜索 + 随机抽样** | FTS5 文本 + 带属性的池条目 |

- **状态表**回答"现在是什么样"；**事件表**回答"发生过什么 / 埋了什么线"；**剧本表**回答"这个世界本来是什么样"。

## 3. 行动层：裁决工具是"数据层的读写器"

行动层产出**结果的归属**（结果不能由叙述者自由编造）。两个家族：**给选项**（结果归玩家）和**掷骰**（结果归随机）。

### 3.1 掷骰的三种形态——由"操作数从哪来"区分

> 安科传统里 d10 的 "0" 常指 10。**本项目是 AI 安科，不必遵循传统**——只要"AI 声明的判定范围"和"骰子产出的结果"能对上即可。骰面语义交给规则书。

| 骰种 | 操作数来源 | 产出 | 是否碰状态表 |
|------|------------|------|--------------|
| **选项骰** | GM 预设的选项表 | 命中某选项 → **叙事档位** | 否 |
| **对抗骰** | 状态表的两个属性 | 胜 / 负 | **读**状态表 |
| **状态骰** | 骰子 ± 状态表某数值（如 `金币 + 5d3000`） | 一个**数值** | **读+写**状态表 |

聚类：**裁决骰**（选项骰，出叙事后果）+ **数值骰**（对抗/状态骰，与状态表耦合）。

**消解"骰子设计很乱"的关键洞察：掷骰不是独立子系统，它是状态表的读写器。** 对抗骰/状态骰存在的理由，正是战斗团要的不是叙事档位而是血量/伤害这种**数值**。

### 3.2 给选项

结果归玩家。后果在玩家选之前锁定（"声明后果在先"）。单人转轮 / 多人投票的差异属塑形/模式层。

## 4. 塑形层：把通用 agent 变成"不讨好的好 GM"

塑形层是 anko_driver 的差异化所在（见 [01 问题域](../01-业务分析/问题域.md)）。它的职责单一：**对抗 AI 的讨好本能，让它当好 GM**——尊重骰子、声明后果在先、该让玩家选时不替选。

> **关键认知：L1/L2/L3 不是第四层，而是和"三层"正交的另一个轴。** 三层（数据/行动/塑形）= 系统**由哪些部件构成**；L1/L2/L3 = 塑形这件事**靠什么强度达成**。

三种杠杆，可靠性递减、适用对象不同：

- **L1 工具强制**：借行动层那些**必经工具**，让正确行为在结构上不可绕过——必掷骰、声明后果在先、给选项/掷骰用不同工具名分流。最硬，但只能管"能被结构卡住"的部分。
- **L2 skill / guideline 教**：塑形层本体。教 AI 何时用哪种裁决、别软着陆、怎么判断"该选还是该骰"。可被忽视，但是**判断类问题的唯一载体**。
- **L3 Hook 审计**：事后比对、抓违规（掷骰绕过率、声明后果与叙事是否一致）。兜底，不阻止当下。

**三失败 × 三杠杆的主次矩阵**（★ = 主力）：

| 失败模式 | L1 工具强制 | L2 skill 教 | L3 Hook 审计 |
|---|---|---|---|
| **F1 跳骰** | ★ 必掷 | 流程 skill | 绕过率审计 |
| **F2 软着陆** | ★ 声明后果在先 | 禁令措辞 | 后果-叙事比对 |
| **F3 选错方式** | 工具名分流 | ★ 决策框架（靠教） | 错配抽查 |

→ **可靠性 L1 > L2 > L3，但越靠"判断"的失败（F3）越只能靠教（L2）。** 这正是项目重心落在"用提示词教 AI 当 GM"而非堆代码的原因：能结构化卡住的（F1/F2 的硬约束）尽量下沉到 L1，卡不住的判断（F3）交给 L2。

> 本节只定概念。skill/guideline 的具体注入机制、Hook 的实现、工具响应"补刀"的形态，留给 03/04（见 §7）。

## 5. 两个研究级难题：明确隔离，v1 用朴素版

绝不能在主线里顺手解决，否则项目永远卡在它们上。各自单开 spike：

| 难题 | 为什么难 | **v1 朴素版（主线）** | 真·解法（spike） |
|------|----------|------------------------|------------------|
| **事件表的语义召回** | 本质是 RAG | 事件写 markdown，**FTS5 搜索 + agent 自己读** | 向量 embedding / 语义检索 |
| **让 AI"读懂"世界设定** | 设定要进 AI 判断，不能只躺库里 | (a) 核心调性安装时**焊进上下文**；(b) 相关时 `knowledge_search`；(c) agent 直接读 markdown | 设定结构化推理 / 自动一致性检查 |

> **与 [01 问题域 §5](../01-业务分析/问题域.md) 对齐**："长程叙事一致性"问题里，**目标 / 未结线索 / 世界硬状态属结构化**，归状态表 / 事件表显式追踪、现在就做，**不在本表 spike 范围**；只有**软伏笔的"相关时自动想起"**才是上表"事件表语义召回"这个 RAG 难题，v1 朴素兜底、观测后再定。

## 6. 三层如何咬合（数据流一瞥）

```
玩家行动
  → 塑形层判断：该给选项？该掷骰？哪种骰？
      ├ 给选项  → offer_choice            （结果归玩家）
      └ 掷骰
          ├ 选项骰 → roll_outcome          （出叙事档位）
          ├ 对抗骰 → 读状态表两属性 → 裁决  （出胜负）
          └ 状态骰 → 读状态表数值 ± 骰 → 写回状态表（出数值）
  → 叙述（受塑形层"别软着陆"约束 + 工具响应补刀）
  → 需要世界/卡池内容时 → 剧本表（search / sample）
  → 推进后剧情/伏笔 → 事件表（写）；回顾时 → 事件表（召回）
```
