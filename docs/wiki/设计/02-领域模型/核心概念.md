# 核心概念：三层 / 三存储 / 三骰

> **本页职责**：定死本项目用来描述解法的概念与词汇——这是整个 wiki 的"领域骨架"，下游架构与子系统设计都引用本页。
> **上游依赖**：[01-业务与定位](../01-业务与定位/)（本页是对那些问题的概念回应）。
> **状态**：🟢 已成型（迁自北极星文档；2026-06-01 据 01 补强塑形层 / 担子映射 / 第三类问题对齐）。

> 词条级的精确定义见 [术语表](../../术语表.md)。

## 0. 为什么需要这层概念

设计之所以会"零碎、混乱"，根因是**三个层被同时设计**。任何一层在另一层还没定形时去对接，都会乱。
解药：**承认有三层，一次只设计一层。**

```
③ 塑形层   Agenda 议程 → Principles 原则 → Moves 动作（反讨好 F1/F2/F3）
② 行动层   resolver：给选项 + 三种骰（选择者 玩家 / 随机）
① 数据层   四业务域：实体卡 / 事件 / 世界 / 规则
```

**这三层是对 [01 问题域](../01-业务与定位/问题域.md) "镜像两副担子"的概念回应：**

| 01 的担子 | 由哪层接 |
|---|---|
| **记账担子**（维护状态，能力问题） | ① 数据层 |
| **失控倾向**（尊重骰子 / 不软着陆，行为问题） | ② 行动层的强制工具 + ③ 塑形层 |

记账担子用最朴素的数据层兜住；失控倾向才是重头——它落在"行动层产出不可编造的结果 + 塑形层对抗讨好本能"。下面三层按 ①→②→③ 展开。

> 这套"执行离手、状态外置、参照锁源"的取向有外部佐证：中日英独立开发者各自撞墙后收敛到同款架构，见 [调研-期待与预测](../01-业务与定位/调研-期待与预测.md) 第三节。

## 1. 贯穿全项目的原则：agent 即引擎

Dicelore **嫁接在 AI agent 上**。所以：

- **agent 本身就是阅读器和召回器。** 不自建 embedding / 向量库 / ML 流水线。
- 数据层只需 **SQLite（KV + FTS5 全文搜索）** 就够 v1。
- **任何"要不要上 MongoDB / 向量检索 / RAG / ML"的冲动，默认答案是"v1 不上"。** 真有必要的单开 spike，不进主线。

## 2. 数据层：三层架构 + 「不变/当前」正交轴

> 承接 D2（业务域一级轴）+ D5（运行时改名 sheet→state 等）+ D6（A′ 三层重构，本节落地）。
> 业务域（sheet/event/world/rule）仍是"存哪一类内容"的分类轴（D2 判断不变）；A′ 在其上加了**三层架构 +「不变/当前」正交轴**，解"同一概念在作者层有一等形态、到运行时却塌成底层存储原语、AI 拿不到以概念为单位的读写"的 conceptual integrity 债（[backlog-core 主题 A′](../05-现状与计划/backlog-core.md)）。

### 2.1 三层 + 正交轴

| 层 | 职责 | 性质 |
|----|------|------|
| **物理层** | 少数几张表，存储唯一不重复 | 兜概念 |
| **视图层** | 按 `kind`+facet 投影，**零存储**（单源不复制状态）| 给 AI 干净业务面 + 对下游 toolgen 稳定逻辑列契约 |
| **工具层** | 一大批业务场景 MCP | AI 按场景调数据 |

**正交轴「不变/当前」**：作者写死的（lore/rule/pool）读为主、写路径受限；随剧情变的（state/叙事/记忆）高频可变。决定写路径与快照范围——[快照](../04-子系统设计/内层能力库.md)只回滚"当前"域、rule 带外热更不随回合回滚。

### 2.2 物理表（存储唯一）

| 表 | 兜概念 | 不变/当前 |
|---|---|---|
| `state` | player/npc/world 当前状态 + relation + flag + clock（**六概念塌一张表**，靠 `kind`+facet 分化，非六张表）| 当前 |
| `lore` | 作者世界散文底料（FTS5）| 不变 |
| `rule` | 机制/判定合同（AI 只读、人版本化）| 不变 |
| `pool` | 卡池/随机表 | 不变（GM 可现编）|
| `front` | 威胁层（引用一个 state clock）| 当前 |
| `plotline` | 故事线（status 状态机）| 当前 |
| `foreshadow` | 伏笔（planted/recalled 状态机）| 当前 |
| `log` | append-only 原始流水 + moment 标记 | 当前 |
| `history` | 周期 compact 蒸馏记忆 | 当前 |
| `watcher` | 底层数值触发器 | 当前 |

> 叙事三表（front/plotline/foreshadow）是 A′ 从 event 域独立出来的**一等抽象**（D6）——有 status 状态机 + 聚合视图（`tension_board`），不再散落 event log 靠 FTS 硬搜。schema 详情 + 欠账项（`front_omen` 子表、`plotline`/`foreshadow` 的 `anchors_json`+`surface_trigger` 列）见 [内层能力库 §4](../04-子系统设计/内层能力库.md)。

### 2.3 视图层（零存储投影）

按 `kind`+facet 从物理表投影出业务视图，**不存副本**：6 个内建视图（`player`/`npc`/`world`/`relation`/`clock`/`tension_board`）+ **团本客制视图**（toolgen `defineView`——作者在团本 `tools:` 段声明只读 SQL view，经 sqlGuard 挡写，编译成 GM 读工具；与内建视图同机制、不同注册入口）。视图名+投影列是对下游 toolgen 读编译的**稳定逻辑列契约**——物理层可重构而不破坏团本声明。详情见 [内层能力库 §4](../04-子系统设计/内层能力库.md)。

### 2.4 可见性（横切，deny-by-default 白名单·D3）

`visible` 列三态（0 默认隐 / 1 已 show / 2 强制隐=暗值）横切**所有玩家可见表**：`state`/`lore`/`pool`/`log` 已有；叙事三表（front/plotline/foreshadow）**当前缺 `visible` 列 = 欠账**（D6），要补齐与 sheet/lore 同构。机制：`show` 持久揭示（attr/entity 级，暗值 cell 即使 entity-show 也不露）+ `reveal_once` 一次性快照披露（不翻底层 visible、走 `kind=reveal` event）。

> **可见性 ≠ status**：status 是叙事三表的**生命周期状态机**（§2.7），管"对象处于哪个生命阶段"；visible 管"玩家看不看得到"。两者正交——一个 `recalled` 的 foreshadow 仍可 `visible=0/1`。

### 2.5 记忆三层

`log`（全原始流水）→ `moment`（GM 标关键行，`is_moment=1`）→ `history`（周期 compact 蒸馏）。`recall` 优先 history+moment、再兜底 FTS log。**非向量 RAG**（守 §1 agent 即引擎；软伏笔的语义召回是 §5 研究难题，v1 朴素兜底）。

### 2.6 业务域四要点（承 D2，判断不变）

1. **二级轴存活**：访问画像（精确 get/set ｜ 追加+召回 ｜ 搜索+抽样）降为二级轴，解释各域 CRUD 形状差异，不作一级分类。
2. **world 可写、rule 只人写**：world 作者灌注 + GM 运行期现编；rule 人版本化热更新、**AI 只读**（让被规则卡住的 AI 不能改卡子 = 不给讨好本能松绑）。
3. **"现在怎样" vs "本来怎样"**：可变此刻事实（派系现有实力、谁还活着）属 `state`；作者写死底料（NPC 人设背景）属 `lore`。同一 NPC，当前 HP 在 state、人设背景在 lore。
4. **可见性横切**：见 §2.4。

### 2.7 叙事三表生命周期（status 状态机）

叙事三表各有 status 状态机，管生命周期阶段（GM 推进叙事用，与可见性正交）：

| 表 | status 取值 | 转移 |
|---|---|---|
| `front` | `active` → `spent` | 活跃威胁 → 耗尽/解决 |
| `plotline` | `active` → `closed` | 存在未收口 → 收口（v1 合并 open=active） |
| `foreshadow` | `planted` → `recalled` / `abandoned` | 埋下 → 回收 / 放弃 |

`tension_board` 视图按 status 过滤"未结张力"（active front / open\|active plotline / planted foreshadow / armed watcher）。GM 经叙事工具推进 status（`front_open`/`plotline_advance`/`foreshadow_recall` 等）。

> **已 derive 定清**：精确语义（含 v1 plotline 合并 open=active、foreshadow recalled 仍需 GM show、abandoned 不示玩家）见 [内层能力库 §4.6.1](../04-子系统设计/内层能力库.md)。以上取值现状来自代码默认值（`db.ts`）+ A′ spec §3 + `tension_board` 过滤 + `stdlib/narration.ts` 工具字面量，wiki 此前无单源定义——本节为首次 derive。

## 3. 行动层：裁决工具是"数据层的读写器"

行动层的统一概念是 **resolver**：把未定局面产出一个**叙述者无法伪造的结果**、推动剧情（结果不能由叙述者自由编造）。resolver 沿两根轴变化——**选择者**（玩家 / 随机）×**结果形状**（label 档位 / verdict 胜负 / number 数值 / content 抽中的内容）。

> **"给选项"不是独立家族，而是"玩家选 label"的 resolver；"掷骰"是"随机选"的几种 resolver。** 这把旧版"给选项 / 掷骰两家族"统一了：差异只在选择者与结果形状，不是两个物种。

### 3.1 掷骰的三种形态——由"操作数从哪来"区分

> 安科传统里 d10 的 "0" 常指 10。**本项目是 AI 安科，不必遵循传统**——只要"AI 声明的判定范围"和"骰子产出的结果"能对上即可。骰面语义交给规则书。

| 骰种 | 操作数来源 | 产出 | 是否碰 sheet 域 |
|------|------------|------|--------------|
| **选项骰** | GM 预设的选项表 | 命中某选项 → **叙事档位** | 否 |
| **对抗骰** | sheet 域的两个属性 | 胜 / 负 | **读** sheet |
| **状态骰** | 骰子 ± sheet 域某数值（如 `金币 + 5d3000`） | 一个**数值** | **读+写** sheet |

按**结果形状**看：选项骰=label、对抗骰=verdict、状态骰=number；还有第四种 **content**（从卡池随机抽一行）也是 resolver，但工具落在 world 域（`world_sample`）。（旧聚类：裁决骰=出叙事后果 / 数值骰=与 sheet 耦合出数值。）

**消解"骰子设计很乱"的关键洞察：掷骰不是独立子系统，它是 sheet 域的读写器。** 对抗骰/状态骰存在的理由，正是战斗团要的不是叙事档位而是血量/伤害这种**数值**。

> 这条洞察的归宿：**状态骰（number）的工具实现已下沉为 sheet 写**（`sheet_update` 带骰形态、非独立工具），见 [总体架构 §4.2](../03-架构/总体架构.md)；概念上它仍是"随机选 number"的 resolver。

### 3.2 给选项＝玩家选 label 的 resolver

结果归玩家。后果在玩家选之前锁定（"声明后果在先"）。单人转轮 / 多人投票的差异属塑形/模式层。

## 4. 塑形层：把通用 agent 变成"不讨好的好 GM"

塑形层是 Dicelore 的差异化所在（见 [01 问题域](../01-业务与定位/问题域.md)）。它的职责单一：**对抗 AI 的讨好本能，让它当好 GM**——尊重骰子、声明后果在先、该让玩家选时不替选。

> **定位陈述（Dicelore 的灵魂）**：Dicelore 把 PbtA 用**社会约定**换来的 GM 纪律，翻译成对一个**无羞耻心、有讨好偏置的 agent** 的**机械强制**。人类 GM 靠规则书文字和牌桌默契自愿遵守的纪律，对一个被调教得"乐于助人"的 LLM 不可靠——所以 **PbtA 信任 GM 自律之处，正是 Dicelore 必须用 L1/L2/L3 焊死之处**。下面的三段式教条借自 PbtA，但每一条的兑现都比原版多一层强制。

### 4.1 教条的三段式：Agenda → Principles → Moves

塑形层的 L2 教条采用 **PbtA**（Powered by the Apocalypse，英文跑团圈最成熟的 GM 行为形式化）的三段式组织——**为什么 → 怎么做 → 做什么**：

| 段 | 是什么 | Dicelore 落点 |
|---|---|---|
| **Agenda 议程** | GM 的北极星：为什么坐在这（凌驾一切的姿态） | 恒驻核心 skill 顶部 + 开局注入 |
| **Principles 原则** | 怎么行事的具体准则（原 `guideline`） | 随 skill 注入的 L2 教条本体 |
| **Moves 动作** | 具体能做什么 + 何时做（原 `dispatcher` 形状表 + 判定时机） | **横跨 L1+L2**：工具集本身封闭（L1）+ 教怎么在工具间选（L2） |

**Agenda 议程（四条，第 0 条凌驾其余）：**

0. **你是世界的诚实仲裁者，不是玩家的取悦者。**（Dicelore 特有，PbtA 没有——人类 GM 没有讨好病。这是 Dicelore 与 PbtA 的分水岭，也是上面定位陈述的祈使版。）
1. **描绘一个会自己呼吸的世界（Portray a living world）**——世界有自己的因果、数值与进程（world / sheet / watcher 驱动），独立于玩家期望而存在，不是为取悦玩家布置的背景板。
2. **让玩家的选择带来真实的后果（Make choices matter）**——把玩家当真：后果声明在先、骰子说了算；冒险感来自"选择有重量"，不是来自你安排的精彩桥段。
3. **玩出来看会发生什么（Play to find out）**——你不预先知道结局，也不该朝某个"满意的结局"叙事；结局由骰子、watcher 和玩家的选择共同决定。

> **Agenda 的价值 = 给 F 轴一个"为什么"的根。** F2 软着陆同时违背第 2 条（后果不真）和第 3 条（你在朝预定结局叙事）；F1 跳骰违背第 3 条。这把 F 轴从"别这么做的禁令"升级为"违背了你来这里的根本目的"——LLM 更吃这种约束。
>
> **Moves 是 GM 唯一被允许做的动作**（PbtA 铁律：GM Moves 是 GM 仅有的合法动作；一旦把它做成"GM 平时干的泛泛之事"，系统就糊掉了）——Dicelore 把这条从"规则书文字"升级为 **L1 工具面的机械封闭**（工具集之外无路可走），Moves（dispatcher）只负责教"该用其中哪个"。Principles / Moves 的详细本体、以及 **fail-forward 的双边护栏**（坏结果既不洗白也不退化成死胡同）见 [Skills 包](../04-子系统设计/Skills包.md)。

### 4.2 三种强度杠杆：L1 / L2 / L3（与三段式正交）

> **关键认知：L1/L2/L3 不是第四层，而是和"三层"正交的另一个轴。** 三层（数据/行动/塑形）= 系统**由哪些部件构成**；L1/L2/L3 = 塑形这件事**靠什么强度达成**。

三种杠杆，可靠性递减、适用对象不同：

- **L1 工具强制**：借行动层那些**必经工具**，让正确行为在结构上不可绕过——必掷骰、声明后果在先、给选项/掷骰用不同工具名分流。最硬，但只能管"能被结构卡住"的部分。
- **L2 skill / Principles 教**：塑形层本体（Agenda / Principles / Moves 三段式，见 §4.1）。教 AI 何时用哪种裁决、别软着陆、怎么判断"该选还是该骰"。可被忽视，但是**判断类问题的唯一载体**。
- **L3 Hook 审计**：事后比对、抓违规（掷骰绕过率、声明后果与叙事是否一致）。兜底，不阻止当下。

**三失败 × 三杠杆的主次矩阵**（★ = 主力）：

| 失败模式 | L1 工具强制 | L2 skill 教 | L3 Hook 审计 |
|---|---|---|---|
| **F1 跳骰** | ★ 必掷 | 流程 skill | 绕过率审计 |
| **F2 软着陆** | ★ 声明后果在先 | 禁令措辞 | 后果-叙事比对 |
| **F3 选错方式** | 工具名分流 | ★ 决策框架（靠教） | 错配抽查 |

→ **可靠性 L1 > L2 > L3，但越靠"判断"的失败（F3）越只能靠教（L2）。** 这正是项目重心落在"用提示词教 AI 当 GM"而非堆代码的原因：能结构化卡住的（F1/F2 的硬约束）尽量下沉到 L1，卡不住的判断（F3）交给 L2。

> 本节只定概念。skill / Principles 的具体注入机制、Hook 的实现、工具响应"补刀"的形态，留给 03/04（见 §7）。

## 5. 两个研究级难题：明确隔离，v1 用朴素版

绝不能在主线里顺手解决，否则项目永远卡在它们上。各自单开 spike：

| 难题 | 为什么难 | **v1 朴素版（主线）** | 真·解法（spike） |
|------|----------|------------------------|------------------|
| **event 域的语义召回** | 本质是 RAG | 事件写 markdown，**FTS5 搜索 + agent 自己读** | 向量 embedding / 语义检索 |
| **让 AI"读懂"世界设定** | 设定要进 AI 判断，不能只躺库里 | (a) 核心调性安装时**焊进上下文**；(b) 相关时 `world_search`；(c) agent 直接读 markdown | 设定结构化推理 / 自动一致性检查 |

> **与 [01 问题域 §5](../01-业务与定位/问题域.md) 对齐**："长程叙事一致性"问题里，**目标 / 未结线索 / 世界硬状态属结构化**，归 sheet / event 域显式追踪、现在就做，**不在本表 spike 范围**；只有**软伏笔的"相关时自动想起"**才是上表“event 域语义召回”这个 RAG 难题，v1 朴素兜底、观测后再定。

## 6. 三层如何咬合（数据流一瞥）

```
玩家行动
  → 塑形层判断：该给选项？该掷骰？哪种骰？
      ├ 给选项  → resolve_choice          （结果归玩家；选项暂存到回合末才呈现）
      └ 掷骰
          ├ 选项骰 → resolve_outcome       （出叙事档位）
          ├ 对抗骰 → 读 sheet 两属性 → 裁决  （出胜负）
          └ 状态骰 → 读 sheet 数值 ± 骰 → 写回 sheet（出数值）
  → narrate 散文（轮内可多次穿插，非终结步骤；受塑形层"别软着陆"约束 + 工具响应补刀）
  → 需要世界/卡池内容时 → world 域（search / sample）
  → 推进后剧情/伏笔 → event 域（写）；回顾时 → event 域（召回）
```

> **一轮 = agent 一个自然回合**（玩家输入 → 回合自然结束），AI 在其中**像作者写一段**：任何工具都可重复任意次、任意顺序穿插。`narrate` 是**散文 stream**、轮内可多次，**不是**终结步骤；`resolve_choice` 的选项**暂存到回合自然结束才物化呈现给玩家**（轮内可反复改写、末次为准）——这俩是仅有的两个带特殊时序语义的调用，其余即时生效。完整的"一轮范式 + 输出层三流（散文 / 机械回显 / 结构化返回）"是物理层话题，见 [总体架构 §6](../03-架构/总体架构.md)。

## 决策与权衡

本页的领域骨架——三层、四业务域、可见性、PbtA 三段式、运行时改名——背后是五个架构级决策。下面精炼记录「定了什么 + 否了什么备选」；历史背景从略,本节即唯一权威决策记录。

<details>
<summary><b>D1 · 行为塑形三层模型（数据 / 行动 / 塑形），一次只设计一层</b></summary>

- **决策**：设计一度「零碎混乱」的根因是三个层被**同时**设计。明确分三层（数据 / 行动 / 塑形）、**一次只设计一层**，下游引用上游、不反向。本页 §0 的三层骨架 + 全 wiki 的组织主轴皆由此而来。
- **后果**：成为本页与整个 wiki 的组织主轴，也是「设计域」单向推导链（01→05）在概念层的承接。

</details>

<details>
<summary><b>D2 · 数据层一级轴 = 四业务域（sheet / event / world / rule），访问画像降为二级轴</b></summary>

- **决策**：数据层一级轴改为**四业务域**；旧「按访问模式分三存储（状态 / 事件 / 剧本）」的访问画像降为**二级轴**（每域怎么 CRUD）。根因：① **world（作者世界底料）与 rule（绑住 GM 的规则合同）** 业务上完全不同，旧版却糊成一张「剧本表」；② 「剧本表只读」假设过强——语料显示 GM 会运行期现编世界内容、规则也需热更新。**world 可写**（作者灌注 + GM 现编）；**rule 只由人类写**（作者版本化 + 团主 / 管理员实时），**AI 对 rule 只读**——让被规则卡住的 AI 能改规则 = 给讨好本能松绑，故堵死。
- **被否**：维持「三存储 / 访问模式」作一级轴（掩盖 world 与 rule 的业务差异）。「agent 即引擎、v1 不上 ML」不变。
- **落点**：本页 §2 四域表 + 要点 1–4。

</details>

<details>
<summary><b>D3 · 可见性模型：deny-by-default 白名单 + show 持久揭示 + reveal_once 快照披露</b></summary>

- **决策**：可见性是横切 sheet / world / event 三域的领域事实，采**默认拒绝的白名单**（GM 全见、显式授权玩家所见）。存储 = 各域加一列「对玩家是否可见」（用列、不用键前缀，免撞 `前缀:键` 约定）；sheet 三态（默认隐 / 已 show / **强制隐=暗值**）。默认：world / sheet 全隐，event 按记录方式（散文与机械事实默认可见、GM 注记默认隐）。**show**（持久揭示、渲染实时值、attr 级或 entity 级递归，暗值 cell 即使 entity-show 也不露）/ **reveal_once**（一次性把隐藏目标的冻结副本披露一次、不翻底层 visible，用 event 域实现「副本」）。
- **被否**：① 默认可见（泄漏暗值）；② 纯 `前缀:键` 约定存可见性（撞既有约定）；③ entity-show 无强制隐藏标记（泄漏暗值）。
- **落点**：本页 §2 要点 5；存储列与机制详情见 [总体架构 §3](../03-架构/总体架构.md) / [04](../04-子系统设计/)。（工具早期名 `shot` 已统一更名 `reveal_once`、语义不变。）

</details>

<details>
<summary><b>D4 · 全盘对齐 PbtA：升塑形层为 Agenda → Principles → Moves 三段式 + 保留独有抽象</b></summary>

- **决策**：Dicelore 独立重建出的 GM 塑形架构，本质就是 PbtA 最硬核的分支——差别只在 **PbtA 靠社会约定让人类 GM 自律，而 Dicelore 面对的 GM 是有讨好本能、无社交羞耻心的 LLM，凡 PbtA 信任 GM 自律之处都须机械强制**。据此：① 术语对齐有强对应物者（`guideline → Principles`、`dispatcher 形状表 + 两道闸 → Moves`、三档结果），**工具名不改、仅文档对齐**；② **保留独有抽象**（resolver 二轴 / 四业务域 / 三层 L1·L2·L3 / F1·F2·F3 / watcher）不硬套 PbtA 壳；③ 塑形层 L2 采三段式 Agenda（为什么）→ Principles（怎么）→ Moves（做什么），**Agenda 第 0 条「你是世界的诚实仲裁者，不是玩家的取悦者」为 Dicelore 特有、凌驾其余**（与 PbtA 的分水岭）；④ F2 升为**双边护栏**（坏结果既不洗白成好结果、也不退化成「什么都没发生」＝ fail-forward）；⑤ 新增 **Front/Clock 团本内容类型**（建在 watcher + sheet 钟之上、非新底层）。
- **被否**：① 只锚注不改名（框架不吸收新结构，放弃 Agenda / Front 收益）；② 最大化套壳（连 resolver 二轴 / 四域 / F 轴也套 PbtA 词——用为人类设计的词去装针对 AI 的独有机制，损失精度）；③ Front 留未来（放弃作者备团的核心单元，与正典背离）。
- **落点**：本页 §4.1（三段式 + Agenda 四条）；Principles / Moves 本体、fail-forward 手法表见 [Skills 包](../04-子系统设计/Skills包.md)。

</details>

<details>
<summary><b>D5 · 运行时数据层 phase-1 改名落地（sheet→state / event→log / world_doc→lore），§2 完整重构已由 D6 完成</b></summary>

- **决策**：运行时载体改名——`sheet`→`state`（加 `kind` / `rel_*` / `clock_*` 列）、`event`→`log`（加 `is_moment`）、`world_doc`→`lore`；FTS 表同步改名。**部分取代 D2 的命名**（「一级轴 = 业务域」判断不变，仅改名 + state 加 kind 轴）。**MCP 工具名与 schema 变量名保留不变**（`sheet_update` / `event_append` 等），只改底层 CRUD 函数名。
- **§2 重写已做**：原推迟的"四域→多表模型完整重构"（state 的 kind/relation/clock 分化、叙事 front/plotline/foreshadow、记忆 history）已由 D6（2026-07-08）完成——见 §2 三层架构 + 叙事一等抽象 + 视图层 + 记忆层。
- **被否**：① 全部用 expand-contract（中央共享表 log 破中间态绿）；② 改名期就重写 §2（建表期返工、违单源）；③ 同期改 MCP 工具名（无谓破工具契约）。

</details>

<details>
<summary><b>D6 · A′ 三层重构 + 叙事一等抽象（物理统一 + kind 视图 + 业务工具）—— §2 完整重构落地</b></summary>

- **决策**：执行 D5 预留的 §2 完整重构。数据层从"四域直映表"重构为**三层（物理/视图/工具）+「不变/当前」正交轴**。拱心石：player/npc/world/relation/flag/clock **六概念塌成一张 `state`**（靠 `kind`+facet 分化，非六张表）；叙事三件套 front/plotline/foreshadow = **三张独立物理表**（从 event 域独立的一等抽象，带 status 状态机 + `tension_board` 聚合视图）；记忆三层 log/moment/history。视图层零存储投影，是对下游 toolgen 的稳定逻辑列契约；团本客制视图（toolgen `defineView`）同机制延伸。来源：[backlog-core 主题 A′](../05-现状与计划/backlog-core.md)（spec 草稿 2026-06-21，本轮沉本节后删）。
- **被否**：① 每概念一张物理表（state 六概念六张表，过度归一）；② 留通用 scratch sheet 不归类（sheet 误当人物卡的根因）；③ 叙事塞回 event log（无 status 状态机、无聚合视图，AI 拿不到"以概念为单位"的读）。
- **落点**：本页 §2 重写（三层+正交轴+叙事+视图+记忆+可见性双轨）；schema 详情见 [内层能力库 §4](../04-子系统设计/内层能力库.md)；工具面见 [MCP工具面](../04-子系统设计/MCP工具面.md)。
- **欠账**（开发到一半转去梳理项目包结构；spec 全对、以下待补）：`front_omen` 子表（凶兆阶梯物化）、叙事三表（front/plotline/foreshadow）缺 `visible` 列（§2.4，status 是生命周期、与 visible 正交）、`plotline`/`foreshadow` 的 `anchors_json`+`surface_trigger` 列、state 类型化读写工具（player_card/npc_list 等，落了才删裸 `sheet_*`）、relation 工具、记忆工具（mark_moment/history_compact/recall）、presentation 接视图层（RT-FE4 的根因）。见 [backlog-core 主题 A′](../05-现状与计划/backlog-core.md) + 路线图。

</details>
