## 1. 一句话

一个后端、**两条入口路径**——`lore`（构建团本）与 `dice`（跑团）——经**同一套 DB 存储栈 + 同一组运行单元/Agent 抽象**承接，差异只在挂哪套 MCP/skill/权限。接缝是**团本包**：lore 产出它，dice 经 **import 信任闸门**消费它，"创建完直接玩"即走这条缝。

## 2. 结构落点：四根 + storage-port（backend 存储/服务 ‖ harness 两条 agent 线）

双路径不再落在单 `orchestrator` 的三分布局里，而是横跨**四根**（结构权威见本页「决策与权衡」节 / [总体架构](../03-架构/总体架构.md)）。后端这一侧涉及其中三根：

```
backend/src/         存储 + 服务宿主
├── store/           四业务域 + FTS（DB 权威存储）
├── resolve/ present/ catalog/ build/ expr/ toolgen/ eval/ session/
│                    裁决·呈现·团本库·构建纯逻辑·取数文法·工具编译·评测纯件·会话KV
├── sessionBackend.ts  storage-port 实现（openSessionBackend(db)）
├── api/             边：薄路由(dice /sessions·lore /catalog·ws 升级)
├── server.ts        组合根（按会话注入 SessionBackend 给 harness）
└── cli.ts

harness/src/         agent 运行时
├── runtime/         宿主级共享骨架(Session/SessionRegistry 接口 + streamDriverTurn + WsHub + agent/reasoning)
├── dicegm/          dice 跑团线(mcp 工具面/adapter/skills/DiceGm/DiceSession + turnLoop/rollGate/notify/recovery/presentation/sessions)
├── loregm/          lore 构建线(LoreSession + skills + 上传/抽取)
└── eval-dicegm/ eval-loregm/   评测 harness 脚本

packages/interface/  storage-port 抽象（SessionBackend 端口类型）
```

- **`Session`（运行单元）= 一个被路径类型参数化的接口**（kind: dice|lore），非两个独立抽象，住 `harness/runtime`。隔离边界=权限边界=工具面边界，三者合一。`DiceSession`/`LoreSession` 是两实现（分居 `harness/dicegm`、`harness/loregm`）。
- **Agent 按角色命名**：`DiceGm`（跑团 GM）/ `LoreBuilder`（构建，现复用 DiceGm 包构建 MCP）。`streamDriverTurn`（`harness/runtime`）是两路径共享的瘦流式 helper。
- **`SessionRegistry<S>` 泛型、path-agnostic**：构造 path-specific 会话由调用方经 `create` 回调做（runtime 不依赖路径实现）。多租户/跨机的接入点；现 `InMemorySessionRegistry` 单机实现。
- **物理隔离不变量**：dice 专属码（`harness/dicegm`）与 lore 专属码（`harness/loregm`）物理分目录，各路径装配只 import 自己那套——结构上杜绝「装错另一路径的工具/skill」。

### 2.1 storage-port：agent 经端口访存储，不直连 backend store

两路径的 agent 工具面（`harness` 的 mcp）**不直连 backend 的 store**，而经注入的 **`SessionBackend` 端口**访存储：

- `packages/interface` 中立定义 `SessionBackend`（按领域分组聚合：`Store` & `Resolver` & `Snapshots` & `Catalog` & `Presentation` & `Meta` & `Toolgen`）；端口接口按领域概念命名，**不带 `Port`/`I` 后缀**。
- `backend` 实现端口：`openSessionBackend(db)` 用闭包绑定 db、组装实现对象，**方法不暴露原始 db**；harness 调用方不传 db。
- 组合根 `backend/server.ts` 开局解析会话 db → `importPack` 物化 → `openSessionBackend(db)` → **按会话注入** `harness` 的会话工厂建 `DiceSession`/`LoreSession`。`createMcpServer` / `DiceSession` 收 `SessionBackend` 而非 `db + 直接 import`。
- **依赖倒置断模块级环**：原 `backend/api → harness/DiceSession → harness/mcp → backend/store` 的包级环，被反转为 `harness → interface ← backend` 后只剩组合根这一条方向边。这是「后端将来可换存储实现 / 加 cache / 远程化、harness 全程无感」的地基，也与 anti-F1 铁律（mcp 只声明存储意图、backend 执行）同构。
- **包级 harness↔backend 互指接受**：剩余 4 处 import 全是组合根/入口（mcp stdio 入口 + 3 个 CC hook 脚本，hooks 是 CC spawn 的独立进程、结构上无法注入），cosmetic、gate 全绿。端口表面、命名约定、迁移分阶段、包级环裁决等完整设计见本页「决策与权衡」节（本页不重抄）。
- `packages/core` 已溶解：引擎/存储 → `backend`，agent 工具面/适配/技能 → `harness`，骰子数学等纯库 → `packages/*`。两路径的 agent 仍经 [in-process MCP 工厂](MCP工具面.md) 调存储（in-process 挂载，见设计页决策节），只是该工厂的 handler 现经端口而非直连。

## 3. 存储：DB-only + 两级 + 文件只在边界

- **不变量**：内部全是 DB；**文件只在两个边界**（上传 ingress / 导出 egress），从不落地为内部存储。
- **两级**：
  - **Catalog 团本包库**（`backend/src/catalog/`，集中录 DB，与运行库同栈）：团本**定义**（`source=author`）。
  - **per-session 运行库**（一局一 SQLite）：运行态（`source=ai` + import 物化）。
- **团本身份 = UUIDv5(name)**（确定性、同名同 id、防重名）。
- **版本：线性历史、无分支**。两个动作：`commit`（保存点）/ `tag`（真发布，dice 对外只认 tag）。每 commit = 路径→内容的**全量文件树快照**；commitId = SHA-1(parent+message+files) 确定性。

## 4. 接缝：import 信任闸门（`backend/src/catalog/import.ts`）

dice 开局 = `importPack(catalog, runDB, adventureId, ref)`：checkout 某版本 → **`validatePack` 重验**（拒空包/未知路径段/state 缺列）→ 物化进运行库。**永不信任来源**（包可能手改/旧版/git import 而来），跑起来前必重验——整套安全的承重墙。

- 包格式（与构建层、数据层 import 映射对齐）：`manifest.md` / `prologue.md`（开场 prompt，**必备**，见设计页决策节）/ `lore/*.md`→lore / `rules/*.md`→rule / `pools/*.csv`→pool / `state/*.csv`(带 kind)→state / **`fronts/*.md`→front（md 正典：frontmatter Clock + 凶兆阶梯表，见设计页决策节否决 CSV 扁平）/ `plotlines/*.csv`→plotline / `foreshadows/*.csv`→foreshadow / `anchors/*.csv`→anchor**（plotline/foreshadow/anchor 为扁平叙事域，CSV 行=一个对象）。
- 覆盖域：lore/rule/pool/state + **叙事域 front/plotline/foreshadow/anchor**（叙事层落 main 后已接入，2026-06-23）。`history` 是运行时记忆（compact 生成）、非作者声明，不在 import/build。

## 5. 构建层（`backend/src/build/` + `LoreSession`）

- **`Draft`**：lore_agent 按域累积内容（manifest/lore/rule/pool/state），`toPackFiles()` 序列化成 import 认的包，`commitDraft` 提交为团本版本。
- **构建 MCP**（`createBuildMcpServer`，`dicelore_build_*`：set_manifest/write_lore/write_rule/add_pool/set_state/commit/tag + 叙事域 add_*）over `Draft`+`Catalog`，挂在 `LoreSession`（`harness/loregm`）。与运行时 `TOOLS` 编译期不交叉。**源摄入改会话工作区（2026-07-01）**：源材料经 REST `POST /lore-sessions/:id/materials` 流式上传落 `<sessionsDir>/lore/sessions/<id>/workspace/materials/`，构建 agent 补 `Bash/Grep/Glob/Write/Edit`（cwd=workspace）自行清洗分块 + 文件导航提炼——**退役 BM25 检索**（删 `backend/src/build/retrieval/`、去 `ingest`/`search` 工具）。详见 [团本构建工具链](团本构建工具链.md)。
- **构建 skill**（`dicelore-build-pack` 构建工作流 + `dicelore-build-core` 开场白/身份教条，`harness/loregm/skills`）：经 lore plugin 按引用加载（见 §8 skill 可达）；`dicelore-build-pack` 打法已从「阶段0 ingest → 各阶段 search」重写为 **agentic 文件构建**（`Bash`/`Grep`/`Read` 摸源清洗分块 + 提炼）。
- 链路：**作者经构建工具造 Draft → commit 到 Catalog → dice 开局 import → 运行库 → 跑团**（engine + API + WS 真服务器验证通过）。

## 6. 分发：git 单向投影（`backend/src/catalog/git.ts`）

DB 权威，git 只是**进出口投影**（非双向镜像）。`exportGit` 把线性 commit 史 materialize 成**真 git 仓库**（blob/tree/commit loose objects + refs/tags，`git log`/`git tag` 可读，自实现零依赖）；`importGit` 读回 → 新 DB 线性史。线性↔线性无需 merge。

## 8. Agent 适配缝 + Play 会话生命周期

**Agent 适配缝（`harness/runtime/agent.ts`，spec 2026-06-23 AD-1/3）**：driver 不再 `(host)=>Agent`（耦合 DiceSession），改 host-无关的 `AgentFactory = (init: AgentInit) => Agent`。`AgentInit = { mcpServer, openingPrompt, plugin?: PluginRef, workspace?: string, model? }`（`skills: SkillRef[]` 已换成 `plugin?: PluginRef`——见「skill 可达」；`workspace?` 供 lore 构建工作区，见 §5 / [团本构建工具链](团本构建工具链.md)）——起一个会话 agent 所需全部。会话单元（DiceSession/LoreSession）各自 self-build AgentInit 传 factory；server 注入 factory 选适配器（真=`DiceGm`，fake=`FakeDiceGm`）。**任意 agent/harness 实现 `AgentFactory` 即接入**，CC SDK（`DiceGm`）是首个适配器（v1 骑定 CC → 升为「CC 是首个适配器」，见本页「决策与权衡」节）。

**开场 prompt（`harness/dicegm/openingPrompt.ts`，AD-2；2026-07-01 退役内联教条）**：`buildOpeningPrompt(backend) = buildSessionContext(signpost) + 团本 prologue`——**不再拼 gm-core 教条**（经注入的 `SessionBackend` 取数，不直连 store）。signpost = 引擎给的 GM 身份/Agenda/纪律；prologue = 每团本根 `prologue.md`（构建 agent 产，import 回传、**不物化进 store**，只 `session_meta`）。**GM 教条改由 plugin 加载的 skill 单路径投递**（dice=`dicelore-gm-core`、lore=`dicelore-build-core`）——原「内联进 openingPrompt 作保证投递」的双层兜底**已退役**（见「skill 可达」+ 术语表 `openingPrompt` 词条）。

**skill 可达（`harness/{dicegm,loregm}/openingPrompt.ts` `ensureDicePlugin`/`ensureLorePlugin` + `harness/runtime/skillPlugin.ts`，2026-07-01 改按引用加载）**：skill 打成 **local plugin**（各角色线根补 `.claude-plugin/plugin.json`），boot 期经 `ensureSkillPlugin` **幂等 + 版本感知**把只读母本（`harness/src/{dice,lore}gm/skills/**` + `.claude-plugin/**`）物化到**数据根** `$/{dice,lore}/skills`（`$ = DICELORE_DATA_DIR ?? **Play 两层生命周期（AD-4~7）**：open（`POST /sessions/:id/open`）import 团本→运行库 + 写 `session_meta`（adventure_id/ref/adventure_name/prologue/started=0）；kickoff（`POST /sessions/:id/start`）以 prologue 为首轮 impetus 跑**开场回合**（无玩家输入）→ 置 started=1，**幂等**；`GET /sessions` 列表带 adventure_name/started（前端会话栏前缀团本名 + 两层 UI：未开场=大金「开始游戏」钮、已开场=输入框）；`DELETE /sessions/:id` 注销内存 + 删 `.db`。

**两路径的流式模型（v1）**：**dice 走 WS 流式**——`/sessions/:id/ws` 升级，回合内 turn_started/narration_commit/turn_ended 实时推前端，玩家逐句看 GM 叙述。**lore 走 REST 黑盒**——`backend/api/lore.ts` 仅 REST 端点、无 WS 升级路由；`LoreSession.handleMessage` 跑一整个构建 driver turn 到 turn_end/error 即收尾、返回 `turnId`，**不流式广播**；前端**走轮询/阻塞等待**构建完成，构建过程对作者是黑盒。lore WS 流式（构建台实时看构建 agent 在干什么）**推 v2**（决策见 [06 路线图第二批 RT-5](../05-现状与计划/backlog-后端.md)）。故 `LoreSession` v1 **不持有 WsHub、无 attachWs/detachWs**——v2 补 WS 端点时再重建。

## 决策与权衡

> 本节精炼重述本页两条决策的关键选择与被否项。storage-port 的全套四根结构 / 端口表面 / 命名约定 / 迁移分阶段是跨多页的结构权威，本页只讲与后端双路径相关的面。

### D1 后端双路径架构（lore 构建 / dice 跑团，一后端共享底层）

- **背景**：后端原是"只跑团的单路径"（全 dice 味）。但声明式工具生成层确立 lore（构建期）/ dice（运行期）两个 agent、两个时刻；前端有 lore/dice 两条路径，"创建完团本要能直接玩"逼着两路径同处一个后端、共享底层。
- **决策（业务设计要点）**：
  - **① 一个后端、两条入口路径**，共享底层（存储栈 / 运行单元 / Agent 抽象 / MCP 工厂）；接口留"跨机 / 多租户"缝，实现先单机。
  - **② `Session`（运行单元）＝ 一个被路径类型参数化的接口**，隔离 ＝ 权限 ＝ 工具面三边合一；`DiceSession`/`LoreSession` 两实现；Agent 按角色命名（`DiceGm`/`LoreBuilder`）；`SessionRegistry<S>` 泛型 path-agnostic。
  - **③ 内部 DB-only，文件只在两边界**（上传 / 导出）。
  - **④ 两级存储**：Catalog 集中录（团本定义，UUIDv5(name) 身份）+ per-session 运行库；版本线性无分支 + commit/tag 两动作（tag ＝ 真发布）。
  - **⑤ 接缝 ＝ 团本包，中间隔 import 信任闸门**——dice 开局 checkout → `validatePack` 重验 → 物化运行库，永不信任来源。
  - **⑥ 构建期声明 → 框架编译**：`Draft` → 构建 MCP（`dicelore_build_*`）→ commit Catalog；与运行时工具编译期不交叉。
  - **⑦ 分发 ＝ git 单向投影**（DB 权威，自实现 dep-free git objects，真 git 可读；非双向镜像）。
  - **⑧ 多模型**：`stripReasoning` 剥 think 段 + 多 provider 走 `ANTHROPIC_BASE_URL` 兼容代理。
  - **⑨ 物理隔离不变量**：路径专属码分目录，结构上杜绝装错。
- **被否**：① 两路径强行共用同一 Sandbox 实现而非参数化接口；② 团本构建走系统文件（应走 DB）；③ git 双向镜像（挖深 merge 坑，与线性无分支矛盾）；④ 引 isomorphic-git（避 package-lock 纠缠，自实现够用）。

### D2 storage-port 在后端的落点：依赖倒置 + 包级 harness↔backend 互指接受

- **背景**：旧布局把存储 / 裁决 / 呈现 / 构建全堆在一个 grab-bag 包里。拆成职责分明的根时撞到包级循环依赖：`backend/api → harness/DiceSession → harness/mcp → backend/store` 成环——orchestrator 既驱动会话、会话的 mcp 工具面又直读 store。单纯挪文件解不掉，必须断一条边。用户决策：不靠挪位避环，而是建 storage-port——因为目标本就是"agent 不直连 store、经接口声明存储意图、后端将来可加 cache / 远程 / 多租户"，与 anti-F1 铁律（AI 只给引用、引擎执行）同构，推进一层：mcp 只声明存储意图、backend 执行。
- **决策（后端涉及的面）**：
  - **依赖倒置建 storage-port**：中立包 `packages/interface` 定义 backend 面向 harness 的端口接口（`Store` & `Resolver` & `Snapshots` & `Catalog` & `Presentation` & `Meta` & `Toolgen` 聚合为会话级 `SessionBackend`）；**harness 只依赖端口**，**backend 实现端口**（`openSessionBackend(db)` 用闭包绑定 db、方法不暴露原始 db），**组合根 `backend/server.ts` 按会话注入**。断环原理：`harness→backend` 这条边被反转为 `harness→interface←backend`，只剩组合根这一条方向边，模块级无环。`openSession` 路径策略不进端口（宿主决策，归 `harness/runtime/paths`）。
  - **命名约定**：端口接口不带 `Port`/`I` 后缀（C#/Java 遗留装饰）；"端口"只作架构叙述词、不进类型名。
  - **后端内部按惯例分层**：`store/` 按业务域分组（sheet/event/world/narrative/interaction，db/views/fts + snapshot/usage 留顶层基础设施），api/expr/resolve/present 已对齐惯例不动。
- **包级 harness↔backend 互指——裁决：接受**：剩余 4 处 import 全是组合根 / 入口（mcp stdio 入口 + 3 个 CC hook 脚本）——composition root 在入口把抽象接到具体实现是其本分、不算耦合违例，且 hooks 是 CC spawn 的独立进程脚本（结构上无"收注入"接缝）。包级互指是 cosmetic（npm workspaces 软链双向、TS 无 project references、运行时无顶层求值环——gate 全绿为证），断它唯一办法是把 4 入口整体搬去 backend 侧（碰运行期 hook 安装资产），代价 > 收益故不断。
- **被否**：① 挪位避环（把 api 放进 harness）——能断环且零端口，但拿不到"后端可换存储 / 加 cache / 远程化"目标收益；② store 留 `packages/`——但 mcp 仍直连 store、要拆的耦合还在，仅作"先不做端口"退路；③ 断包级 harness↔backend 环——唯一断法搬 4 入口去 backend，为 cosmetic 收益冒运行期 hook 安装风险。

- **背景**：原 `DiceGm.runTurn` 每回合 `stageSkills`（`cpSync` 整拷 skill 到 `tmpdir/.claude/skills` + 设 `cwd=staged` 让 `settingSources:["project"]` 扫到 + `finally` `cleanupSkills`）；lore 侧对称。三病：① 每回合复制（随并发/团本数恶化）；② cwd 被 skill 发现绑架（挡 [团本构建工具链](团本构建工具链.md) 的 cwd=workspace）；③ `allowedTools:['Skill']` 已被 SDK `0.3.185` 废弃（sdk.d.ts:1285）。查证 SDK 原生 `plugins:[{type:'local',path}]`（:1683）+ `skills:'all'`（:1845）可按引用加载、与 cwd 解耦、零拷贝。

### D4 会话对话记录 + rewind 抽进 `harness/runtime`（transcript 铸 UUID 权威 + IoC 回退注册器）（2026-07-02）

- **背景**：dice 侧 `DiceGm` 早有 `<sessionId>_session.jsonl` 对话记录（CC transcript 风格：`turn`=玩家/作者输入、`msg`=SDK 每条 tool_use/tool_result/result、`turn_end`/`error`），但焊死在 `DiceGm` 里；**lore 侧不落**（`LoreSession` 建 driver 没穿 sessionId/sessionsDir → 适配器退化全局 logger）。而"落对话记录"与"退回"本质是**会话运行时关切、两侧一致、只差工作目录**（`dice/` vs `lore/`）。build 侧 eval 需要一份可监控、可回退的 loregm 对话记录作原料。
- **决策**：
  - **① transcript 抽进 `harness/src/runtime/transcript.ts`、两侧共用**：`SessionTranscript` 持 append-only **UUID 父子链树** + `<sessionDir>/HEAD` 指针；append 铸 `crypto.randomUUID()`、`parentUuid=当前 HEAD`、写行后 HEAD 前进并落 HEAD 文件。**transcript 是全局唯一 UUID 生成源**（任何要锚到某轮的组件都用它铸的 uuid）。会话路径单源 `sessionDir(dataDir,kind,id)`（backend-free 纯函数，backend 组合根 import）。
  - **② 真 `.jsonl` 回退（不是只广播 hook）**：`moveHead(uuid)` 把 HEAD 挪回树内任一节点 → 下次 append 从此**分叉**（回退+重生成）；`livePath()` 从 HEAD 沿 parentUuid 到根给"当前活动分支"，废弃分支留文件里作历史。jsonl append-only、永不删。
  - **③ rewind 做成 IoC 注册器**（仿 `store/snapshot.ts` 的 `SnapshotParticipant`）：`harness/src/runtime/rewind.ts` 的 `Rewind`+`RollbackHook`，`rewindTo(uuid)`=校验 ∈树 → 逐已注册 hook `rollbackTo`（还原领域态，任一抛错则**不移 HEAD**、上抛，避免 transcript 与领域态错位）→ `moveHead`。runtime backend-free，各侧把自己的 rollback hook 注册进来（**兼容层**）。
  - **④ db 快照锚 transcript UUID（权威方向：transcript 先铸、db 后锚）**：dice snapshot 表加 `transcript_anchor` 列 + `restoreToAnchor(uuid)`（additive，保留现有 seq-based 撤轮）；turnEnd 时把 transcript uuid 作 anchor 传 checkpoint；组合根注册 `dice-db` RollbackHook；`POST /rewind` 加可选 `{toUuid}`（带=UUID 驱动，不带=旧撤轮向后兼容）。**snapshot 树由此继承 transcript 树形状**。
  - **⑤ loregm 由此落 jsonl**：`LoreSession.handleMessage` 穿 `sessionId`+`dataDir`+`kind:'lore'` → loregm 落 `$ROOT/sessions/lore/<id>/<id>_session.jsonl`（REST-only 不变）；lore-draft RollbackHook v1 no-op 占位（transcript 层回退对 lore 真生效=moveHead 领域无关，仅 Draft 领域态还原留 [RT-8](../05-现状与计划/backlog-后端.md)）。
- **欠账**：dice transcript `turn_end` 行不落（`DiceGm.turnEnd()` 在 `yield turn_end` 后被 streamTurn 的 `.return()` 跳过、不可达），db 锚改用 `head()` 绕开、`rewindTo` 正常，但 `rewindLast` 对 dice 落空（[RT-7](../05-现状与计划/backlog-后端.md)：turn_end 落行责任上移 DiceSession）。
- **被否**：① transcript 写入留在 DiceGm（两侧各写一套、lore 不落）；② rewind 只广播 hook 不动 transcript（`.jsonl` 回退是空的——真回退必须移 HEAD）；③ db 自成一套 seq/id 身份树（与 transcript 树两套身份、无法"snapshot 继承 transcript 形状"）；④ 重写 dice 现有 seq-based 快照（改用 additive 加 anchor 列）。

### D5 per-turn token 搭 turn-end 信号走（2026-07-10·usage-stream + co-play/co-build）

- **背景**：CO 成本可观测性——用户纠偏「per-turn token 应放在跑团/构建页每一轮里、随 session 走，不单独查询」。已合的 `GET …/usage` 聚合端点保留（总览用），本决策只做「让每回合 token 搭上 turn-end 信号」的协议+后端改动。
- **决策**：
  - **协议**：`packages/shared/src/stream.ts` 的 `turn_ended` 加**可选** `usage{inputTokens,outputTokens,cacheReadTokens,cacheCreationTokens}`（可选=向后兼容，旧消息/FAKE 无 usage 时前端容缺），语义=本回合四类 token 合计（同 `TurnUsage`/`UsageTotals`）。
  - **dice**：`turnLoop` 在回合内**本地累加**每次 usage 事件（不查库，复用已有 `onUsage` 流），`turn_ended` 带上 `usage`（无事件则省略字段）。原 `onUsage → recordUsage` 落库不变（本决策只额外把同份数据搭进流）。
  - **lore**：`LoreSession.handleMessage` 的 driver 回合循环捕获 `ev.type==="usage"` 累加，返回从 `{turnId}` 扩为 `{turnId, usage?}`；`POST /sessions/loregm/{id}/messages` 原样回。**v1 不落库**（loregm REST-only in-memory，usage 仅随响应回前端内联，落库留 v2）。
  - **前端内联**（co-play/co-build，见 [玩家客户端-视觉 §9.2](玩家客户端-视觉.md)）：跑团页按回合分组每回合块尾内联成本行、构建页构建轮消息附 usage，估价共用 `frontend/src/features/cost/pricing.ts`。
- **被否**：单独 per-turn 查询面板（用户纠偏为内联随流）；lore usage v1 落库（临时 in-memory·留 v2）。
