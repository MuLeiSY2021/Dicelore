## 0. 选型总原则

三条贯穿全部选型的判据，源自 01/02：

1. **agent 即引擎**（[02 §1](../02-领域模型/核心概念.md)）：阅读 / 召回 / 推理由宿主 agent 做，框架不自建 ML。
2. **嫁接而非自建**（[01 卡位](../01-业务分析/问题域.md)）：可挂到任意 agent，不绑模型、不绑前端。
3. **v1 朴素、不过早优化**（[01 §4/§5](../01-业务分析/问题域.md)）：能用最朴素手段兜住的，绝不上重型方案；真要上的单开 spike。

> 深层逐条理由另见 [05-决策记录 ADR](../05-决策记录-ADR/)；本页给"选什么 + 权衡 + 否了什么"。

## 1. TypeScript 全栈

**选择**：Core 能力库 + MCP server + 团本生成器，统一 TypeScript。

**理由**：
- **官方首推**：mcp-builder 把 TS 列为推荐栈——SDK 一等公民、静态类型、MCPB 打包兼容、模型生成 TS 质量高。
- **强 schema**：Zod 定 MCP 工具入参 / 出参，契合"AI 只给引用、后果在先必填"这种需要严格结构的工具面。
- **未来多人不用重写**：路线图的多人远程（[场景 B](../01-业务分析/用户与场景.md)）= Streamable HTTP server，正是 TS + 无状态 JSON 的主场——现在用 TS 省掉一次重写。
- **数据层无损**：`better-sqlite3` 提供同步 API + 原生 FTS5，SQLite 方案完整保留。
- **Skills（L2）与语言无关**：markdown 教条不受影响。

**权衡**：放弃现有 Python 代码——但经重估，真正会丢的只有 trivial 的掷骰引擎（一个文件、移植成本极小）；`anko_db` schema 本就要按四业务域重写、两个 MCP server 本就要重建。**正处在"从零重推、MCP 全重建"的节点，现在切成本最低。** 维护者需学 TS，但 Python 同样在学习中、学习成本对等，**故无额外成本**。

**被否**：
- **Python 全栈**（曾是 v1 选择，本轮反转）：头号理由"复用现有代码"经重估站不住（见上）；且未来多人远程要重写。详见 [ADR-0006](../05-决策记录-ADR/)。
- **Go / Rust**：性能在本场景无关紧要，MCP 生态与门槛吃亏。

## 2. MCP + Skill：行为塑形的两个载体

**选择**：MCP 承载 **L1 工具强制**，Skill 承载 **L2 教条**。

**理由**：这俩恰好对应 [02 §4](../02-领域模型/核心概念.md) 塑形的两个可落地杠杆——
- **MCP** 是跨 agent 的工具标准：把"必掷骰、声明后果在先、工具名分流、AI 不给值"焊进工具 schema（L1），任何 MCP 客户端通用。
- **Skill** 是可移植的 markdown 教条载体：把"别软着陆、何时选何时骰"写成 skill（L2），不绑某个 agent 的内核。

**权衡**：Skill 是 Claude / Agent SDK 的概念，**并非所有 agent 原生支持**。解法——**教条内容可移植，注入机制因 agent 而异**，由 adapter 处理。详见 [跨agent与适配层](跨agent与适配层.md)。

**被否**：
- **纯散装 system prompt**：01 调研的反面教材——不可移植、规则书膨胀爆上下文（meyomeyome 实测 11 万字）。MCP+Skill 把"框架层"与"团本内容"分开，正是治这个。
- **多 agent 互审纠偏**（学术路线，wordplay 论文）：绑定具体规则书、重、非通用层，违背"嫁接任意 agent"。
- **自建 agent runtime / 自己跑模型**（AI Dungeon 等封闭产品路线）：丢掉"可嫁接"卡位。
- **把规则写成代码引擎**（状态机 / 约束验证器）：[01 §4](../01-业务分析/问题域.md) 已明确否决——治的是记账担子，治不了失控倾向这个真病。

## 3. SQLite（KV + FTS5），v1 不上 ML / 向量库

**选择**：数据层用 SQLite，每局一文件，KV 存 sheet（实体卡）、FTS5 存 world/event/rule 的全文检索。

**理由**：
- **agent 即引擎**（[02 §1](../02-领域模型/核心概念.md)）：召回交宿主 agent，框架不建 embedding / 向量流水线。
- **零外部依赖、嵌入式**：每局一个 `.db` 文件，易备份、易迁移、易随存档分发，契合单人本地 v1。
- **FTS5 够朴素召回**：world/event 的搜索用全文索引 + tag 兜底即可。

**权衡**：语义召回（软伏笔"相关时自动想起"）v1 只朴素兜底（FTS5 + tag + agent 自己读），真 RAG 暂缓单开 spike（[01 §5](../01-业务分析/问题域.md)、[02 §5](../02-领域模型/核心概念.md)）。是否够用要等 v1 真实失败数据。

**被否**：
- **向量库**（Chroma / pgvector / Pinecone）：违背 agent 即引擎、v1 过早优化、引入外部依赖。
- **Postgres / MongoDB**：外部依赖；多人远程才可能需要，v1 单人单文件够。
- **状态机引擎 / 约束验证器**：游戏引擎路线，治错病（[01 §4](../01-业务分析/问题域.md)）。
- **纯文件（只 JSON / markdown、无 DB）**：兜不住"精确 get/set + 全文检索"的记账担子。

## 4. 传输：stdio（v1）

**选择**：MCP 走 **stdio** 本地传输。

**理由**：v1 是单人、本地、单 session——stdio 零网络配置、作为 agent 子进程跑即可（mcp-builder 也把 stdio 列为本地集成首选）。

**被否 / 推迟**：**Streamable HTTP**（远程、多客户端）留给**多人论坛安价**阶段（[用户与场景 场景 B](../01-业务分析/用户与场景.md)）。v1 不引入，避免过早承担网络 / 并发复杂度。→ §1 选 TS 全栈正为这步铺路：届时切 Streamable HTTP 不必换语言。

## 5. 团本：Markdown 主体 + CSV 仅结构化数据

**选择**：团本（manifest）以 **Markdown** 为叙述主体，**CSV 仅用于表格化结构数据**（物品面板、随机表 / 卡池、范式参数）。

**理由**：
- 世界观 / 设定 / NPC 人设这类要喂进 AI 判断的内容，Markdown 最自然、AI 最好读（合 agent 即引擎）。
- 卡池 / 随机表 / 物品面板本质是**行×列的策划表**——CSV 原生贴合，且**游戏策划本就用 CSV**（Excel / 表格工作流），作者门槛最低。
- 呼应 [01 场景 C](../01-业务分析/用户与场景.md)"作者只填团特有的三件事"——薄到近声明的 manifest。

**权衡**：CSV 是扁平的，**深层嵌套配置表达不了**。但本项目结构化数据绝大多数是表格（卡池 / 随机表 / 面板），正中 CSV 强项；少数嵌套参数（如分档区间）拍平成列即可（`label, min, max`），实在零碎的小块留 Markdown frontmatter。

**被否**：YAML / JSON（策划不熟、表格数据写着比 CSV 累）；全 Markdown（卡池 / 随机表无法被工具精确解析）。

> manifest 的**具体形态、字段约定、CSV 列规范** → [04-子系统设计](../04-子系统设计/)；版本化分发的迁移语义 → 04 / 未来。

## 6. 跨 agent 可移植（点到，详见专页）

**选择**：通用核心 = MCP 标准 + Skill markdown 内容 + SQLite；agent 专属配置（Hook / 裁判 subagent / 安装路径）只是**可选优化层**。

**理由**：这是 [01 卡位](../01-业务分析/问题域.md) 的立身之本——"可嫁接任意 agent 的行为塑形层"。绑死单一 agent = 退化成封闭产品、丢掉差异化。

**被否**：绑死单一 agent 内核。

→ 通用核心 vs adapter 的边界、注入机制、降级策略 → [跨agent与适配层](跨agent与适配层.md)。

## 7. 被否方案总表（速查）

| 维度 | 选了 | 否了 | 一句话理由 |
|---|---|---|---|
| 语言 | TypeScript 全栈 | Python / Go / Rust | 官方首推 + 强 schema + 未来多人省重写；Python 现有代码优势经重估不成立 |
| 塑形载体 | MCP（L1）+ Skill（L2） | 散装 prompt / 多 agent 互审 / 自跑模型 / 代码引擎 | 可移植、分框架与内容、不造引擎 |
| 数据层 | SQLite KV+FTS5 | 向量库 / PG / Mongo / 纯文件 | agent 即引擎、零依赖、v1 不过早优化 |
| 传输 | stdio | Streamable HTTP | v1 单人本地；HTTP 留多人阶段 |
| 团本 | MD 主体 + CSV 表格数据 | YAML/JSON / 全 MD | 策划本就用 CSV、表格数据原生贴合；AI 读 MD、工具解析 CSV |
| 可移植 | 通用核心 + 可选 adapter | 绑死单一 agent | 守"嫁接任意 agent"卡位 |
