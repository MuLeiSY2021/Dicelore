> wiki 分**两个顶层域**：[指南/](指南/)（对使用者）与 [设计/](设计/)（对内·AI 维护）；外加一份全 wiki 共用的顶层 [术语表](术语表.md)。

## 三条让它"缜密"的硬规矩

1. **单向推导**：下游页只能引用上游页（架构引用业务，不反向）。要改业务才回头改架构——杜绝循环 litigate。
2. **单源**：一件事只在一处是权威。词条只在术语表；问题账只在 05；决策只内嵌对应设计页。别在多处复制，要么链接、要么沉淀。
3. **每页一职责**：每页开头一句"本页职责：回答什么问题"。一句话说不清就拆页。

## 决策怎么记（决策 B：无独立 ADR 区）

历史上 28 条 ADR 堆在一个 README 里、与设计页双写漂移。现在**取消独立 ADR 区**，决策三处落地：

1. **「为什么这么设计」内嵌对应设计页的「决策与权衡」节**——决策与被否选项就近长在它影响的设计页里，改设计页时一并改决策（根治"只改 ADR 不改 wiki"的传播缝）。
2. **薄 [决策变更日志](设计/决策变更日志.md)**——一行一条「日期 — 改了什么 — 链到哪页」，只留时间线、不含设计内容。
3. **[归档/历史决策日志](设计/归档/历史决策日志.md)**——原 28 条 ADR README 原文逐字归档（非权威、以设计页为准），作历史与安全网。

## 怎么读这个 wiki

- **使用者**（要玩 / 要造团本 / 要接接口）→ 直接进 [指南/](指南/) 挑对应受众页。
- **维护者 / AI**（要懂为什么 / 要推进 / 要改设计）→ 进 [设计/](设计/)，从 [01-业务与定位](设计/01-业务与定位/) 按编号顺读到 [05-现状与计划](设计/05-现状与计划/)；每个 session 起手先看 [05-现状与计划](设计/05-现状与计划/) 了解现状与欠账。
- **查词** → [术语表](术语表.md)。
- **查某决策的来龙去脉** → 先看对应设计页「决策与权衡」节，时间线看 [决策变更日志](设计/决策变更日志.md)，原文看 [归档](设计/归档/历史决策日志.md)。
