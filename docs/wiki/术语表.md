# 术语表

> **本页职责**：每个术语在一处精确定义，全 wiki 引用此处，杜绝同词不同义。
> **上游依赖**：[核心概念](设计/02-领域模型/核心概念.md)。
> **状态**：🟢 起步（随概念成型持续补充）。

| 术语 | 定义 |
|------|------|
| **安科 / anko** | 骰子驱动的玩法：命运交给随机。对应 `resolve_outcome` 等掷骰路径。罗马音 canonical=**`anko`**（不变）。 |
| **安价 / anka** | 玩家选择/投票驱动的玩法：命运交给人。对应 `resolve_choice`。罗马音 canonical=**`anka`**（对齐 skill 名 `dicelore-flow-anka`、README 'anka form'）；收编变体 `anki`（旧拼写，作 variant）。 |
| **GM** | Game Master，主持人。本项目里由被嫁接的 AI 扮演。 |
| **F1 跳过骰子** | 失败模式：该判定却直接写结果，无失败可能。 |
| **F2 不尊重结果** | 失败模式：骰出失败却"软着陆"、悄悄给玩家好结果。 |
| **F3 选错处理方式** | 失败模式：该给选项却替玩家骰，或反之——错配机制。 |
| **resolver** | 行动层的统一概念：把未定局面产出一个"叙述者无法伪造"的结果、推动剧情。沿两轴变化——**选择者**（玩家 / 随机）×**结果形状**（label 档位 / verdict 胜负 / number 数值 / content 抽中内容）。"给选项"= 玩家选 label；三种骰 = 随机选的不同形状。见 [核心概念 §3](设计/02-领域模型/核心概念.md)。 |
| **选项骰** | resolver：随机选 label。操作数=GM 预设选项，产出叙事档位。 |
| **对抗骰** | resolver：随机选 verdict。比两个表达式（DC = 一边为常数），产出胜负。 |
| **状态骰** | resolver：随机选 number。骰子±sheet 数值、写回 sheet。**已下沉**为 `sheet_update` 的带骰形态（非独立工具，见 [总体架构 §4.2](设计/03-架构/总体架构.md)）。 |
| **裁决骰 / 数值骰** | 随机 resolver 的两类聚类：裁决骰（选项骰，出叙事后果）/ 数值骰（对抗·状态骰，与 sheet 耦合出数值）。 |
| **sheet 钟 / 游戏时间** | 游戏内回合 / 时刻（如 `世界.回合`、`世界.时间`）是 sheet 的一个可变事实，**怎么推进由团本 rule 定**，框架不内置"回合"概念；时间型 watcher 触发即对它的条件。框架自身只拥有单调事件序号 `seq`。见 [内层 §4.2](设计/04-子系统设计/内层能力库.md)。 |
| **watcher（sheet 数据触发器）** | 监视某 sheet 属性谓词（`{张三.HP} < 30`、`{世界.天} >= 18`），**满足即就地触发**：`sheet_update` 写完引擎重算、命中则 payload 经出参回 AI ＋ 落 `kind=watcher_fired` event。**edge-triggered**（跨越沿触发一次、条件解除才 re-arm）＋ `mode` once/repeat。**泛化并取代旧 timer**（时间到期 ＝ 监视钟 attr 的特例），从 hook 解绑、属 core 能力。`watcher_set` 创建。**`Clock` / `Front` 是建在它之上的团本内容单元（见下两条）。** 见 [总体架构「决策与权衡」节](设计/03-架构/总体架构.md)、[内层 §4.2](设计/04-子系统设计/内层能力库.md)。 |
| **Clock（倒计时钟）** | 团本内容单元：一个 sheet 钟 attr（如 `世界.入侵进度`，含 min/max/mode）＋ 监视它的 watcher 的典型组合。借自 PbtA 的 countdown clock；失败 / 事件推进钟、钟满触发 Front。**非新底层机制**（＝ sheet 钟 ＋ watcher 的封装）。见 [团本与 manifest](设计/04-子系统设计/团本与manifest.md)。 |
| **Front（阵线）** | 团本内容单元：一组关联威胁的预置 prep——名字 ＋ 利害问题 ＋ 一个 Clock ＋ **阶梯式凶兆表**（钟值 → 该触发的 payload），落地为一组**预声明 watcher**。借自 PbtA 的 Fronts/Threats；是"会自己推进、给世界上发条"的压力源。见 [团本与 manifest](设计/04-子系统设计/团本与manifest.md)。 |
| **Front / Clock 大小写（概念 vs 实现层）** | **有意区分、勿压平**：**大写 `Front` / `Clock`** ＝团本内容单元**概念**（PbtA 借词，见上两条词条）；**小写 `front` / `clock`** ＝实现层标识符（`fronts/*.md` 文件段、`front` 表、`front_advance` 工具、frontmatter 的 `clock` 字段、`stateSetClock`）。登记此区分规则：正文以**概念身份**指代时用大写；指 file path / table / tool id / column 时用小写。 |
| **已落地改名登记（canonical + 收编旧名）** | 下列改名**已落代码**，登记 canonical 写法、旧名作 variants 收编（仅作历史对照，勿在交叉对照处批量压平）：<br>• 运行时存储表：**`lore`**（旧 `world_doc`）/ **`rule`**（旧 `rule_doc`）/ **`pool`**（旧 `world_pool`）——见 `db.ts` 与 `store/world/`。<br>• GM 教条段名：**`Principles`**（原 `guideline`）/ **`Moves`**（原 `dispatcher`）/ **`reveal_once`**（原 `shot`）——详见上「Agenda/Principles/Moves」与「reveal_once」词条。<br>• 领域触发器：**`watcher`**（泛化并取代旧 `timer`）——见上「watcher」词条。<br>**注意**：MCP 工具名 `world_show` / `world_search` / `world_sample` / `world_register` 的改名**明确推迟**（见 [核心概念「决策与权衡」节](设计/02-领域模型/核心概念.md)），不在此登记范围。 |
| **sheet 域（逻辑层 vs 物理表 state）** | canonical：逻辑/对外层统一 **`sheet`**（`SheetCell` / `sheetRows` / `sheetCells` / `setSheet` / `set_sheet` 工具）；物理 SQLite 表名 **`state`** 仅作 store 内部实现，经别名缝（`tableParticipant("sheet","state")`）单点收口。**有意区分、勿压平**——逻辑名与物理表名是两层。收编变体（逻辑层应改 sheet）：`StateCell` / `Draft.setState()` / `stateRows` / `stateCells` / `set_state` / existsMatch 命名空间 `"state"`。 |
| **event 域（逻辑层 vs 物理表 log）** | canonical：逻辑/对外层统一 **`event`**（`eventAppend` / `eventSince` / `eventRecall`、kind 值 `"event"`、browse source `"event"`）；物理 SQLite 表名 **`log`** 仅作 store 内部实现，经别名缝单点收口。**有意区分、勿压平**。收编变体（逻辑层应改 event）：存储函数 `logAppend` / `logSince` / `logRecall`、existsMatch 命名空间 `"log"`、browse source `"log"`、工具描述「浏览 world/rule/log」。 |
| **world 域（lore / pool 是其子类，非顶层域）** | canonical：**`world`** 是四业务域之一（顶层域）；**`lore` / `pool` 是 world 域下的子类**（文件段 / 函数名 / 表名），**不是顶层域**。两者在 import 同走 `store/world`（`loreUpsert` / `poolAdd`）。**勿把 lore 误升为顶层域**（如「lore 域」是误写，应作「world 域（lore 子类）」）。 |
| **可见性 / `visible`** | 横切 **sheet / world / event** 三域的事实：每条记录带"对玩家是否可见"标记。**默认拒绝**——world / sheet 全隐，event 按记录方式定（散文 / 机械事实默认可见、GM 注记默认隐）。输出层据此过滤渲染给玩家的内容。存储为各域一列，机制见 [总体架构 §3](设计/03-架构/总体架构.md)。 |
| **show（揭示）** | 把隐藏记录**持久**翻为可见（`visible=true`），输出层每轮渲染其**实时值**（跟着变）。`sheet_show` / `world_show`。粒度可 attr 级或 entity 级递归（长效、覆盖该 entity 未来新增 attr），但**强制隐藏标记**的 cell 即使 entity-show 也不露。 |
| **reveal_once（快照披露）** | 把某隐藏**目标**（sheet cell **或 world 条目**）**此刻内容以冻结副本披露一次**，**不**翻持久可见位；落点 = 写一条 `kind=reveal` 的「可见 event」。对 sheet（值暗变）→ 玩家见旧值、**下次 reveal_once 才刷新**；对 world（基本静态）→ 一次性披露、不入持久可见集。给"瞥一眼 / 侦查 / 占卜 / 鉴定"。中优先级披露动作。 |
| **强制隐藏标记** | sheet cell（暗值，如"好感度暗值"）上的标记：即便其 entity 被 entity 级 show 递归揭示，该 cell 仍不可见——防 entity-show 泄漏暗值。 |
| **spoiler 档 / 防剧透档** | **前端渲染层**档位（`strict` / `loose` / `off`·默认 strict），每种 GM 行为按档渲染不同详细度。**与 `visible` 正交**：visible 是数据层「是否已披露」标记（0/1）、spoiler 是呈现层显隐档。stream 全量下发含 visible=0，防剧透靠前端 spoiler + AI Principles、非后端截流（**取消**旧「visible=0 永不下发硬底线」）。存 session-meta `spoiler_tier`、经 `POST …/config` 立即生效。见 [玩家客户端-接口 D8](设计/04-子系统设计/玩家客户端-接口.md)。 |
| **明骰档位 plan / narration** | 明骰每档命中后的后果分两层：**`plan`**＝AI 真实计划（驱动机械层·可含暗值/真相·骰前锁定），**`narration`**＝玩家可见叙述（可留白悬疑·不吐暗值）。两字段随 `roll_staged` 全量下发、显隐交 spoiler 档；掷骰后据命中档 `plan` 驱动机械（一物两用）。见 [玩家客户端-接口 D9](设计/04-子系统设计/玩家客户端-接口.md)。 |
| **暗骰（hidden roll）** | GM 调 `resolve_outcome`/`resolve_contest` 带 `hidden:true`：引擎**立即掷**（不 stage、不等玩家）、结果 event `visible=0`（stream 照发）、发 WS `hidden_roll`（带完整结果·前端按 spoiler 档渲染）。与明骰（玩家闸控掷）互不物化。**不走 pendingRoll**（语义错位）。见 [玩家客户端-接口 D9](设计/04-子系统设计/玩家客户端-接口.md)。 |
| **战后复盘（debrief）** | dicegm 会话终态：GM AI 调 `game_end` 后会话转「战后复盘」态（**不直接归档**·A2 终局→战后复盘·幂等），harness 加载 `debrief-mode` skill 切 AI 行为（不推进剧情、答玩家对结局/过程/人物的提问）。A1 `status` 取值 `{活跃, 空, 战后复盘, 归档}`。退出：branch 回档续玩 / 复用 `DELETE` 显式结束归档 / 关会话保持态。loregm 无复盘态（作者域终态是 commit 提交团本）。见 [玩家客户端-接口 D5](设计/04-子系统设计/玩家客户端-接口.md)。 |
| **分支（branch） / rewind** | 一个 dicegm 会话下可有多个 **branch**（各一 jsonl 事件日志·独立 seq/快照·默认分支 `main`）；**当前分支**＝会话当前活跃 branch，所有 messages/roll/choices 作用于它。**`branch`**＝copy 当前分支到 `fromSeq` 产新分支、自动成当前分支（保留原分支）；**`rewind`**＝覆盖**当前分支**（截断到 toSeq、丢弃其后·不新增分支）。切换＝`checkout` + 重算 presentation。仅 dicegm 支持（loregm 走 Draft 增量·v1 不做）。见 [玩家客户端-接口 D5](设计/04-子系统设计/玩家客户端-接口.md)。 |
| **GM session 续接（resume）** | **一个团本＝一个 SDK session**：LLM 对话历史从开局到 game_end 连续续接（首回合存 SDK `session_id` 于 meta、后续回合 `buildInit` 注入 `resume: <session_id>`·SDK 自持久化 transcript）。canon（`world_*`/`event_recall`/`rule_search`）由此**回归辅助 MCP**（不再被迫每回合全靠它重建记忆）。resume 失败＝报错提示历史丢失、需开新局。见 [玩家客户端-接口 D6](设计/04-子系统设计/玩家客户端-接口.md)。 |
| **访问画像** | 数据层二级轴：某业务域**怎么 CRUD**（精确 get/set ｜ 追加+召回 ｜ 搜索+抽样）。即旧版「状态表 / 事件表 / 剧本表」的区分，**降级为二级轴**——解释为什么各域读写不同。wiki 正文统一用 sheet / event / world 域名指代，旧别名仅作历史对照。 |
| **声明后果在先** | 在掷骰/玩家选择之前就锁定每个结果的后果，防 F2 软着陆。 |
| **fail-forward / F2 双边护栏** | F2 的完整版：坏结果**既不能洗成好结果**（上边界＝anti-讨好，原 F2）、**也不能退化成"什么都没发生"**（下边界＝anti-死胡同，借自 PbtA fail-forward）——要咬下去并打开新局面。craft（三档结果 / 软招·硬招 / 后果手法菜单 / 末日钟 / "有时失败就是失败"）见 [Skills 包](设计/04-子系统设计/Skills包.md)。 |
| **agent 即引擎** | 阅读/召回由宿主 agent 完成，框架不自建 ML。 |
| **讨好本能** | AI 被调教成"乐于助人"而衍生的倾向：给好结果、回避冲突、不让玩家失败。是 F1/F2/F3 的总根源，也是塑形层的对抗目标。 |
| **两副担子** | [01 问题域](设计/01-业务与定位/问题域.md) 对"主持一局要扛什么"的拆分：记账担子 + 失控倾向。人与 AI 各擅长一副、各搞砸另一副（镜像）。 |
| **记账担子** | 能力问题：维护游戏状态（人物卡/资源/卡池/计时器/隐藏值/回合阶段）。由①数据层接。 |
| **失控倾向** | 行为问题：尊重骰子、声明后果在先、该让玩家选时不替选——AI 凭本能做反。由②行动层强制工具 + ③塑形层接。 |
| **塑形层** | 第③层。职责是"把通用 agent 变成不讨好的好 GM"，通过 L1/L2/L3 三杠杆达成。见 [核心概念 §4](设计/02-领域模型/核心概念.md)。 |
| **L1 / L2 / L3** | 塑形的三种强度（与"三层"正交）：L1 工具强制（结构不可绕过）、L2 skill/Principles 教（判断类唯一载体）、L3 Hook 审计（事后兜底）。可靠性 L1>L2>L3。 |
| **Agenda 议程 / Principles 原则 / Moves 动作** | 塑形层 L2 教条的三段式组织（借自 PbtA）：**Agenda**＝GM 北极星（为什么，含 Dicelore 特有第 0 条"你是诚实仲裁者、不是取悦者"）；**Principles**＝具体行事准则（**原 `guideline`**，如"别软着陆"）；**Moves**＝该做什么动作 ＋ 判定时机（**原 `dispatcher` 形状表**，横跨 L1 工具封闭 ＋ L2 选择教学）。"GM Moves 是 GM 唯一合法动作"这条 PbtA 铁律在 Dicelore 由 L1 工具面机械兑现。见 [核心概念 §4.1](设计/02-领域模型/核心概念.md)、[Skills 包](设计/04-子系统设计/Skills包.md)。 |
| **服务器位 GM** | 安科/安价里主持人的定位：解说+推进+协调玩家+演 NPC 的"游戏服务器/裁判"，而非人际魅力来源。故 AI 接管损失的"人味"比传统跑团小。见 [安科安价是什么](设计/01-业务与定位/安科安价是什么.md)。 |
| **长程叙事一致性** | 第三类问题：记不住伏笔/目标/未结线索/世界发展。结构化部分归记账担子（现在做），软伏笔的语义召回是 RAG 难题（v1 朴素兜底、暂缓）。见 [01 问题域 §5](设计/01-业务与定位/问题域.md)。 |
| **替代派 / 相棒派** | 对"AI 当 GM"的两类期待：替代派要 AI 全包（吃塑形层）、相棒派要 AI 当副驾减负（吃数据层）。分层架构两派通吃。见 [用户与场景](设计/01-业务与定位/用户与场景.md)。 |
| **团本（Adventure）/ 规则书 / manifest** | 团本作者为一局准备的**成品内容包**：世界观＋机制差异＋随机池数据（**规则书**）＋开场与剧情线（**剧本**），打包成一个自洽可玩单元。**英文定名 `Adventure`**——取 TRPG 正典（D&D 的 adventure/module、CoC 的 scenario 同类物：自带规则书＋剧本的成品冒险包）；不用 `Scenario`（已被 eval 种子占用、且通常假定规则书另在）、不用 `Campaign`（粒度偏大）。理想形态是薄到近声明的 manifest（选 skill ＋ 填数据）。形态见 [团本与 manifest](设计/04-子系统设计/团本与manifest.md) ＋ [团本构建工具链](设计/04-子系统设计/团本构建工具链.md)。<br>**唯一正确写法**（scope=both）：代码标识符＝`Adventure` / `adventureId` / `adventureName`；中文显示＝「团本」。收编变体（机械改名欠账，待统一）：`tuanben` / `tuanbenId` / `tuanben_name` / `TuanbenSummary`；以及**当团本实体讲**的裸 `pack`——`packName`、前端 `packs` state（实为 `AdventureSummary[]`）、`/packs` 路由、`openPlaySession` 的 `pack*` 命名、`campaign` / `campaign pack` / `campaign module` / `Campaign builder`。改名单源记于 [06 backlog](设计/05-现状与计划/)。**注意**：`Pack` / `PackFile` / `commitPack`（见下条「Pack / 文件打包层」）是**文件打包层**，与 Adventure 实体是两个概念，**勿合并**。 |
