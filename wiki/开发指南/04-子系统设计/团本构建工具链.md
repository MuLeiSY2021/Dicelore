## 0. 设计靶心：降低作者门槛

普通用户不会手写 manifest / CSV / frontmatter。**构建台的存在就是让"丢一整本《凡人修仙传》进去 → 得到一个可玩团本"成立**——把"格式正确性"的负担从人和 AI 移进工具，把"读几百万字"的负担移进检索库，把"看不懂文本格式"的负担移进可视化。

**构建台 = 同一个 Claude Code 换装**：加载构建 skill + 构建期 MCP（`dicelore_build_*`），**而非**运行时那套 `resolve_*` / `sheet_update`。两套工具面互不出现在对方场景。

整体形态对称于运行时（[内层能力库](内层能力库.md)）：

```
        团本包（MD + CSV 文件）  ← 唯一真相（见本页决策节）
                 ▲
        ┌────────┴────────┐
   ① 团本包读写层（纯逻辑·结构化 CRUD over 文件 + 校验）
        └────────┬────────┘
        ┌────────┴────────┐
   ② MCP 门面            ② Web 门面（本地服务 + 可交互前端）
   (agent 走它读写)       (用户在浏览器读写)
        两边都即写即读·共享同一套读写层
                 ▲
   ③ 源摄入：会话工作区 + agentic 文件构建（源上传落盘，agent 用 Bash/Grep/Read 摸源提炼）
   ④ 构建 skill dicelore-build-pack（agentic 文件打法编排 ①②③）
```

## 1. 团本包读写层（纯逻辑核心）

构建台的地基，**与门面正交、可单测**（同 [内层能力库](内层能力库.md) 的分层哲学）。把"对包的结构化操作"封成原子函数，吃进所有格式细节。

- **结构化 CRUD over 文件**：`add_npc` / `update_npc` / `add_pool_entry` / `update_pool_entry` / `write_lore` / `set_param_band` / `set_sheet_cell` / `set_manifest` …。底层负责：写对 CSV 列序、生成/更新 frontmatter、放进正确子目录（[团本与 manifest §1](团本与manifest.md) 布局）。**调用者（agent / 用户）永不手搓 CSV 语法或 frontmatter。**
- **文件为真相、即写即读**（见本页决策节）：每次操作即落文件、即可重读；无内存态、无双向同步。
- **内置校验器**（贯穿每次写 + 可整包跑）：
  - CSV 列合法（表头匹配 [§4 列规范](团本与manifest.md)、`weight` 数值、`visible` 合法值）；
  - `manifest.flows` 选的流程 skill **真实存在**（[Skills 包](../02-MCP与Skill体系/Skills包.md) 注册表）；
  - `manifest.clock` 指向的 attr 在 sheets / 约定内；
  - **引用完整性**：`entry` 锚点存在、param 表被引用的列齐全、NPC 散文与其 sheet 卡（若有）entity 名一致；
  - 路径/编码：UTF-8、文件名无非法字符。
- **校验产出结构化报告**（`[{level: error|warn, file, msg, hint}]`），供门面回显——错误早发现、定位到具体文件/行。

> 读写层 + 校验器是 TS 纯函数模块，**不依赖 Claude Code、不依赖 Web**；MCP 门面与 Web 门面都只是它的薄包装。

## 2. 双门面（MCP / Web）

同一套读写层，两个调用入口，**对称、即写即读、无实时同步**（见本页决策节）。

### 2.1 MCP 门面（agent 走它）

构建期 MCP server，工具集 `dicelore_build_*`，是读写层 CRUD + 校验 + 检索（§3）的 schema 化封装（Zod，对齐 [MCP工具面 §0 通用约定](../02-MCP与Skill体系/MCP工具面.md)）。`dicelore` CLI 在**构建模式**下注册这套 server（而非运行时那套）。agent 在构建 skill（§4）引导下调用它增量产出团本。

### 2.2 Web 门面（用户走它）

**作者侧专属的可交互界面**，本地起一个**轻量 http 服务**（Node 内置 `http`，零重框架，守 [技术选型 §6.1 低安装](../03-架构/技术选型.md)），前端读包目录渲染成人类可读视图（团本说明书：世界设定排版 / NPC 卡片 / 卡池表格 / 分档表），并允许用户**直接增删改条目**。

- 用户每次编辑 → 调读写层 → 即写文件 → 重读渲染（即写即读）。
- agent 改完文件后，用户**刷新**即见最新（v1 无 WebSocket 推送，见本页决策节）。
- **可视化即审阅**：校验报告（§1）在界面上高亮到具体条目，用户看懂、点改或口头反馈让 agent 改。

> **构建期 Web 门面 ≠ 运行时 GUI**：[adapter 页](adapter与L3审计.md) 裁定的"运行时 GUI 属未来"不变——玩游戏 v1 仍走终端。此处只为**作者构建**开一个一等公民界面，与运行时游玩界面解耦（共享文件包、不共享前端）。详见本页决策节。

## 3. 源摄入：会话工作区 + agentic 文件构建（2026-07-01 重设计，退役 BM25 检索）

承接 [总体架构 §7](../03-架构/总体架构.md)"用宿主 agent 读取抽取"。几百万字的原材料远超 agent 上下文——**原起步方案是关键词 FTS5 + jieba 切块建库、按阶段检索**（`ingest`/`search`），但对自带文件工具的 CC agent，BM25 切块割裂且无语义优势，且大源经 LLM 中继（原文当 `ingest({text})` 工具参数输出）成本随规模爆炸。**2026-07-01 改为会话工作区 + agentic 文件构建**（见本页决策节 D2）：

- **每会话持久工作区**：`<sessionsDir>/lore/sessions/<id>/workspace/`（`materials/` 装上传源、根装 agent scratch）；纯业务数据、**不含 `.claude`/skill**（skill 经 plugin 从数据根 `$/lore/skills` 按引用加载，见 [后端双路径架构 §8](后端双路径架构.md)）。每 session 建一次、跨轮存活、随 session 文件夹删（v1 无独立 GC）。
- **流式素材上传**：REST `POST /lore-sessions/:id/materials`（原始字节流、`?filename=` 带名）流式落盘 `materials/`，大小闸 `DICELORE_MATERIAL_MAX_MB`（默认 100，超限中途 413 + 清半成品，不整体入内存）；build-mcp `put_material({sessionId,filename,localPath})` 取作者本机路径流式上传——**大源不经 LLM 中继**（content 不进工具参数）。
- **agent 自行摸源提炼**：构建 agent 补 `Bash/Grep/Glob/Write/Edit`（cwd=workspace），用 `wc`/`head`/`grep`/`sed`/`split`/`python3` + `Grep`/`Read` 摸结构、清洗分块、按阶段 `Grep` 定位 + `Read` 读块 → 提炼 → `dicelore_build_*`（不再 `search`）。
- **格式不做能力提升**：后端不做格式转换（`pdftotext`/`pandoc` 一律不做）、不代偿模型读取能力；`materials/` 原样存上传原文，模型读不了的（如无视觉时的图/PDF）由作者自己预处理成可读文本后再传。
- **临时品、不进成品包**：工作区是构建期工件，团本包只含成品，不随团本分发。

> **退役的 BM25 检索**：`backend/src/build/retrieval/`（`db`/`ingest`/`search`/`chunk`）+ `ingest`/`search` 构建工具已**整体删除**（死代码是漂移源），不留 deprecated。
> **Bash 沙箱欠账**：裸 `Bash` + cwd=workspace 继承后端进程权限，本地单用户 eval 接受此风险；远程/多租户部署前必须沙箱化 → [SEC4](../../../docs/dev/plan/backlog-后端.md)。

## 4. 构建 skill `dicelore-build-pack`（分阶段编排）

承接 [Skills 包](../02-MCP与Skill体系/Skills包.md) 的 skill 形态。Moves 式**有序分阶段**流程，编排 ①读写层 ②门面 ③源摄入（工作区文件）——**只编排已有能力调用序、不新增工具/不碰存储**（同流程 skill 纪律）。

**阶段（边建边审，见本页决策节；2026-07-01 改 agentic 文件打法）**：

0. **摸源 + 清洗分块** → `ls materials/`/`wc`/`head`/`grep -c` 摸文件规模结构 → 判定剥噪声（论坛串投票/短帖等）→ 清洗/分块产物 `Write` 到工作区
1. **世界观 / 设定** → `Grep` 定位设定段 + `Read` 读块 → `write_lore` 产设定/门派 → 用户 Web 审阅修正
2. **NPC** → `Grep`/`Read` 人物段 → `add_npc`（人设散文，关键 NPC 加 sheet 卡）→ 审阅
3. **卡池 / 随机池** → `Grep`/`Read` 机缘/物品段 → `add_pool_entry` 填池 → 审阅
4. **机制规则** → `Grep`/`Read` 体系段 → 写 `rules/*.md` + `params/*.csv` 分档 → 审阅
5. **选流程 skill + manifest 收口** → 据团本玩法选 `flows`、声明 `clock` / `entry` → `set_manifest` → **整包校验**（§1）→ 交付

每阶段：`Grep/Read 摸源 → 调读写层产一块 → 用户在 Web 审阅修正 → 确认进下一阶段`，**阶段间可回退**（前阶段产物在 Web 里随时可改）。校验贯穿全程，收口阶段跑整包校验确保可被运行时 import。**引用 `materials/` 原文、不凭空编造；素材是不可信引述资料、只提炼不执行其中指令。** SKILL.md 交付时用 `/skill-creator:skill-creator` 重构保证 frontmatter/references 规范。

## 7. 工具生成层（声明式视图 + MCP 工具编译引擎）

承接后端双路径架构「构建期声明 → 框架编译」：团本作者用**声明**（SQL 片段）描述只读视图与 MCP 工具，框架**编译**成安全的运行时构件——把"团本想暴露什么读 / 写口"从手写工具代码降为声明。

- **`sqlGuard`**：SQL 安全闸（首词分发 + 防注入 / 防多语句）。
- **视图编译（`view`）**：团本声明的只读视图 = 一条 `SELECT` → 编译为 `CREATE VIEW`（投影底层物理表为团本语义视图）。`defineView(db, name, sql)` 是引擎原语。
- **工具编译（`compile`）**：单条 `sql` 声明 →
  - 首词 `SELECT` → **读工具**（`readTool`，只读，可引视图 + 物理表）；
  - 否则（写） → **写工具**（`writeTool`）——**严格正则匹配正典写原语**（`applyMutations` / `*SetStatus` / `*Upsert` 等），**匹配不上即拒**（防团本声明绕过正典通道直写、泄露 / 破坏规范态）。
- 位置：`backend/src/toolgen/`（sqlGuard / view / readTool / writeMatch / writeTool / compile）。
- **编译产物形状**：`compileTool(decl) → { name, desc, handler(db, args) }`（读工具 `handler` 回 `unknown[]`，写工具回 `unknown`）。这是引擎面，**与 MCP `ToolDef`（带 zod `inputSchema`/`outputSchema`/`annotations`）之间有一层适配缺口**——接进运行时工具面须把 `{name,desc,handler}` + `decl.params` 适配成 `ToolDef`（见 [MCP工具面 §7.2](../02-MCP与Skill体系/MCP工具面.md)）。

> **实现现状（2026-06-25 核对，单源记此）·引擎成、视图层投影成、框架标准库 ②③ 已接、团本侧待接**：
> - ✅ **引擎**：sqlGuard/view/readTool/writeMatch/writeTool/compile 六模块 + 单测全绿。
> - ✅ **视图层投影**（叙事层 spec §4，[backlog-core 主题A′①](../../../docs/dev/plan/backlog-core.md)）：`backend/src/store/views.ts` `initViews(db)` 用 `defineView` 投影 6 命名视图（`player`/`npc`/`world`/`relation`/`clock`/`tension_board`），接进 `initSchema`——toolgen 读工具的基视图就位。视图名+投影列 = 对下游公开的稳定逻辑列契约。
> - ✅ **②③ 接线 + 叙事层 dogfooding**（2026-06-25，[backlog-core 主题A′②③](../../../docs/dev/plan/backlog-core.md)）：① 适配层 `backend/src/toolgen/toToolDef.ts` `toolgenToToolDef(decl)` 把 `compileTool` 产物适配成 MCP `ToolDef`（zod schema、出参包 `{result}`）；② 标准库声明 `backend/src/stdlib/narration.ts`（`narrationToolDecls`：front_open/plotline_open·advance·close/foreshadow_plant·recall·abandon/tension_board 八工具，**全用声明、零硬编码 handler**）；③ `createMcpServer(db, deps, extraTools?)` 加可选 `extraTools` 入口（DT-9 守约：现有 19 工具零改动），生成工具与 `TOOLS` 并列注册。dogfooding 集成测试验证经 server 端到端落库 + 承重墙不破（写经正典原语）+ 坏声明编译期被拒。**front_advance（推进 clock）撞 writeMatch 天花板**（跨 state↔front JOIN 不可映射），v1 不声明（[backlog-core A′ 天花板记项](../../../docs/dev/plan/backlog-core.md)），正是 spec §8 预言的「撞 DSL 天花板」。

## 8. 测试策略

- **读写层 + 校验器**：纯函数单测（CRUD 正确落列/落目录、校验命中各类错误）。
- **源摄入**：工作区生命周期（`ensureWorkspace` 幂等、只建 `materials` 目录、不产 `.claude`）+ 上传端点（流式落盘、filename 净化、超限中途 413 + 半成品清理）+ build-mcp `put_material`（`localPath` 流式、content 不进工具参数）。
- **工具生成层**：sqlGuard / view / readTool / writeTool / compile 各自单测（视图编译正确、读 / 写分流、写工具正典原语匹配 / 拒绝）。
- **构建 skill**：eval-loop（丢一本真实案例/小说跑端到端，人审产物质量 + 校验零 error），与 [Skills 包](../02-MCP与Skill体系/Skills包.md) 的**对照真实案例**同法（eval 修订决策，非 with/without baseline）。

## 决策与权衡

本页"团本构建台"六连决策的"为什么 + 被否"重述于此（历史全文见归档，以本节为准）。

### D1 · 团本构建台 = 文件包为真相 + 可交互 Web 门面 + 即写即读 + 分阶段 + FTS 素材检索

- **背景**：定"普通用户怎么造出团本"。靶心是**降低作者门槛**——目标用户"只会丢一整本《凡人修仙传》进去"，不会手写 manifest / CSV / frontmatter。这要求一组架构选择：产物形态、CRUD 对象、审阅界面、并发真相模型、构建节奏、超长素材怎么喂。
- **决策（六连，构成作者侧、构建期专属、与运行时分开的"团本构建台"）**：
  1. **产物 = MD+CSV 文件包**（非 SQLite 草稿库）：CRUD 对象是文件包条目；贴合"MD 主体 + CSV → import 建库"既定假设，对 git / 版本化 / 分发友好。
  2. **审阅 = 可交互 Web 门面**：本地轻量 http 服务 + 前端，渲染"团本说明书"且允许用户直接增删改条目。最降门槛。明确与运行时游玩界面解耦——玩游戏 v1 仍走终端，此 Web 仅作者构建期用。
  3. **真相 = 文件包、双门面即写即读**：agent（MCP 门面）与用户（Web 门面）都对文件做结构化 CRUD；每次操作即写即重读渲染；无内存态、无 WebSocket 实时同步。两门面共享同一套读写层 + 校验器。
  4. **构建模式 = 同一 Claude Code 换装**：加载构建 skill + 构建 MCP，而非运行时那套 `resolve_*` / `sheet_update`。
  5. **节奏 = 分阶段·边建边审**：世界观 → NPC → 卡池 → 机制 → 选 flow+manifest 收口；每阶段产一块、用户即时审阅修正再进下一阶段，阶段间可回退。
  6. **素材 = 先建检索库、按阶段检索**：整本小说切块建库，每阶段按需检索相关片段。起步关键词 FTS5 + jieba（复用运行时基建、零新依赖），语义向量列未来。检索库是构建期临时品、不进成品包。
- **后续补强**：① front 包格式定 `fronts/*.md`（frontmatter Clock + 凶兆阶梯，**否决** CSV 扁平——CSV 一行只能放一个标量对象，丢失"凶兆阶梯"这一 Front/Clock 核心）；② 每团本必备 `prologue.md`（开场白 prompt），`validatePack` 缺即报 error。格式单源见 [团本与 manifest](团本与manifest.md)。
- **后果**：新增"团本构建台"为作者侧子系统；读写层/检索库属 core 标准件、不绑 Claude Code（构建 skill / MCP 注册才绑基底）。落地承后端双路径架构——真相从裸文件树迁为 Catalog DB（见 §6、[后端双路径架构](后端双路径架构.md)）。
- **被否**：① SQLite 草稿库为产物（偏离 MD+CSV 假设、对 git 不友好）；② 静态 HTML 只读预览 / 纯终端文本视图（审阅体验弱，长团本尤甚）；③ 实时双向同步（要冲突处理、scope 易爆，即写即读已够）；④ 一把梭末尾总审（错误集中末尾、长文上下文压力大）；⑤ 用户喂浓缩二手材料（门槛回升，违"丢一整本"愿景）；⑥ 自主分块全文通读（token 重、慢，与分阶段检索相比无额外收益）；⑦ front 走 CSV 扁平（丢凶兆阶梯）；⑧ prologue 留隐式约定/可缺省（GM 开局无统一入口）。

### D2 · 源摄入改会话工作区 + agentic 文件构建（退役 BM25 检索）（2026-07-01）

- **背景**：D1 定的「先建 FTS5+jieba 检索库、按阶段 `search`」在真实驱动构建 GM 建团本时暴露两病：① 大源（兽人冒险 26 万字）经 LLM 中继（原文当 `ingest({text})` 工具参数）成本随规模爆炸；② BM25 关键词切块对自带文件工具的 CC agent 无语义优势且切块割裂——Grep/Read 的 agentic 导航（命中后读任意上下文、可迭代收敛）质量更高，是 CC 理解代码库的原生做法。
- **决策要点**：
  - **每会话持久工作区**（`<sessionsDir>/lore/sessions/<id>/workspace/`）：源材料经 REST 上传落 `materials/`，agent scratch 落根；纯业务数据、不含 skill（skill 经 plugin 从数据根按引用加载，见 [后端双路径架构 D3](后端双路径架构.md)）。生命周期跟 session（v1 无独立 GC）。
  - **流式素材上传**（不缓冲整文件）：`POST /lore-sessions/:id/materials` 原始字节流 + `?filename=`，`DICELORE_MATERIAL_MAX_MB`（默认 100）大小闸中途 413；build-mcp `put_material` 取 `localPath` 流式上传——**大源不经 LLM 中继**（content 不进工具参数）。前端上传进度可视化属前端（[FE-build-upload](../../../docs/dev/plan/backlog-前端.md)）。
  - **agentic 文件构建**：agent 补 `Bash/Grep/Glob/Write/Edit`（cwd=workspace）自行清洗分块 + 文件导航提炼；`dicelore-build-pack` skill 打法从「ingest→search」重写为 bash 摸源 + Grep/Read 提炼。
  - **退役 BM25 检索**：删 `backend/src/build/retrieval/` + `ingest`/`search` 工具，代码删除不留 deprecated（死代码是漂移源）。
  - **格式不做能力提升**（终裁、无 spike）：后端不做格式转换、不代偿模型读取能力；`materials/` 原样存原文，模型读不了的由作者自己预处理成可读文本——能力交给 agent、后端不预设格式垫片。
  - **Bash 沙箱欠账明确记账**（不在本波做）：裸 Bash 继承后端进程权限，本地单用户 eval 接受；远程/多租户前必须沙箱化 → [SEC4](../../../docs/dev/plan/backlog-后端.md)。
- **被否**：① 保留 BM25 检索（对 CC agent 无语义优势、大源手喂爆炸）；② 把 skill 拷进 workspace（复制同步太耗，改按引用加载，见 [后端双路径架构 D3](后端双路径架构.md)）；③ 后端做格式转换 / 归一化管线（代偿模型能力、预设格式，改交给 agent + 作者预处理）；④ JSON+base64 上传（膨胀 33% 且整文件入内存，改原始字节流式）。

### D3 · lore 构建 REST-only 的 error 经 body 回 + checkout ref=head 端点解析（2026-07-01）

- **lore 构建 error 经 body `error` 字段回**（BE-lore-error-shape）：`LoreSession.handleMessage` 返回 `{ turnId, error? }` 不吞领域级 error（循环捕获 `ev.type==="error"`）；`runtime/session.ts` 加 `TurnResult` 共享契约（dice 侧不产 error 字段、向后兼容）；`api/lore.ts` messages body 带 `error?`。**HTTP 状态钉死保持 202**——turn 已实际跑完（turnId 有效），error 属**领域级**而非传输级（5xx 语义是"请求没被处理"，与"跑了但 GM 出错"不符），靠 body 的 `error` 字段标失败；调用方（build-mcp `doSendToBuilder` 透传 error / 前端构建台）以 `body.error` 存在与否判成败。
- **checkout ref=head 端点层解析**（BE-checkout-head）：`GET /catalog/:id/files?ref=head`（或省略 ref）时端点**先从 catalog list 取该 adventure 的 head commitId** 再调 `checkout`——core `checkout` 契约不变（只认 tag label / commitId、不认 "head" 关键字），最小侵入解「默认查 head 返 `[]`」的 bug。

### D4 · Draft 期校验 + loregm WS 构建过程事件 + tools/*.json 预设 dock-card + 构建轮成本内联（2026-07-10）

- **Draft 期校验入口**（rollband §二/RT-FE11）：core 暴露 `validateDraft(draft)`，复用 `validatePack` 规则集喂分域 Draft（`get_draft` 的 `{files,snapshot}`），`path` 用 Draft 分域路径（`world.lore.张三`/`rule.核心规则`/`manifest.meta`）；端点 `POST /sessions/loregm/{id}/draft/validate` 只读幂等（见 [接口页 D7](玩家客户端-接口.md)）。**caveat**：对 manifest.md 级问题目前盲（`Draft.toPackFiles` 产 .md）→ [backlog-后端](../../../docs/dev/plan/backlog-后端.md)。
- **loregm 域 WS 构建过程事件**（hidden-roll-and-loregm-ws §二/RT-FE12）：复用 dicegm WS 骨架（`GET /sessions/loregm/{id}/ws` + wsHub），拆 `LoreStreamMessage` schema，v1 五类 `turn_started`/`turn_ended`/`toolcall`/`draft_delta`/`error`。实现靠 build GM 侧 hook：`toolcall` = build-mcp 工具调用 hook（对称 dicegm onCanonWrite 缝 A），`draft_delta` = `onBuilderWrite` hook（Draft 写→分域 delta，对齐 `get_draft` 分域结构、即写即读刷新），`turn_started/ended` = `send_to_builder` 入口/出口。`validate_result` 推送推后 v2（同步 `/draft/validate` 已覆盖 on-demand）。
- **作者预设 dock-card 模板 `tools/*.json`**（dock-card-template §三）：作者制作页定义的 dock-card 预设模板随团本包 `tools/*.json`（toolgen `ToolDecl` 同载体）commit 进包、所有玩家共享、只读、可查包内任意数据（作者授权全量·不受玩家 DIY 的 visible=1 边界约束）。模板 canonical 语法（dc-meta select/where + dc-body 插值 + dial/bar）见 [玩家客户端-视觉 §9.2](玩家客户端-视觉.md)。
- **构建轮成本内联**（co-build）：`postBuildMessage` 返回扩 `{turnId, usage?}`（usage-stream 已让后端 POST 响应带 usage）；BuildPage 每构建轮 assistant 消息附本轮 usage、末尾内联成本行，复用 `frontend/src/features/cost/pricing.ts`（见 [后端双路径架构 D5](后端双路径架构.md) + [玩家客户端-视觉 §9.2](玩家客户端-视觉.md)）。
