# 组件7「玩家客户端」接口文档

> **本页职责**：玩家客户端**编排后端**对外的**规范接口契约**——① 前端 ↔ 后端的 **REST**（动作进 / 呈现快照出）+ **流式 WS/SSE**（叙述 / 呈现增量 / 回合事件）；② **MCP → 后端 notify webhook**（变更通知缝的 payload）。**这是 devs 照之实现、`packages/shared` 类型镜像、双壳与自定义 MCP 对接的那份合同。**
> **上游依赖**：[玩家客户端（组件7 设计）](玩家客户端.md)（边界 / 数据流 / 通知缝裁定）、[adapter §7 呈现模型生成器](adapter与L3审计.md)、[内层能力库](内层能力库.md)（四域 / 可见性 / `seq`）、[MCP工具面](../02-MCP与Skill体系/MCP工具面.md)（出参形状）。
> **状态**：🟢 已定 v1 契约骨架（2026-06-20）；**2026-06-22 与实现对齐**：①§0 补「两条缝」框架、②§5 重写通知缝为**进程内回调（默认）+ HTTP webhook（跨进程未来）**双形态（追 in-process 挂载决策）、③§2/§3+4 各表加**实现状态列**、④新增 §9「端到端联调」。**确切字段名以 `packages/shared` 实现定稿为准**，本页定形状与语义。

## 0. 通用约定

> **先分清「两条缝」——它们的解耦诉求相反，曾被设计混淆**（2026-06-22 厘清）：
>
> | 缝 | 谁↔谁 | 拓扑事实 | 该不该解耦 | v1 机制 |
> |---|---|---|---|---|
> | **缝 A** | dicelore MCP ↔ 编排后端 | **永远同机同进程**（GM=Agent SDK headless、MCP=in-process 挂载，见 [玩家客户端页决策节](玩家客户端.md)） | **不需要**——同进程加 HTTP 是过度设计 | **进程内 `onCanonWrite` 回调**（§5） |
> | **缝 B** | 编排后端 ↔ 呈现 UI（web） | **web 可远程、未来多租户**（一个后端服多个 web，[组件7 §7](玩家客户端.md) 场景 B） | **需要**——会话路由 / 鉴权 / 多实例分片都在这条 | REST + WS（§2/§3+4），按 `sessionId` 寻址 |
>
> 历史上曾把「MCP→HTTP webhook」当作缝 A 的解耦机制，但缝 A 同机、并不需要解耦；后续决策已把挂载定为 **in-process 默认**，缝 A 落为进程内回调（见 [玩家客户端页决策节](玩家客户端.md)）。**真正需要远程化 / 多租户的是缝 B**——它本就是 REST+WS、天然可跨网络。HTTP webhook 仍保留为缝 A 的**跨进程未来形态**（远程 / 自定义 MCP），见 §5。

- **传输**：前端 ↔ 后端（缝 B）走 **HTTP（REST）+ WS 或 SSE（流式）**；两种分发壳（Tauri localhost / Web 网络）**同一套**，不用 Tauri 私有 IPC（[组件7 §7](玩家客户端.md)）。
- **协议版本**：每条消息 / 响应带 `protocol: "dicelore.client/1"`（或 HTTP 头 `X-Dicelore-Protocol: 1`）。破坏性变更进位（§6）。
- **会话标识**：所有端点按 `sessionId` 寻址；v1 单会话亦显式带（为 Web 多会话铺路）。
- **鉴权**：Tauri ＝ 仅环回（loopback）、无网络暴露；Web ＝ REST/WS 带会话 token（细节属 [场景 B](../../项目介绍/用户与场景.md) / 未来）。**缝 A 进程内回调无需鉴权**；HTTP webhook 形态（§5）才用每会话 token。
- **类型源**：本页 JSON 形状由 `packages/shared` 导出为 TS 类型，前端 / 后端共用、免重复定义。
- **职责切分**：**动作走 REST**（请求/响应清晰、易鉴权）；**GM 产物走流式**（叙述逐字、回合事件）；**后端引擎规范态变更走缝 A（进程内回调 → 后端映射）→ 转流式**（§5）。

## 1. 呈现模型数据形状（流②核心，REST 快照与 WS 增量共用）

承接 [adapter §7 呈现模型生成器](adapter与L3审计.md)：读 store / event、**按 `visible` 过滤**后的玩家可见投影。

**全量快照**（`GET …/presentation` 与 WS 重连补齐用）：

```jsonc
{
  "protocol": "dicelore.client/1",
  "sessionId": "…",
  "seq": 1234,                       // 本快照反映到的 store seq（对账基准）
  "sheets": [                        // 呈现台：可见 cell，按 entity 分组
    { "entity": "张三",
      "cells": [ { "attr": "HP", "value": "12", "visible": 1 },
                 { "attr": "金钱", "value": "77", "visible": 1 } ] }
  ],
  "mechanics": [                     // 机械回显：可见的 verdict/mutation/watcher_fired
    { "seq": 1230, "kind": "mutation", "text": "金钱 +3d100=74 → 77",
      "data": { "张三.金钱": { "op": "rolled", "from": 3, "expr": "+3d100=74", "to": 77 } } }
  ],
  "choices": {                       // 待选项：最新已物化 kind=choice；无则 null
    "eventId": 1234,
    "options": [ { "index": 0, "label": "推门进去", "consequence": "惊动守卫" },
                 { "index": 1, "label": "绕到后窗", "consequence": "耗时但隐蔽" } ]
  },
  "pendingRoll": {                   // 待掷（明骰闸控）：最新 status=awaiting 的 pending_roll 规格；无则 null
    "eventId": 1234,
    "shape": "contest",              // "outcome"（区间 label）| "contest"（胜负 verdict）
    "label": "压价说服",
    "yourSide": { "name": "说服", "exprDisplay": "1d20+{说服}" },  // exprDisplay 仅供展示，真值不下发
    "dc": 15,                        // shape=contest 时亮 DC；可选
    "bands": [ { "label": "失败", "min": 1, "max": 9 },
               { "label": "成功", "min": 10, "max": 20 } ]         // shape=outcome 时给区间预览；可选
  },
  "narrativeCursor": 1228            // 已渲染到的 narrate event seq（重连去重用）
}
```

- `visible` 取值沿用 [内层 §4.1](内层能力库.md)：`1` 可见（投影只含可见 cell；`0`/`2` 不出现）。
- `mechanics` 仅含 `visible=1` 的机械事实；`note` 等 GM 隐事件不入。
- `choices` 由 Stop hook 物化的 `kind=choice` event 投影（[组件7 §5](玩家客户端.md)）。
- `pendingRoll` 由后端引擎 `pending_roll` 槽(status=`awaiting`)投影(玩家闸控明骰设计 §3/§6)：**只含规格、无结果**。`exprDisplay` 是展示串(如 `1d20+{说服}`)，**真值/掷出点数恒不下发**(anti-F1：点数恒由引擎在点击时计算)。该卡随 `roll_committed`(§4)清空。

**增量**（webhook 驱动、经 WS 推；只带变化部分）见 §4 `presentation_delta`。

> **动作返回 `202`**：后端把输入喂给 GM 后立即返回，GM 的叙述/裁决/呈现变更全部走流式（§4）。前端凭 `turnId` 把流上的事件归到这一回合。dicegm 走 WS 流；loregm v1 REST-only（`POST …/messages` 直接返 `{turnId, usage?}`，另经 loregm WS 推构建过程事件·见 D6）。

> **动作返回 `202`**：后端把输入喂给 GM 后立即返回，GM 的叙述 / 裁决 / 呈现变更**全部走流式**（§4）。前端凭 `turnId` 把流上的事件归到这一回合。
>
> **`POST /roll` 与 `POST /choices` 的本质差异**：`/choices` 是**跨回合异步**——GM 给选项 → 回合结束 → 玩家选 → 作**下一回合**输入。`/roll` 喂的是一次**回合内阻塞**的明骰：GM 调 `resolve_*_open`(明骰)时该工具调用**本身阻塞**(仿 AskUserQuestion)，引擎持久化 `pending_roll` 规格(无结果)并经 §4 `roll_staged` 弹卡 → 玩家点击 `POST /roll` → 引擎**此刻**掷(`commitPendingRoll`，复用 dice/expr 求值、写 verdict event)→ 结果作为**工具返回值在同一 GM 回合内**回给 GM，不跨回合、不经 Stop 物化。前端只**触发 + 播动画**(§4 `roll_committed`)，点数恒由引擎算(anti-F1)。时序详见 玩家闸控明骰设计 §3/§4。裸 CC(无前端可阻塞)时 `resolve_*_open` 当场立即掷、直接返回，降级不卡死。

## 3 + 4. 流式通道（WS / SSE，server → client）

连接：`GET /sessions/{id}/stream`（SSE）或 `…/ws`（WS 升级）。**只推 server→client**（动作走 §2 REST）。**连接后先发 `snapshot` 再发增量**；断线重连＝先 `GET /presentation` + `GET /events?since=` 补齐，再续流（去重用 `seq` / `narrativeCursor`）。

**消息目录**（每条带 `type` + `protocol`）。**「实现状态」列对账后端 `harness/src/dicegm/turnLoop.ts` + `notify.ts`、前端 `frontend/src/features/play/useSession.ts`**：

| `type` | 何时 | payload | 落点（UI） | 实现状态 |
|---|---|---|---|---|
| `turn_started` | 玩家输入被 GM 接手 | `{ "turnId" }` | 进入"生成中" | 🚧 后端发、**前端丢弃**（注④） |
| `narration_delta` | GM `narrate` 流出 token | `{ "turnId", "text" }` | 叙述区（打字机，流①） | ❌ 未实现（注⑤） |
| `narration_commit` | 一条 narrate 落 event | `{ "seq", "text" }` | 叙述区定稿 + 推进 `narrativeCursor` | ⚠️ 收发通，但 **`seq` 语义债**（注⑥） |
| `presentation_delta` | 后端引擎规范态变更（**缝 A `onCanonWrite` 驱动**，§5） | `{ "seq", "changes": { "sheets?", "mechanics?", "reveal?", "watcherFired?" } }` | 呈现台增量 + 机械回显（流②） | ✅（前端收到即 `GET /presentation` 全量对账，注⑦） |
| `choices` | Stop hook 物化待选项 | §1 `choices` 形状 | 待选项按钮 | ✅（前端收到走 refetch，未直接用 payload） |
| `roll_staged` | GM 调 `resolve_*_open`、引擎暂存 `pending_roll`（待掷） | `{ "pendingRoll" }`（§1 `pendingRoll` 形状；`bands` 每档含 `plan`+`narration` 双字段·全量下发·见 D9） | 弹 BG3 掷骰卡（亮 DC / 区间 / `exprDisplay`） | ✅ |
| `roll_committed` | 玩家点击触发、引擎掷出 verdict | `{ "eventId", "rolls": [18], "total": 18, "dc?": 15, "outcome": "success" }`（`outcome`: `"success"`/`"fail"` 或 band-label） | 骰子动效 + 成败高亮，清掷骰卡 | ✅ |
| `hidden_roll` | GM `resolve_*` 带 `hidden:true`、引擎立即掷（暗骰） | `{ "eventId", "label", "result", "dc?", "band?" }`（**带完整结果**·后端不截流·前端按 spoiler 档渲染·见 D9） | 暗骰 mech：严格档显「进行了判定」隐结果、关闭档显完整 | ✅ |
| `turn_ended` | 回合自然结束 | `{ "turnId", "seq", "usage?" }`（usage=本回合四类 token 合计·可选向后兼容·见 [后端双路径架构](后端双路径架构.md) turn-end 携 usage） | 收尾，可接受下一输入；回合尾内联 per-turn 成本 | ✅ |
| `game_end` | 终局（`game_end`/`you_death`） | `{ "reason", "outcome", "status": "debrief" }`（转战后复盘态不归档·见 D5） | 终局复盘态（不遮罩） | ✅ |
| `context_compacting` | 上下文接近窗口上限、SDK auto-compact | `{ "phase": "start"\|"done", "result?": "success"\|"failed" }`（第 11 类·见 D6） | 压缩提示 + indeterminate 进度条；done 后占用%回落 | ✅ |
| `error` | 后端 / GM 错误 | `{ "code", "message" }` | 提示 | ✅ |

> **实现状态注（流式，联调必读）**：
> - **注④ 前端现处理 9/10 消息**（2026-06-24 核验纠旧）：`useSession.ts` 的 switch 现接 `turn_started`/`narration_commit`/`presentation_delta`/`choices`/`roll_staged`/`roll_committed`/`turn_ended`/`game_end`/`error` 九种；唯 `narration_delta` 不处理（后端也不发，见注⑤）。故「生成中」态、回合收尾、错误提示、终局画面**前端侧均已接**；但 `game_end` 后端从不发（§10.1 B3）→ 终局画面实际仍不触发。
> - **注⑤ `narration_delta` 不发**：`AgentSdkDriver` 把 GM 的一段 `narrate` 作**整块**返回，`turnLoop` 直接发 `narration_commit`，**无 token 级流式**。打字机效果待 Agent SDK 流式接入（in-process 挂载决策的下一阶段「token 级逐字 narration」，见 [玩家客户端页决策节](玩家客户端.md)）。
> - **注⑥ `narration_commit.seq` 语义债**：`turnLoop` 用**回合内 `let seq=0` 自增计数器**填该字段（`turn_ended.seq` 同源），**不是全局 event `seq`**；而 §1 快照的 `narrativeCursor` 期望全局 event seq。两者对账口径不一致——重连去重逻辑目前不可靠。应统一为「落 event 的全局 `seq`」。
> - **注⑦ 增量退化为全量对账**：前端收到 `presentation_delta` / `choices` 一律 `GET /presentation` 重取全量（`useSession.ts` 的「Phase 1 对账」），**不解析 `changes` 增量**。功能正确但每次变更一个 HTTP 往返；增量解析是后续优化。

## 5. 变更通知缝（缝 A：MCP → 后端）

[组件7 §6](玩家客户端.md) 的缝 A：dicelore MCP（组件2）写规范态成功后，把「改了什么」告知编排后端，后端映射成 §4 `presentation_delta` / `roll_committed` 推前端（流②）。**有两种形态**——v1 因 MCP 与后端同进程（in-process 默认，见 [玩家客户端页决策节](玩家客户端.md)）走 **5.1 进程内回调**；**5.2 HTTP webhook** 是跨进程 / 远程 / 自定义 MCP 的未来形态，二者**语义等价、只是传输不同**。

| 形态 | 传输 | 适用 | v1 状态 |
|---|---|---|---|
| **5.1 进程内 `onCanonWrite` 回调** | 函数调用（零 HTTP） | MCP in-process 挂载（同进程） | ✅ **已实现、v1 默认** |
| **5.2 HTTP webhook** | `POST {NOTIFY_URL}` | MCP 跨进程 / 远程 / 自定义 MCP host | ❌ 未实现（契约已备，留未来） |

### 5.1 进程内回调（v1 默认，已实现）

- **机制**：后端引擎 `createMcpServer(db, deps)` 接受 `deps.onCanonWrite`；工具写规范态成功后**在 handler 外层**回调 `onCanonWrite(evt: CanonWriteEvent)`（不进 `runTool`、引擎零改动，见 [玩家客户端页决策节](玩家客户端.md)）。`evt` 带 `toolName` / `seq` / `output`。
- **映射**：`SessionHost.onCanonWrite` → `mapCanonWrite(evt)`（`harness/src/dicegm/notify.ts`）→ `WsHub.broadcast`：
  - `resolve_outcome_open` / `resolve_contest_open` 出参 → **`roll_committed`**；
  - 其余规范态写 → **`presentation_delta`**（前端收到 `GET /presentation` 全量对账，注⑦）。
- **现实粒度（实现 gap）**：`mapCanonWrite` 当前把「非明骰」的所有写**一律塌成 `presentation_delta`**，**未**按 5.2 的 `kind`（mutation / visibility / reveal / watcher_fired / choice_staged / game_end / bulk）细分，也**未**单独发 `game_end`。靠前端全量对账兜住正确性，但 `game_end` 等需要专门 UI 的信号丢失（见注④/§3+4）。
- **无鉴权 / 无序列化**：同进程直传对象，缝 A 不涉及 token（§0）。

### 5.2 HTTP webhook（跨进程未来形态）

> 用于 MCP 作独立进程 / 远程部署 / 自定义 MCP host —— 此时无法进程内回调，改走 HTTP。**契约（payload）已在 `packages/shared/src/notify.ts` 备好（`NotifyPayload` / `NotifyKind`），但后端尚无 `/internal/notify` 接收端点、agent 工具面尚无出站 HTTP 客户端**。

- **端点（后端侧）**：`POST {DICELORE_NOTIFY_URL}`（如 `http://127.0.0.1:{port}/internal/notify`）。
- **鉴权**：头 `X-Dicelore-Notify-Token: {token}`——后端启动时生成、经 env 传给 MCP、MCP 回带；Web 多会话下 token 即会话路由键。
- **payload**（＝ `packages/shared` 的 `NotifyPayload`）：

```jsonc
{
  "protocol": "dicelore.notify/1",
  "sessionId": "…",
  "seq": 1235,                  // 本次写后的 store seq
  "kind": "mutation",           // mutation | event | visibility | reveal | watcher_fired | choice_staged | game_end | bulk
  "delta": { /* 与 §4 presentation_delta.changes 同构的局部；bulk 时为空，提示后端回读全量快照 */ }
}
```

- **`kind` 与触发工具**：`mutation`←`sheet_update`；`event`←`event_append`(可见)；`visibility`←`sheet_show`/`world_show`；`reveal`←`reveal_once`；`watcher_fired`←watcher 就地触发；`choice_staged`←`resolve_choice`（暂存，**不**含后果给前端，仅提示"有暂存"）；`game_end`←`game_end`；`bulk`←`restore`/`branch`（整表，后端回读 `GET /presentation`）。这套细分粒度是 5.1 进程内回调将来应补齐的目标。
- **不发**：自定义 MCP（out-of-canon，[组件7 §8](玩家客户端.md)）。
- **响应**：`204`；后端失败不应阻塞 MCP 主流程（fire-and-forget + 后端自有重连补齐兜底）。

> **契约层归属**：`packages/shared/src/notify.ts`（`NotifyPayload`/`NotifyKind`）**只服务 5.2 webhook 形态**；5.1 进程内路径用后端引擎的 `CanonWriteEvent` → `StreamMessage`（`mapCanonWrite`），**不经 `NotifyPayload`**。故 `notify.ts` 契约目前**无运行时消费者**（备而未用），勿误以为它在缝 A 现役。

## 决策与权衡

> 本节精炼重述本页相关决策的**接口 / wire 契约面**（明骰消息、两条缝的传输形态）；这两条决策的非接口面（立项、实时引擎面实现）落在 [玩家客户端（设计）](玩家客户端.md) 的决策节。历史全文以归档「历史决策日志」为准。

### D1 玩家闸控明骰：明 / 暗骰名分流 + 阻塞式 resolver + anti-F1 边界

- **背景**：一条参与感诉求——若所有掷骰都由引擎自动决定，玩家会觉得"还不是 AI 直接决定我的命运"。解法 ＝ 把"掷骰这个**动作**"交还玩家（点击触发、亮 DC 与点数、见证成败），但"点数"仍归引擎。两件事正交：玩家点按钮 ≠ 玩家定数。不动的地基 ＝ anti-F1（随机全在引擎）+ 可见性（`visible` 列语义）。
- **决策（落本页 §1 `pendingRoll` / §2 `POST /roll` / §4 `roll_staged`·`roll_committed`）**：
  - **① 两条正交轴**：点数权威恒引擎（客户端篡改伪造不了真值）；掷骰动作 + 透明度新增一轴——明骰 ＝ 玩家点击触发 + 亮 DC + 见证，暗骰 ＝ 引擎自动掷。
  - **② L1 名分流、无布尔参**：不给 `resolve_*` 加 `gated`/`visible` 布尔参，而是 `resolve_outcome`（区间 label）/ `resolve_contest`（胜负 verdict）各拆 `_hidden`（暗、引擎自动掷）/ `_open`（明、玩家闸控掷）。`resolve_choice`（玩家选 label）不动——与明骰同属"玩家面向交互式 resolver"族。
  - **③ 明骰 ＝ 阻塞式 MCP 调用**：`resolve_*_open` 工具调用本身阻塞，结果作工具返回值在**同一 GM 回合内**回给 GM——不跨回合、不经 Stop 物化。能力存在 → 持久化 `pending_roll`（规格、无结果）→ 经 `roll_staged` 弹卡 → 玩家点 `POST /roll` → 引擎此刻掷 + 写 verdict event → 回合内返回。
  - **④ anti-F1 边界**：点数恒由引擎在玩家点击时计算；前端只触发 + 播骰子动画 + 成败高亮；`pendingRoll` 只下发规格（`exprDisplay`），**真值 / 掷出点数恒不下发**。
  - **⑤ 裸 CC 降级**：无可阻塞前端 → handler 当场立即掷、直接返回（不阻塞、跳过人机往返），不卡死。
- **与 choice 的关系**：明骰 ＝ 玩家闸控掷、choice ＝ 玩家选——并列于"玩家面向交互式 resolver"族，互不物化为对方；choice 跨回合异步（作下一回合输入），明骰回合内阻塞返回。详见 §8。
- **被否**：① `resolve_*` 加 `gated`/`visible` 布尔参（消歧靠 L1 工具名分流）；② 明骰走 Stop 物化跨回合（破"回合内见证成败"、丢阻塞返回这一拍）；③ 把点数下放客户端 / 前端播种（破 anti-F1，客户端可伪造真值）。

### D2 两条缝：缝 A 进程内 onCanonWrite 回调 + 缝 B REST/WS（webhook 降为缝 A 未来形态）

- **背景**：实时引擎面把 dicelore MCP 定为 in-process 默认挂载，这意味着 MCP↔后端同进程。据此厘清"两条缝"——历史上曾把"MCP→HTTP webhook"当作缝 A 的解耦机制，但缝 A 同机、并不需要解耦。
- **决策（落本页 §0 两条缝框架 / §5 通知缝双形态）**：
  - **缝 A（dicelore MCP ↔ 编排后端）永远同机同进程，不需要解耦**：落为**进程内 `onCanonWrite` 回调**（零 HTTP）——工具写规范态成功后在 handler 外层回调，后端映射成 `presentation_delta` / `roll_committed` 推前端。无鉴权 / 无序列化（同进程直传对象）。
  - **缝 B（编排后端 ↔ 呈现 UI）web 可远程、未来多租户，需要解耦**：本就是 REST + WS、天然可跨网络；会话路由 / 鉴权 / 多实例分片都在这条，按 `sessionId` 寻址。`@dicelore/shared` 已超前预留 `SessionStatus`/`SessionSummary` 为缝 B 多租户铺路（多实例分片仍属未来）。
  - **HTTP webhook 保留为缝 A 的跨进程未来形态**（远程 / 自定义 MCP host）：契约（`NotifyPayload`/`NotifyKind`）已在 `packages/shared` 备好，但无运行时消费者（备而未用）；与进程内回调语义等价、只是传输不同。
- **被否**：① 把 HTTP webhook 当缝 A 必需的解耦机制（缝 A 同机，同进程加 HTTP 是过度设计）；② onCanonWrite 写进 `runTool`（破"不改引擎"边界，外层包同样拿得到出参）。

### D3 会话面对称拉平 `/sessions/{kind}`（2026-07-10·session-surface-flatten）

- **背景**：会话内部 `Session`/`TurnResult` 已统一，仅 HTTP 表皮分裂（dicegm `/sessions/*` vs loregm `/lore-sessions/*`）、且两侧缺显式建会话（懒建）、loregm 缺列表/meta（acceptance-loop RT-ns/RT1/RT6/RT7）。
- **决策（落本页 §2）**：① 表皮按 `kind ∈ {dicegm,loregm}` 参数化，基址 `/sessions/{kind}/{id}`，两 kind 共享生命周期骨架、域子资源各异；② **破坏性直改**、不留 `/lore-sessions` 307 别名（开发期无外部消费者、单源）；③ **删懒建**（dicegm `/open`、loregm 首访 `/messages|/materials`）→ 一律显式 `POST /sessions/{kind}` 先建；④ 补 loregm 列表（`GET /sessions/loregm`）+ meta（`GET /sessions/loregm/{id}`）；⑤ `SessionSummary` 统一形状 `{sessionId,kind,status,title,packName?,started,lastActionAt,lastReply?,lastaction?}`——**packName 必填**（不空）、lastaction/lastReply 按 kind 有意义则填。
- **被否**：保留懒建作兜底（双写漂移源）、保留 `/lore-sessions` 过渡别名。

### D4 统一 session config 端点（2026-07-10·model-switch + spoiler-tiering 合流）

- **背景**：model 运行时切换（RT-FE18）与防剧透档位（RT-FE9）都需一个会话级配置写入口。独立 `POST …/model` 会让元动作端点 proliferate。
- **决策（落本页 §2 config 行）**：统一 `GET/POST /sessions/{kind}/{id}/config`（部分更新 body，只传要改字段）。`model` 设 `pendingModel`、**下回合 `drive-turn` 起生效**（`buildQueryOptions` 每回合读 `currentModel`）；`spoilerTier ∈ strict|loose|off` 存 session-meta `spoiler_tier`、**立即生效**（前端渲染层）。**同端点、不同生效时机**。两 kind 都持 config。切 model 后 cache 自然失效（首回合 cacheRead 骤降），不特殊处理。
- **被否**：独立 `POST …/model` 端点（元动作端点膨胀）；spoilerTier 走后端截流（见 D8 两层正交）。

### D5 战后复盘态 + 会话分支模型（2026-07-10·debrief-and-branch）

- **背景**：前端大改暴露状态机两缺口——战后复盘态（RT-FE7）、rewind/branch 两功能（RT-FE8）。
- **决策**：
  - **复盘态**：GM AI 调**现有** `game_end` MCP（不新增 `enter_debrief`）→ 会话转「战后复盘」态**不归档**、幂等；harness 检测 game_end 后加载 `debrief-mode` skill 切 AI 行为（不推进剧情、答玩家提问、软约束不强拦）。A1 `status` 扩 `{活跃,空,战后复盘,归档}`；A2 删「终局→归档」直达边（必经复盘）。退出三条：branch 回档续玩 / 复用 `DELETE` 显式结束归档 / 关会话保持态。
  - **分支**：一 session 多 branch（各一 jsonl、独立 seq/快照），默认分支 `main`。`rewind` = 覆盖**当前分支**（截断，dry-run 前端本地算、零后端改动）；`branch` = copy 当前分支到 fromSeq 的新 jsonl、**自动成当前分支**。`POST/GET /branches` + `POST /branches/{bid}/checkout`（切分支+返快照）。**仅 dicegm 支持分支**（loregm 走 Draft 增量、v1 不做）。
- **被否**：新增 `enter_debrief` 工具/端点（复用 game_end 语义转变）；复盘态硬拦 roll/choices/game_end（靠 skill 软约束）；新 WS `debrief_entered`（靠 game_end + GET meta status）。

### D6 GM session 续接 + 用量/上下文可见性（2026-07-10·gm-session-continuity + usage-and-context）

- **背景（缺陷核实）**：现状每回合 `agentFactory(buildInit())` 新建 DiceGm、`query()` options **无 `resume`**、SDK `session_id` 只进日志未存库 → LLM 对话历史**完全不跨回合续接**，GM「记忆」被迫全靠每回合调 canon 工具重建（canon 越权扛了本应 LLM history 扛的记忆主载体）。
- **决策（续接）**：**一个团本 = 一个 SDK session**。首回合从 system init 取 `session_id` 经 DiceGm 上抛 → `backend.metaSet("sdk_session_id")` 存库；后续回合 `buildInit` 注入 `resume: <sdk_session_id>`（`AgentInit`/`GmQueryOptions` 加 `resume?`，`buildQueryOptions` 透传）。SDK 无 messages 入参、历史由 SDK 自持久化（`persistSession` 默认 true），dicelore 只存一个 session_id 指针、不喂 transcript。DiceGm 保持每回合 new（resume 续接 SDK transcript 与实例无关）。canon 回归**辅助 MCP**。resume 失败（transcript 丢失）→ 报错提示历史丢失、需开新局（用户定调 C4·不静默降级）。为 RT-FE8 rewind/branch 铺 `resumeSessionAt`/`forkSession` 基础。
- **决策（用量/上下文·扩 `GET …/usage`）**：`contextTokens = 最近一轮 (input+cacheRead+cacheCreation)`、`contextWindow` 查 `CONTEXT_WINDOW` 表（`packages/shared`）、`contextPct = contextTokens/contextWindow`、`sessionTotal = Σ 各轮`。扩返回加 `perTurn[]` + `memoryBreakdown?`（prompt 各段 token：systemPrompt/narrativeMemory/sheet/foreshadow/front，v1 做）+ `mcpBreakdown?`（工具 calls + 最近一次估算 token·降级口径·标「估算」，v1 做）。**auto-compact 复用 SDK**（不自研）：`buildQueryOptions` 经 `Options.settings` 注入 `{autoCompactEnabled:true, autoCompactWindow:CONTEXT_WINDOW[model]}`；harness 订阅 SDK `SDKStatusMessage.status==='compacting'` → 广播 **WS 第 11 类 `context_compacting`**（start/done+result，进度用 indeterminate 动画）；foot 占用 >90% 变红 + 「即将触发压缩」提示。**依赖续接先行**（history 累积压缩才有老回合可 summarize）。loregm v1 无 GET/usage、无 compact。
- **被否**：自研历史管理/手动喂 messages（复用 SDK resume）；resume 失败静默 fallback 开新 session（用户要求报错）；精确按工具拆 mcp token（SDK 不暴露·v2）。

### D7 loregm Draft 期校验端点（2026-07-10·rollband-narration-and-loregm-api §二）

- **决策（落本页 §2 loregm 域）**：`POST /sessions/loregm/{id}/draft/validate` 无 body、只读幂等、无副作用，返 `{issues:[{level,path,msg}]}`（level ∈ error|warn）。**复用** core `validatePack` 规则集，经新 `validateDraft(draft)` 入口喂分域 Draft（`get_draft` 的 `{files,snapshot}`）。`path` 用 **Draft 分域路径**（`world.lore.张三`/`rule.核心规则`/`manifest.meta`），区别于 `/catalog/validate` 的文件路径（对应未物化 vs 已提交包两态，是期望差异）。同步 on-demand，与 loregm WS 的异步推送（v2）互补。**caveat**：`validateDraft` 对 manifest.md 级问题目前盲（`Draft.toPackFiles` 产 .md）——见 backlog-后端。

### D8 presentation 接叙事视图层 + visible/spoiler 两层正交（2026-07-10·a-prime §7 + spoiler-tiering §一）

- **背景**：前端「剧情线/世界书浮窗」走 presentation 拿不到 plotline/world（RT-FE4：视图层建了、presentation 没接）；且 visible 语义与防剧透耦成「硬底线永不下发」（原设计），破前端渲染层。
- **决策（presentation 扩字段）**：`PresentationSnapshotSchema` 加 `plotlines?`/`foreshadows?`/`lore?` + `PresentationChangesSchema` 加对应 delta。玩家可见范围：plotline `active+closed` 下发（front 不下发·GM 工具）、foreshadow 只下 `recalled` 且 `visible=1`、lore `visible=1`。不另起端点（dock-card dc-meta select 单源从 snapshot 取）。
- **决策（visible/spoiler 两层正交·推翻硬底线）**：**visible（数据层 0/1）**=「是否已披露」标记，双重角色（AI note：知道哪些别对玩家说、要披露走 `reveal_once`/`sheet_show` 翻 0→1；dock-card 显示依据：显 visible=1）。**spoiler 档（呈现层 strict/loose/off）**=纯前端渲染档、每种 GM 行为按档渲染不同详细度、与 visible 正交。**取消「visible=0 永不下发硬底线」**：WS stream + `GET /events` **全量下发含 visible=0**（废 `visibleOnly` 默认过滤）；防剧透靠前端 spoiler + AI Principles，非后端截流。bay 关闭档才按需拉 visible=0（`includeHidden=true`+分页）。dock-card 显隐看 select 结果集 count（count=0 不渲染 card；agent 翻 visible=1 时前端本地刷新 select 显出、不查后端）。默认档 strict。narration/choices/pendingRoll 不受 spoiler 影响。
- **被否**：visible 三值 `{0,1,2}`（是否披露是布尔）；visible=0 后端截流硬底线（破渲染层）；三档=visible 阈值×mechanics 显隐（改两层正交模型）。

### D9 明骰档位双叙述 + 暗骰 WS + loregm 域 WS（2026-07-10·rollband §一 + hidden-roll-and-loregm-ws）

### 8.1 掷骰式裁决 / 区间裁决 vs choice 物化的关系（**已回填**）

> 原 [视觉页 §8](玩家客户端-视觉.md) 与本页曾留「掷骰式裁决 UI（区间裁决）与现有 `choices` 物化的关系**待厘清**」。**已由 玩家闸控明骰设计 厘清并落本页 §1/§2/§4**：

- **明骰 = 玩家闸控掷，choice = 玩家选**——两者**并列于「玩家面向交互式 resolver」族**：一个玩家提供「我掷了」(值由引擎出)、一个玩家提供「选了哪个」。**互不物化为对方**。
- **机制各异**：
  - **choice**：`resolve_choice` 经 Stop hook 物化 `kind=choice` event → §1 `choices` 投影 → `POST /choices` 作**下一回合**输入（**跨回合异步**）。
  - **明骰**：`resolve_*_open`(`outcome`/`contest` 两形)是**回合内阻塞**调用 → §1 `pendingRoll` 投影 + §4 `roll_staged` 弹卡 → `POST /roll` 触发、引擎掷 → §4 `roll_committed` 回显 → 结果作**工具返回值同回合**回 GM，**不经 Stop 物化**。
- **anti-F1 边界**：明骰点数恒由引擎在玩家点击时计算；`pendingRoll` 只下发规格(`exprDisplay`)，前端只触发 + 播动画，篡改伪造不了真值。
- 与之对照的暗骰(`resolve_*_hidden`，引擎自动掷)不入玩家面向交互流，照旧经**缝 A `onCanonWrite`** → `presentation_delta` 回显。
